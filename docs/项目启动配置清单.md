# Lattice Java 项目启动配置清单

## 1. 这份文档要回答什么

这份文档不是“打勾式清单”，而是直接回答一个问题：

> 这个项目真正启动起来之前，我到底需要配置什么，按什么顺序做？

适用范围：

- 当前仓库本地开发环境
- Spring Boot 3.5.1
- JDK 21
- PostgreSQL 16
- Redis 7

基线来源：

- `docker-compose.yml`
- `src/main/resources/application.yml`
- `src/main/resources/db/schema.sql`
- `pom.xml`

补充说明：

- 当前仓库不再使用 Flyway；DDL 由 `src/main/resources/db/schema.sql` 手动执行。

---

## 2. 一句话结论

如果你按当前仓库默认口径本地启动，这个项目真正需要你处理的事情只有 4 件：

1. 准备好 `JDK 21`、`Maven`、`Docker`
2. 确认现有 `PostgreSQL` 和 `Redis` 容器可用
3. 手动执行 `src/main/resources/db/schema.sql` 初始化唯一业务 schema：`lattice`
4. 启动 Spring Boot

其他配置大多都有默认值，不是启动阻塞项。

---

## 3. 项目真正需要配置的东西

下面按“是否会阻塞启动”来分。

### 3.1 启动必配项

| 项目 | 是否必须显式处理 | 当前口径 |
| --- | --- | --- |
| JDK | 是 | 必须是 `JDK 21` |
| PostgreSQL | 是 | 默认连接 `127.0.0.1:5432/ai-rag-knowledge` |
| Redis | 是 | 默认连接 `127.0.0.1:6379` |
| DDL | 是 | 手动执行 `src/main/resources/db/schema.sql` |
| `lattice` schema | 是 | 只保留一个业务 schema：`lattice` |

### 3.2 有默认值、可不显式配置的项

| 项目 | 默认值 | 什么时候才需要自己改 |
| --- | --- | --- |
| `SPRING_DATASOURCE_URL` | `jdbc:postgresql://127.0.0.1:5432/ai-rag-knowledge?currentSchema=lattice` | 你的数据库地址、库名、schema 不是默认值时 |
| `SPRING_DATASOURCE_USERNAME` | `postgres` | 账号不是 `postgres` 时 |
| `SPRING_DATASOURCE_PASSWORD` | `postgres` | 密码不是 `postgres` 时 |
| `LATTICE_REDIS_HOST` | `127.0.0.1` | Redis 不在本机时 |
| `LATTICE_REDIS_PORT` | `6379` | Redis 端口不是 `6379` 时 |
| `SPRING_AI_OPENAI_API_KEY` | `any-non-empty-value` | 你要真实调用 OpenAI 时 |
| `SPRING_AI_OPENAI_BASE_URL` | `https://api.openai.com` | 你使用 OpenAI 兼容网关、中转平台或公司代理时 |
| `SPRING_AI_ANTHROPIC_API_KEY` | `any-non-empty-value` | 你要真实调用 Anthropic 时 |
| `SPRING_AI_ANTHROPIC_BASE_URL` | `https://api.anthropic.com` | 你使用 Anthropic 兼容网关、中转平台或公司代理时 |

补充口径：

- 从 Spring AI 能力上看，模型 URL 已经可以纯走配置，不需要改业务代码
- 但考虑到你这里的实际使用习惯通常是“自定义模型 URL”，所以建议把 `base-url` 也当成常规显式配置项
- 如果你直连官方模型服务，可以不配；如果你走 88code、oneapi 或其他兼容网关，就必须显式配置

### 3.3 可选增强项

这些不是启动前置条件，只是增强能力：

- `LATTICE_LLM_REVIEW_ENABLED=true`
  - 开启真实 LLM 审查
- `LATTICE_QUERY_SEARCH_VECTOR_ENABLED=true`
  - 开启向量检索
- `LATTICE_QUERY_SEARCH_FTS_CONFIG`
  - 指定全文检索首选 `tsconfig`

## 4. 推荐启动方式

对当前项目，最推荐的本地启动方式是：

1. 直接复用当前机器上已有的 `vector_db` 和 `redis` 容器
2. 手工创建一次 `lattice` schema
3. 用 `mvn -s .codex/maven-settings.xml spring-boot:run` 启动

只有当这两个默认容器不存在时，才回退到仓库自带的 `docker-compose.yml`。

这样最贴近当前仓库的真实开发口径。

## 5. 从零启动的实际操作过程

下面是按顺序执行的一套流程。

### 第 1 步：确认基础环境

先确认这 3 个东西都可用：

- `java` 指向 `JDK 21`
- `mvn` 可用
- `docker compose` 可用

建议执行：

```bash
java -version
mvn -version
docker compose version
```

你还可以顺手确认默认端口没被占：

```bash
lsof -i :5432
lsof -i :6379
lsof -i :8080
```

如果 `5432` 或 `6379` 已经被别的实例占用了，就要么停掉旧实例，要么改你自己的连接配置。

### 第 2 步：确认依赖服务可用

当前项目默认直接复用下面两个容器：

- PostgreSQL：`vector_db`
- Redis：`redis`

建议先检查：

```bash
docker ps --format '{{.Names}}'
```

正常情况下你应该至少看到：

- `vector_db`
- `redis`

如果这两个容器已经存在并可用，就直接进入下一步。

只有当它们不存在时，才在项目根目录执行：

```bash
docker compose up -d postgres redis
docker compose ps
```

这时你通常会看到：

- `lattice_b1_postgres`
- `lattice_b3_redis`

状态最终变成 `healthy`。

### 第 3 步：初始化业务 schema

这个动作在首次启动、DDL 变化或需要清空开发库时执行：

```bash
./scripts/reset-lattice-schema.sh
```

为什么这一步要单独做：

- 当前项目默认业务 schema 是 `lattice`
- 应用启动不会自动建表
- `schema.sql` 是当前唯一 DDL 来源

### 第 4 步：决定要不要显式导出环境变量

如果你完全使用仓库默认口径，本地其实可以不显式导出大部分变量。

但为了避免串环境，建议至少导出下面这组：

```bash
export SPRING_DATASOURCE_URL="jdbc:postgresql://127.0.0.1:5432/ai-rag-knowledge?currentSchema=lattice"
export SPRING_DATASOURCE_USERNAME=postgres
export SPRING_DATASOURCE_PASSWORD=postgres
export LATTICE_REDIS_HOST=127.0.0.1
export LATTICE_REDIS_PORT=6379
```

关于 API Key：

- 当前 `application.yml` 已给了非空默认值
- 所以“只为启动成功”这件事，并不要求你现在就配真实 OpenAI / Anthropic Key
- 只有你要跑真实模型调用时，才需要改成有效 Key

如果你要走真实模型，而且通常使用自定义网关，建议把 Key 和 URL 一起显式导出：

```bash
export SPRING_AI_OPENAI_API_KEY=你的真实 OpenAI Key
export SPRING_AI_OPENAI_BASE_URL=你的 OpenAI 兼容网关地址
export SPRING_AI_ANTHROPIC_API_KEY=你的真实 Anthropic Key
export SPRING_AI_ANTHROPIC_BASE_URL=你的 Anthropic 兼容网关地址
```

如果你只是本地验证启动，不打真实模型调用，也可以继续使用默认占位值和默认官方 URL，不会阻塞服务启动。

### 第 5 步：手动初始化或重建 schema

应用启动不会自动建表。首次启动、表结构变化或需要清空开发库时，显式执行：

```bash
./scripts/reset-lattice-schema.sh
```

脚本会删除并重建 `ai-rag-knowledge.lattice`，然后执行 `src/main/resources/db/schema.sql`。不要再为测试或验收创建 `lattice_xxx` schema。

### 第 6 步：启动应用

如果你是本地开发、要改 `/admin` 静态页面、又不想每次都重启，优先用新的开发态脚本：

```bash
./scripts/run-local-dev.sh
```

这条命令默认会：

- 使用 `local-dev` profile
- 监听 `18082`
- 固定连接 `ai-rag-knowledge` 里的 `lattice` schema
- 主动覆盖 `SPRING_PROFILES_ACTIVE`、`SPRING_DATASOURCE_URL`，避免沿用旧 shell 里的历史验收配置
- 使用纯文本控制台日志，避免 JSON 日志和 Maven 输出混在一起难读
- 打开 DevTools 与静态资源直连，方便调 `/src/main/resources/static/admin/**`

如果暂时没有配置 `deep_research` 的 `planner / researcher / synthesizer / reviewer` 绑定，可以临时加：

```bash
export LATTICE_LLM_DEEP_RESEARCH_STARTUP_VALIDATION_ENABLED=false
```

这只适合“启动 / 工作台 / 资料处理任务”类验收；要验收 Deep Research 本身时，仍必须补齐对应绑定，不要用这个开关绕过能力验收。

如果你刚改过 DDL、或者想把开发库清到最干净，再启动：

```bash
./scripts/run-local-dev.sh --reset-schema
```

如果你必须直接从 IntelliJ 的 `LatticeApplication` Run/Debug 配置启动，也建议保持同样的口径：

- 固定 `local-dev` profile
- 工作目录指向项目根目录
- 通过启动参数显式加载 `src/main/resources` 下的配置、日志和静态资源

推荐参数：

```text
--spring.config.additional-location=optional:file:src/main/resources/
--logging.config=file:src/main/resources/logback-spring.xml
--spring.web.resources.static-locations=file:src/main/resources/static/
```

这样即使 IntelliJ 某次没有及时把 `application*.yml` 复制到 `target/classes`，也不会退回默认 profile、默认 JSON 时间格式，或者因为 `spring.ai.anthropic.api-key` 兜底配置未加载而启动失败。

当前推荐命令：

```bash
mvn -s .codex/maven-settings.xml spring-boot:run
```

如果你全局 Maven 镜像已经稳定，也可以直接：

```bash
mvn spring-boot:run
```

如果你更想跑打包产物：

```bash
mvn -s .codex/maven-settings.xml package -DskipTests
java -jar target/lattice-java-1.0-SNAPSHOT.jar
```

### 第 7 步：看启动成功没有

启动成功至少要满足 3 件事：

1. 日志出现 `Started LatticeApplication`
2. `lattice` 下业务表存在
3. 健康检查是 `UP`

建议执行：

```bash
curl http://127.0.0.1:18082/actuator/health
docker exec vector_db psql -U postgres -d ai-rag-knowledge -c "\dt lattice.*"
```

如果你走的是默认非 `local-dev` 启动口径，再执行：

```bash
curl http://127.0.0.1:8080/actuator/health
docker exec vector_db psql -U postgres -d ai-rag-knowledge -c "\dt lattice.*"
```

如果返回类似下面内容，说明基础启动已经成功：

```json
{"status":"UP"}
```

## 6. 你真正要改配置的场景

下面按场景说明“什么时候才需要改配置”。

### 场景 A：只是本地把服务跑起来

你真正要配的是：

- JDK 21
- PostgreSQL
- Redis
- `lattice` schema，按 `schema.sql` 手动初始化

其他大多数配置可以沿用默认值。

### 场景 B：你要真实调用 OpenAI / Anthropic

这时建议把 Key 和 URL 一起明确下来：

```bash
export SPRING_AI_OPENAI_API_KEY=你的真实 OpenAI Key
export SPRING_AI_OPENAI_BASE_URL=你的 OpenAI 兼容网关地址
export SPRING_AI_ANTHROPIC_API_KEY=你的真实 Anthropic Key
export SPRING_AI_ANTHROPIC_BASE_URL=你的 Anthropic 兼容网关地址
```

说明：

- 如果你直连官方服务，`SPRING_AI_OPENAI_BASE_URL` 可用默认值 `https://api.openai.com`
- 如果你直连官方服务，`SPRING_AI_ANTHROPIC_BASE_URL` 可用默认值 `https://api.anthropic.com`
- 如果你走兼容平台，这两个 URL 一般都要改

如果你不开真实模型调用，这几项都不是启动阻塞项。

### 场景 C：你要启用向量检索

这时需要额外处理两件事：

1. 数据库里存在 `vector` 类型
2. 显式开启向量检索

建议补：

```bash
docker exec vector_db psql -U postgres -d ai-rag-knowledge \
  -c "CREATE EXTENSION IF NOT EXISTS vector;"

export LATTICE_QUERY_SEARCH_VECTOR_ENABLED=true
```

说明：

- 当前基线脚本会在检测到 `vector` 类型存在时，才创建 `article_vector_index`
- 所以向量检索不是默认强制项

### 场景 D：你要指定全文检索配置

当前默认：

- 首选：`jiebacfg`
- 回退：`simple`

只有当你明确知道数据库里没有 `jiebacfg`，或者你要换自己的分词配置时，才需要改：

```bash
export LATTICE_QUERY_SEARCH_FTS_CONFIG=simple
export LATTICE_QUERY_SEARCH_FTS_FALLBACK=simple
```

## 7. 一套可以直接照抄的本地启动命令

如果你的目标只是“把项目在本机跑起来”，可以直接按下面执行：

```bash
cd /path/to/lattice-java

./scripts/reset-lattice-schema.sh

export SPRING_DATASOURCE_URL="jdbc:postgresql://127.0.0.1:5432/ai-rag-knowledge?currentSchema=lattice"
export SPRING_DATASOURCE_USERNAME=postgres
export SPRING_DATASOURCE_PASSWORD=postgres
export LATTICE_REDIS_HOST=127.0.0.1
export LATTICE_REDIS_PORT=6379
export SPRING_AI_OPENAI_BASE_URL=https://api.openai.com
export SPRING_AI_ANTHROPIC_BASE_URL=https://api.anthropic.com

mvn -s .codex/maven-settings.xml spring-boot:run
```

启动后再执行：

```bash
curl http://127.0.0.1:8080/actuator/health
```

如果你本机还没有 `vector_db` / `redis`，就在最前面补一行：

```bash
docker compose up -d postgres redis
```

### 7.1 启动后的第一轮页面验证

服务起来后，建议立刻按下面顺序做一次最小页面验收：

1. 打开 `http://127.0.0.1:8080/admin/settings`
2. 先配置连接、模型、Agent 绑定；如果只是临时导入私有 Git 仓库，也可以稍后直接在 `/admin` 的导入卡里补凭据
3. 再打开 `http://127.0.0.1:8080/admin`
4. 在“资料导入”里上传文件，或直接导入 Git 仓库；Git 导入卡已区分“公开仓库 / 私有仓库”，私有模式支持就地保存凭据
5. 切到“当前处理任务”确认任务状态；上传、Git / 目录资料源同步、direct compile 都应进入统一列表
6. 最后打开 `http://127.0.0.1:8080/admin/ask` 直接提问

当前页面职责已经拆开：

- `/admin/settings`
  - 面向内部维护
  - 只保留管理员设置、解析配置、资料源维护与凭据
- `/admin/developer-access`
  - 面向开发者接入
  - 只保留 CLI、HTTP API、MCP 模板、验证步骤与 FAQ
- `/admin`
  - 面向普通用户
  - 只负责知识库管理、资料导入、同步状态与入库内容
- `/admin/ask`
  - 面向普通用户
  - 只负责知识问答

### 7.2 知识输入主链最小回归

如果你想确认“知识输入系统一体化”主链没有回退，建议在上面的页面验收之后继续做下面这组最小 smoke：

1. 运行 `./scripts/smoke-knowledge-input.sh`
2. 打开 `/admin`
3. 验证“当前状态”“资料导入”“当前处理任务”“资料源与文件”“已入库内容”五个区块都能正常加载
4. 确认“资料导入”里同时存在“本地文件上传”和“Git 仓库导入”两张入口卡片
5. 打开 `/admin/settings`
6. 验证“模型与角色”“文档解析”“资料源维护”都能正常加载，并且私有仓库凭据列表可见
7. 打开 `/admin/ask`
8. 验证在存在活动 `source-runs` 或 `WAIT_CONFIRM` 时，页面提示会同步变化

这轮最小回归的关键观察点是：

- 上传后优先看到 `source-runs` 状态，而不是只看 compile job；关联同步触发的 compile 不应重复展示成第二张卡
- `direct compile` 必须以 `STANDALONE_COMPILE` 出现在“当前处理任务”
- 长时间没有心跳或进度推进的运行中 compile 必须显示 `STALLED`
- 多资料源场景下，文章详情应按 `articleKey + sourceId` 读取，不应因重复 `conceptId` 误命中
- 资料源详情页应能看到“处理历史”，历史中应同时包含资料源同步 run 与绑定该资料源的 standalone compile
- 资料源详情页应能看到 `parseMode / parseProvider`
- AI 页资料源凭据列表只返回脱敏值，不显示明文 secret
- CLI 根帮助中应出现 `source-list`、`source-sync`
- MCP 工具中应存在 `lattice_source_list`、`lattice_source_sync`

## 8. 常见启动失败原因

### 8.1 还是在用旧迁移结果

现象：

- 本地还有 `target/classes/db/migration`
- 启动参数或环境变量里还带着 `SPRING_FLYWAY_*`
- 数据库里还保留旧 `flyway_schema_history`

原因：

- 旧版迁移产物或 schema 历史还在

处理：

```bash
mvn clean
```

必要时重建 schema：

```bash
./scripts/reset-lattice-schema.sh
```

### 8.2 数据库能连上，但业务表没建出来

原因通常是：

- `currentSchema` 不是 `lattice`
- 还没有执行 `scripts/reset-lattice-schema.sh`
- 手动执行 DDL 失败但没有处理错误

### 8.3 Redis 连接失败

优先检查：

- `docker ps --format '{{.Names}}'`
- `LATTICE_REDIS_HOST`
- `LATTICE_REDIS_PORT`

### 8.4 真实 LLM 调用失败

如果你已经切到真实 Key，仍然失败，优先检查：

- Key 是否有效
- 当前代理 / 网络是否能访问模型服务
- 你是否真的需要开启真实调用，而不是只做本地启动验证

## 9. 最终建议

对当前仓库，最实用的启动口径是：

1. 先把 PostgreSQL 和 Redis 跑起来
2. 创建 `lattice` schema
3. 必要时执行一次 `mvn clean`
4. 用 `mvn -s .codex/maven-settings.xml spring-boot:run` 启动
5. 用 `/actuator/health` 和 `\dt lattice.*` 做最小验收
