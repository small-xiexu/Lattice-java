# Lattice 数据库表结构详解

本文基于当前仓库的 Flyway 迁移脚本 `src/main/resources/db/migration/V1__baseline_schema.sql` 与 `src/main/resources/db/migration/V2__query_vector_settings.sql`，整理 `lattice` schema 的现行结构、分层职责、关键关联与使用备注。

## 1. 文档依据

- 建表与注释来源：`src/main/resources/db/migration/V1__baseline_schema.sql`、`src/main/resources/db/migration/V2__query_vector_settings.sql`
- 运行配置来源：`src/main/resources/application.yml`、后台持久化配置表
- 运行语义来源：`src/main/java/com/xbk/lattice/**`

## 2. 整体分层

当前 `lattice` schema 包含 **19 张业务表**。Flyway 自带的 `flyway_schema_history` 由框架维护，不计入业务表数量。

### 2.1 知识与索引层

- `articles`
- `article_chunks`
- `article_snapshots`
- `article_vector_index`
- `source_files`
- `source_file_chunks`

### 2.2 查询闭环层

- `pending_queries`
- `contributions`

### 2.3 编译运行层

- `synthesis_artifacts`
- `compile_jobs`
- `compile_job_steps`

### 2.4 LLM 配置中心层

- `llm_provider_connections`
- `llm_model_profiles`
- `agent_model_bindings`
- `execution_llm_snapshots`

### 2.5 Query 检索配置层

- `query_vector_settings`

### 2.6 治理与版本层

- `quality_metrics_history`
- `repo_snapshots`
- `repo_snapshot_items`

## 3. 几个先看懂的关键点

### 3.1 `articles` 与 `source_files` 是两套主真相

- `articles` 表示“编译后的知识文章”
- `source_files` 表示“编译输入的原始来源文件”
- 查询链路、治理链路、Vault 投影都会同时依赖这两层数据

### 3.2 `article_snapshots` 与 `repo_snapshots` 不是一回事

- `article_snapshots` 是单篇文章级历史版本
- `repo_snapshots` + `repo_snapshot_items` 是整库级快照
- 两者同时存在，分别服务文章回滚与整库版本管理

### 3.3 LLM 路由已进入数据库配置中心

- `llm_provider_connections` 管连接
- `llm_model_profiles` 管模型参数
- `agent_model_bindings` 管场景与 Agent 角色绑定
- `execution_llm_snapshots` 管运行时冻结快照

这套结构已经不只服务 compile 场景，也为 query 侧复用同一套模型中心预留了统一底座。

### 3.4 Query 向量配置已进入后台持久化

- `query_vector_settings` 保存 query 向量开关、embedding 模型与期望维度
- admin 保存后会直接影响运行时 embeddings 请求
- 这张表不替代 `article_vector_index`，而是负责驱动“是否启用向量检索、按哪个模型生成向量”

### 3.5 `article_vector_index` 是条件创建表

- 只有数据库安装了 `pgvector` 类型时才会创建
- 因此它是“可选增强表”，不是所有环境都必然存在

## 4. 表总览

| 表名 | 分层 | 作用 | 关键键 / 关联 |
|------|------|------|---------------|
| `articles` | 知识与索引 | 编译后的知识文章主表 | 主键 `id`；业务唯一键 `concept_id` |
| `article_chunks` | 知识与索引 | 文章分块 | `article_id -> articles.id` |
| `article_snapshots` | 知识与索引 | 文章历史快照 | 主键 `snapshot_id`；按 `concept_id` 标识历史归属 |
| `article_vector_index` | 知识与索引 | 文章向量索引 | `concept_id -> articles.concept_id`；仅在 `pgvector` 可用时创建 |
| `source_files` | 知识与索引 | 源文件主表 | 主键 `id`；业务唯一键 `file_path` |
| `source_file_chunks` | 知识与索引 | 源文件分块 | `file_path -> source_files.file_path`；唯一键 `(file_path, chunk_index)` |
| `pending_queries` | 查询闭环 | 待确认问答记录 | 主键 `query_id`；保存问题、答案、命中概念与源文件路径 |
| `contributions` | 查询闭环 | 已确认问答贡献 | 主键 `id(UUID)`；沉淀 confirmed 问答与 correction 记录 |
| `synthesis_artifacts` | 编译运行 | 编译合成产物 | 主键 `id`；`artifact_type` 唯一 |
| `compile_jobs` | 编译运行 | 编译任务主表 | 主键 `job_id`；记录状态、模式、统计与时间戳 |
| `compile_job_steps` | 编译运行 | 编译步骤执行日志 | 主键 `id`；逻辑关联 `job_id -> compile_jobs.job_id` |
| `llm_provider_connections` | LLM 配置中心 | Provider 连接配置 | 主键 `id`；唯一键 `connection_code` |
| `llm_model_profiles` | LLM 配置中心 | 模型档案配置 | 主键 `id`；`connection_id -> llm_provider_connections.id`；唯一键 `model_code` |
| `agent_model_bindings` | LLM 配置中心 | 场景与 Agent 角色绑定 | 主键 `id`；唯一键 `(scene, agent_role)`；关联模型档案 |
| `execution_llm_snapshots` | LLM 配置中心 | 运行时模型冻结快照 | 主键 `id`；唯一键 `(scope_type, scope_id, scene, agent_role)`；关联连接、模型、绑定 |
| `query_vector_settings` | Query 检索配置 | Query 向量开关与 embedding 基线 | 主键 `config_scope`；当前固定 `default` |
| `quality_metrics_history` | 治理与版本 | 质量指标历史 | 主键 `id`；按 `measured_at` 追踪历史采样 |
| `repo_snapshots` | 治理与版本 | 整库快照主表 | 主键 `id`；记录触发事件、Git 提交与文章总数 |
| `repo_snapshot_items` | 治理与版本 | 整库快照明细 | 主键 `id`；`snapshot_id -> repo_snapshots.id` |

## 5. 关键关系与运行语义

### 5.1 文章与分块

- `articles` 保存最终知识文章
- `article_chunks` 保存文章切片，支持检索、重建与治理
- `article_snapshots` 通过触发器和业务流程保留文章历史

### 5.2 来源文件与证据链

- `source_files` 保存源文件正文、元数据和路径
- `source_file_chunks` 保存源文件切片
- 查询纠错、证据回放、受控同步都依赖这两张表

### 5.3 查询闭环

- 查询入口先写 `pending_queries`
- 用户 `confirm` 后写入 `contributions`
- `discard` 则只清理 pending，不进入长期沉淀

### 5.4 编译作业与步骤追踪

- `compile_jobs` 表示一次 compile 请求
- `compile_job_steps` 记录 Graph 节点或步骤的执行日志
- `synthesis_artifacts` 保存编译过程中生成的聚合产物

### 5.5 LLM 配置与快照冻结

- 管理台通过 `llm_provider_connections`、`llm_model_profiles`、`agent_model_bindings` 维护配置中心
- 运行时按场景和 Agent 角色命中绑定
- 命中结果冻结到 `execution_llm_snapshots`
- 后续节点按快照选路，避免任务执行过程中漂移

### 5.6 Query 向量配置与向量索引

- `query_vector_settings` 保存 query 侧向量开关、embedding 模型与维度基线
- `article_vector_index` 保存按文章聚合后的向量索引结果
- 两者共同决定“是否启用向量检索”与“当前索引是否需要重建”

### 5.7 治理与整库版本

- `quality_metrics_history` 保存质量指标的时间序列
- `repo_snapshots` 保存整库级时间点
- `repo_snapshot_items` 保存整库快照的实体明细

## 6. 使用备注

### 6.1 判断“当前真实表结构”时，以 Flyway 为准

如果文档、管理台或历史说明与库结构冲突，默认以当前 Flyway 迁移脚本为准。

### 6.2 `article_vector_index` 不应被当成必备表

没有 `pgvector` 的环境里，相关功能应退化，但不应导致主链启动失败。

### 6.3 `compile_job_steps` 当前通过逻辑键关联作业

它通过 `job_id` 与 `compile_jobs` 对齐，但当前迁移里没有显式外键，排障时要注意这一点。

### 6.4 `flyway_schema_history` 属于框架维护表

排查迁移问题时会用到它，但它不属于业务域建模的一部分。
