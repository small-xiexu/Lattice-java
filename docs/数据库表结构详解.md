# Lattice 数据库表结构详解

本文基于当前仓库唯一的 Flyway 基线脚本 [`V1__baseline_schema.sql`](../src/main/resources/db/migration/V1__baseline_schema.sql)，整理 `lattice` schema 的现行结构、分层职责、关键约束与运行语义。

## 1. 文档依据

- 建表与注释来源：[`V1__baseline_schema.sql`](../src/main/resources/db/migration/V1__baseline_schema.sql)
- 运行配置来源：[`application.yml`](../src/main/resources/application.yml)
- 运行语义来源：`src/main/java/com/xbk/lattice/**`

判断“当前真实表结构”时，如果文档、管理台或历史说明与数据库结构冲突，默认以 Flyway 基线脚本为准。

## 2. 整体总览

当前 `lattice` schema 包含：

- **26 张固定业务表**
- **2 张条件创建的向量索引表**
- `flyway_schema_history` 由框架维护，不计入业务表

只有当数据库已安装 `pgvector` 类型时，才会额外创建：

- `article_vector_index`
- `article_chunk_vector_index`

### 2.1 资料源与同步层

- `source_credentials`
- `knowledge_sources`
- `source_sync_runs`
- `source_snapshots`

### 2.2 知识资产与证据层

- `articles`
- `article_chunks`
- `article_snapshots`
- `source_files`
- `source_file_chunks`
- `article_source_refs`
- `article_vector_index`（条件创建）
- `article_chunk_vector_index`（条件创建）

### 2.3 编译与合成层

- `compile_jobs`
- `compile_job_steps`
- `synthesis_artifacts`

### 2.4 模型与解析配置层

- `agent_model_bindings`

- `document_parse_connections`

## 3. 几个先看懂的关键点

### 3.1 资料源已经是正式的一等公民

- `knowledge_sources` 表示正式资料源，而不只是一个目录路径
- `source_sync_runs` 表示某次资料同步或导入运行
- `source_files` / `source_file_chunks` 承接同步后的源文件与分块落库
- `compile_jobs` 可以直接关联 `source_id` 与 `source_sync_run_id`

### 3.2 文章唯一性已经是 source-aware

`articles` 当前不是简单按 `concept_id` 唯一，而是：

- 全局唯一键：`article_key`
- 资料源内唯一键：`(source_id, concept_id)`

这意味着同一个 `concept_id` 可以在不同资料源中共存，前提是 `source_id` 不同。

### 3.3 源文件唯一性也是 source-aware

`source_files` 的核心唯一约束不是单列 `file_path`，而是：

- `(source_id, relative_path)`

同名或同路径文件在不同资料源中可以并存。

### 3.4 Query 配置分成两层

- `query_vector_settings` 决定是否启用向量检索，以及当前使用哪个 embedding profile
- `query_retrieval_settings` 决定多路召回与融合权重，例如 FTS、Source、Contribution、Vector 的权重和 `rrf_k`

这两张表分别解决“开不开”和“怎么融”的问题。

### 3.5 向量索引是条件创建能力

- 没有 `pgvector` 时，向量索引表不会创建
- 主链仍可依赖 FTS / Source / Article 等召回方式运行
- 因此向量表属于增强能力，不是启动硬前置

### 3.6 当前基线里已经包含默认种子数据

Flyway 基线会自动写入：

- `knowledge_sources` 中的默认资料源：`legacy-default`
- `query_retrieval_settings` 中的默认单例配置：`id=1`

这两项都属于当前 schema 的正式组成部分。

## 4. 表总览

### 4.1 资料源与同步层

| 表名 | 作用 | 主键 / 唯一键 / 关键关联 |
| --- | --- | --- |
| `source_credentials` | 资料源访问凭据 | 主键 `id`；唯一键 `credential_code` |
| `knowledge_sources` | 资料源主表 | 主键 `id`；唯一键 `source_code`；`last_sync_run_id -> source_sync_runs.id` |
| `source_sync_runs` | 资料源同步运行记录 | 主键 `id`；`source_id -> knowledge_sources.id`；`matched_source_id -> knowledge_sources.id`；`compile_job_id -> compile_jobs.job_id` |
| `source_snapshots` | 资料源级快照 | 主键 `id`；`source_id -> knowledge_sources.id`；`sync_run_id -> source_sync_runs.id` |

### 4.2 知识资产与证据层

| 表名 | 作用 | 主键 / 唯一键 / 关键关联 |
| --- | --- | --- |
| `articles` | 编译后的知识文章主表 | 主键 `id`；唯一键 `article_key`、`(source_id, concept_id)`；`source_id -> knowledge_sources.id` |
| `article_chunks` | 文章分块 | 主键 `id`；`article_id -> articles.id` |
| `article_snapshots` | 文章历史快照 | 主键 `snapshot_id`；按 `concept_id`、`article_key` 建索引；`source_id -> knowledge_sources.id` |
| `source_files` | 源文件主表 | 主键 `id`；唯一键 `(source_id, relative_path)`；`source_id -> knowledge_sources.id`；`source_sync_run_id -> source_sync_runs.id` |
| `source_file_chunks` | 源文件分块 | 主键 `id`；唯一键 `(source_file_id, chunk_index)`；`source_file_id -> source_files.id` |
| `article_source_refs` | 文章到源文件的证据引用关系 | 主键 `id`；唯一键 `(article_key, source_file_id, ref_type)`；`source_id -> knowledge_sources.id`；`source_file_id -> source_files.id` |
| `article_vector_index` | 文章级向量索引 | 条件创建；主键 `article_key`；`article_key -> articles.article_key`；`model_profile_id -> llm_model_profiles.id` |
| `article_chunk_vector_index` | 文章分块级向量索引 | 条件创建；主键 `article_chunk_id`；`article_chunk_id -> article_chunks.id`；`article_id -> articles.id`；`model_profile_id -> llm_model_profiles.id` |

### 4.3 编译与合成层

| 表名 | 作用 | 主键 / 唯一键 / 关键关联 |
| --- | --- | --- |
| `compile_jobs` | 编译任务主表 | 主键 `job_id`；`source_id -> knowledge_sources.id`；`source_sync_run_id -> source_sync_runs.id` |
| `compile_job_steps` | 编译步骤执行日志 | 主键 `id`；唯一键 `step_execution_id`；按 `(job_id, sequence_no)` 建索引 |
| `synthesis_artifacts` | 编译后的聚合产物 | 主键 `id`；唯一键 `artifact_type` |

### 4.4 模型与解析配置层

| 表名 | 作用 | 主键 / 唯一键 / 关键关联 |
| --- | --- | --- |
| `llm_provider_connections` | LLM Provider 连接配置 | 主键 `id`；唯一键 `connection_code` |
| `llm_model_profiles` | 模型档案配置 | 主键 `id`；唯一键 `model_code`；`connection_id -> llm_provider_connections.id` |
| `agent_model_bindings` | 场景与 Agent 角色绑定 | 主键 `id`；唯一键 `(scene, agent_role)`；`primary_model_profile_id -> llm_model_profiles.id`；`fallback_model_profile_id -> llm_model_profiles.id` |
| `execution_llm_snapshots` | 运行时模型快照冻结 | 主键 `id`；唯一键 `(scope_type, scope_id, scene, agent_role)`；`binding_id -> agent_model_bindings.id`；`model_profile_id -> llm_model_profiles.id`；`connection_id -> llm_provider_connections.id` |
| `document_parse_connections` | 文档解析供应商连接配置 | 主键 `id`；唯一键 `connection_code` |
| `document_parse_route_policies` | 文档解析路由策略 | 主键 `id`；唯一键 `policy_scope`；`image_connection_id -> document_parse_connections.id`；`scanned_pdf_connection_id -> document_parse_connections.id`；`cleanup_model_profile_id -> llm_model_profiles.id` |

### 4.5 Query 与反馈层

| 表名 | 作用 | 主键 / 唯一键 / 关键关联 |
| --- | --- | --- |
| `pending_queries` | 待确认问答记录 | 主键 `query_id` |
| `contributions` | 已确认问答贡献 | 主键 `id(UUID)` |
| `query_vector_settings` | Query 向量检索配置 | 主键 `config_scope`；`embedding_model_profile_id -> llm_model_profiles.id` |
| `query_retrieval_settings` | Query 检索融合配置 | 主键 `id`；当前按单例配置使用，默认 `id=1` |

### 4.6 治理与版本层

| 表名 | 作用 | 主键 / 唯一键 / 关键关联 |
| --- | --- | --- |
| `quality_metrics_history` | 质量指标时间序列 | 主键 `id` |
| `repo_snapshots` | 整库快照主表 | 主键 `id` |
| `repo_snapshot_items` | 整库快照明细 | 主键 `id`；`snapshot_id -> repo_snapshots.id` |

## 5. 关键关系与运行语义

### 5.1 资料源到知识资产的主链

核心链路大致是：

`knowledge_sources -> source_sync_runs -> source_files -> articles`

补充关系：

- `source_sync_runs.compile_job_id` 可以指向某次 compile
- `compile_jobs.source_sync_run_id` 也会反向挂回资料同步运行
- `source_snapshots` 保存资料源级别的快照结果

### 5.2 文章、快照和证据的关系

- `articles` 是当前生效中的文章真相
- `article_snapshots` 记录文章历史版本
- `article_chunks` 保存文章分块，支撑检索与向量化
- `article_source_refs` 负责把文章和源文件之间的证据关系显式落库

### 5.3 待确认问答与贡献沉淀

- 查询入口先写 `pending_queries`
- 用户执行 `confirm` 后沉淀为 `contributions`
- `discard` 会结束 pending 生命周期，但不会进入长期知识沉淀

### 5.4 模型中心与运行时快照

模型选路链路是：

`llm_provider_connections -> llm_model_profiles -> agent_model_bindings -> execution_llm_snapshots`

含义分别是：

- 连接层：Provider 地址与密钥
- 模型层：模型名、类型、维度、温度、价格等参数
- 绑定层：某个 `scene + agent_role` 应走哪条模型路由
- 快照层：某次真实运行时到底命中了哪条连接、模型与绑定

### 5.5 文档解析配置独立于 LLM 模型中心

- `document_parse_connections` 管 OCR / Document AI 供应商连接
- `document_parse_route_policies` 管图片 OCR、扫描 PDF OCR、识别后整理与降级策略

它们和 LLM 模型中心关联，但已经从旧版“单一默认连接”模型切换为“连接管理 + 路由策略”模型。

### 5.6 Query 检索配置分成“开关”和“融合”两张表

- `query_vector_settings` 管向量检索是否启用，以及当前 embedding profile
- `query_retrieval_settings` 管并行召回、权重与 RRF 融合参数

因此 Query 检索行为不是写死在代码里，而是有后台持久化配置的。

### 5.7 向量索引是双层结构

如果启用了 `pgvector`：

- `article_vector_index` 负责文章级向量
- `article_chunk_vector_index` 负责 chunk 级向量

两张表都挂在 `llm_model_profiles` 上，因此切换 embedding profile 后，索引重建逻辑需要一起考虑。

### 5.8 治理与整库版本

- `quality_metrics_history` 保存质量指标采样
- `repo_snapshots` 保存整库快照头信息
- `repo_snapshot_items` 保存快照明细实体

这套结构支撑文章级治理之外的整库视角治理。

## 6. 使用备注

### 6.1 先看 `article_key`，再看 `concept_id`

当前系统里很多地方已经是 source-aware 设计。排障时如果只看 `concept_id`，容易忽略不同资料源下同概念的并存情况。涉及文章唯一定位时，优先看：

- `article_key`
- `source_id + concept_id`

### 6.2 `compile_job_steps` 有唯一执行 ID，但仍无显式外键

`compile_job_steps.step_execution_id` 当前有唯一索引，但 `job_id` 仍然是逻辑关联，没有显式外键约束。排查孤儿步骤或历史日志时要注意这一点。

### 6.3 `article_vector_index` 与 `article_chunk_vector_index` 都不是必备表

没有 `pgvector` 的环境里，这两张表不会出现。这属于预期行为，不应被误判成迁移失败。

### 6.4 `query_retrieval_settings` 按单例配置使用

当前这张表通过默认 `id=1` 落一条全局配置，不是多租户多作用域设计。后续如果要扩展成多作用域配置，需要额外演进表结构。

### 6.5 `flyway_schema_history` 不是业务域建模的一部分

排查迁移问题时需要看它，但它不属于业务表，也不应纳入领域对象分析。
