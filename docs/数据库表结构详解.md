# Lattice 数据库表结构详解

本文面向项目维护者、接口开发者和 pgAdmin 使用者，说明当前 `lattice` schema 下每一张表的职责、字段含义、关键关系，以及我结合代码确认到的行为备注。

## 1. 文档依据

- 建表与字段注释来源：`src/main/resources/db/migration/V1__baseline_schema.sql`
- 实际结构来源：本机 `vector_db` 容器中 `ai-rag-knowledge.lattice` 的在线 schema
- 运行语义来源：`src/main/java/com/xbk/lattice/**` 下的仓储、编译、查询、治理与 Vault 回写代码

## 2. 整体分层

当前 `lattice` schema 共有 15 张表，可以按职责分成 5 组：

1. 源文件层
   - `source_files`
   - `source_file_chunks`
2. 知识文章层
   - `articles`
   - `article_chunks`
   - `article_snapshots`
   - `article_vector_index`
3. 查询反馈层
   - `pending_queries`
   - `contributions`
4. 运行与治理层
   - `compile_jobs`
   - `llm_usage`
   - `synthesis_artifacts`
   - `quality_metrics_history`
   - `repo_snapshots`
   - `repo_snapshot_items`
5. 基础设施层
   - `flyway_schema_history`

## 3. 几个先看懂的关键点

### 3.1 `articles.review_status` 和 `pending_queries.review_status` 不是一回事

- `articles.review_status` 是文章编译/治理后的文章审查状态，代码里常见值有：
  - `pending`
  - `passed`
  - `needs_human_review`
  - `needs_review`
- `pending_queries.review_status` 是问答查询链路的审查状态，来自 `ReviewStatus` 枚举，常见值是大写：
  - `PASSED`
  - `ISSUES_FOUND`
  - `PARSE_RESCUED`
  - `PARSE_FAILED`
  - `TIMEOUT_FALLBACK`

### 3.2 `articles.lifecycle` 当前存在大小写混用

- 编译链路默认写入 `ACTIVE`
- 生命周期治理服务会写入 `active`、`deprecated`、`archived`
- 业务上它们表达的是同一类状态，但库里当前没有统一大小写约束

### 3.3 `article_vector_index` 是可选表

- 这张表只有数据库安装了 `pgvector` 类型时才会创建
- 当前你的本机环境已经创建成功

### 3.4 `article_snapshots` 有两类来源

- 自动快照：`articles` 的插入/更新触发器自动写入，`snapshot_reason=article_upsert`
- 手工快照：治理回滚、人工纠错等流程会主动插入，`snapshot_reason` 会带业务语义

## 4. 表详解

## 4.1 `articles`

### 作用

知识文章主表，是整个系统最核心的业务表。编译链路、增量更新、人工纠错、生命周期治理、Vault 回写、查询检索、管理台展示，几乎都围绕它展开。

### 主要关系

- 主键：`id`
- 业务唯一键：`concept_id`
- 被 `article_chunks.article_id` 外键引用
- 被 `article_vector_index.concept_id` 外键引用
- 被触发器监听，写入 `article_snapshots`

### 字段说明

| 字段 | 类型 | 作用 | 备注 |
| --- | --- | --- | --- |
| `id` | `bigserial` | 文章行主键 | 纯数据库主键，业务代码更多使用 `concept_id` |
| `concept_id` | `varchar(128)` | 概念唯一标识 | 例如概念 slug；全文系统、管理接口、快照、向量索引都依赖它 |
| `title` | `varchar(255)` | 文章标题 | 展示标题，也参与检索和向量 embedding 文本构造 |
| `content` | `text` | Markdown 正文 | 通常包含 frontmatter 和正文；Vault 回写会重建受管区块 |
| `lifecycle` | `varchar(32)` | 生命周期状态 | 当前观察到的值有 `ACTIVE`、`active`、`deprecated`、`archived` |
| `compiled_at` | `timestamptz` | 最近一次编译时间 | 编译、增量增强时会刷新；生命周期切换一般不改它 |
| `created_at` | `timestamptz` | 首次入库时间 | 由数据库默认值生成 |
| `updated_at` | `timestamptz` | 记录更新时间 | 表定义里有该字段，但当前 `ArticleJdbcRepository.upsert` 没有显式更新它，现状更接近“首次入库时刻” |
| `source_paths` | `text[]` | 来源文件路径集合 | 用于溯源、质量统计、向量 embedding 文本拼接、纠错取证 |
| `metadata_json` | `jsonb` | 扩展元数据 | 常见承载 `description`、`structured`、`lifecycle`、`upstream_corrections` 等信息 |
| `summary` | `text` | 文章摘要 | 编译摘要，管理台和向量索引都会用到 |
| `referential_keywords` | `text[]` | 明确性关键词 | 例如业务码、端口、枚举值、表名、类名等 |
| `depends_on` | `text[]` | 上游依赖概念列表 | 用于依赖图、传播纠错、正文受管区块同步 |
| `related` | `text[]` | 相关概念列表 | 用于关系图和正文“Related Concepts”区块 |
| `confidence` | `varchar(16)` | 内容置信度 | 当前常见值是 `medium`、`high`；库里没有硬枚举约束 |
| `review_status` | `varchar(32)` | 文章审查状态 | 当前常见值是 `pending`、`passed`、`needs_human_review`、`needs_review` |

### 运行备注

- `concept_id` 是这张表真正的业务身份标识。
- `content` 和结构化字段并不是二选一，而是“双写”模式：正文保留 Markdown，人机都能读；结构化字段便于治理和查询。
- `metadata_json` 目前是扩展能力的主要承载位，后续新增治理字段时大概率还会继续往这里放。

## 4.2 `source_files`

### 作用

源文件主表，保存编译输入材料的基础信息、全文内容和扩展元数据。它代表“知识是从哪些原始文件来的”。

### 主要关系

- 主键：`id`
- 唯一键：`file_path`
- 被 `source_file_chunks.file_path` 通过路径外键引用

### 字段说明

| 字段 | 类型 | 作用 | 备注 |
| --- | --- | --- | --- |
| `id` | `bigserial` | 源文件行主键 | 主要用于数据库内部标识 |
| `file_path` | `varchar(512)` | 源文件规范化路径 | 业务唯一键；分块表通过它关联，而不是通过 `id` |
| `content_preview` | `text` | 内容预览 | 一般是简短摘录，方便列表展示和快速识别 |
| `format` | `varchar(32)` | 文件格式 | 如 `md`、`txt` 等 |
| `file_size` | `bigint` | 文件大小（字节） | 用于保留输入文件基础属性 |
| `indexed_at` | `timestamptz` | 最近索引时间 | 每次 `upsert` 更新源文件时会刷新 |
| `content_text` | `text` | 源文件全文 | 后续分块、检索、纠错取证都依赖它 |
| `metadata_json` | `jsonb` | 源文件扩展元数据 | 例如测试里出现的 `pageCount` |
| `is_verbatim` | `boolean` | 是否逐字提取 | `true` 时表示正文更接近原文逐字抽取 |
| `raw_path` | `varchar(512)` | 原始文件路径 | 用于保留导入前或未规范化前的路径 |

### 运行备注

- `file_path` 是这张表的业务主键，所以 `source_file_chunks` 也直接用它做关联。
- `content_text` 才是后续检索和分块的真正输入，`content_preview` 只是缩略信息。

## 4.3 `article_chunks`

### 作用

文章分块表，把 `articles.content` 拆成更适合检索和重建的 chunk。它主要服务于文章层的分块查询和管理能力。

### 主要关系

- 主键：`id`
- 外键：`article_id -> articles.id`

### 字段说明

| 字段 | 类型 | 作用 | 备注 |
| --- | --- | --- | --- |
| `id` | `bigserial` | 分块主键 | 单条 chunk 的数据库主键 |
| `article_id` | `bigint` | 关联文章主键 | 指向 `articles.id`，删除文章时级联删除 |
| `chunk_text` | `text` | chunk 文本内容 | 由 `SemanticChunker` 从文章正文切出来 |
| `chunk_index` | `integer` | chunk 顺序号 | 用于按原顺序还原文章分块 |
| `created_at` | `timestamptz` | 分块生成时间 | 由数据库默认值生成 |

### 运行备注

- 当前仓储的写法是“先删后插”的替换模式。
- 表上没有 `(article_id, chunk_index)` 唯一约束，正常流程靠替换写入避免重复。

## 4.4 `source_file_chunks`

### 作用

源文件分块表，把 `source_files.content_text` 切成 chunk，供源文件检索、问答纠错取证和重建使用。

### 主要关系

- 主键：`id`
- 外键：`file_path -> source_files.file_path`
- 唯一约束：`(file_path, chunk_index)`

### 字段说明

| 字段 | 类型 | 作用 | 备注 |
| --- | --- | --- | --- |
| `id` | `bigserial` | 分块主键 | 单条源文件 chunk 的主键 |
| `file_path` | `varchar(512)` | 关联源文件路径 | 直接关联到 `source_files.file_path` |
| `chunk_index` | `integer` | 分块顺序号 | 同一路径下唯一 |
| `chunk_text` | `text` | 分块文本 | 用于源文件检索和 pending 纠错证据回放 |
| `is_verbatim` | `boolean` | 是否逐字分块 | 从 `source_files.is_verbatim` 继承下来 |
| `indexed_at` | `timestamptz` | 分块索引时间 | 由数据库默认值生成 |

### 运行备注

- `SourceSearchService` 直接遍历它做源文件层命中打分。
- `PendingQueryService.correct()` 也会从这里拿证据，用于纠错后重生成答案。

## 4.5 `article_snapshots`

### 作用

文章快照历史表，记录知识文章在不同时间点的完整状态，用于审计、回滚、治理和历史追踪。

### 主要关系

- 主键：`snapshot_id`
- 没有直接外键指向 `articles`
- 通过 `concept_id` 表示这是谁的历史版本

### 字段说明

| 字段 | 类型 | 作用 | 备注 |
| --- | --- | --- | --- |
| `snapshot_id` | `bigserial` | 快照主键 | 历史记录唯一标识 |
| `concept_id` | `varchar(255)` | 文章概念标识 | 对应 `articles.concept_id`，但未做外键约束 |
| `title` | `varchar(255)` | 快照标题 | 当时版本的标题 |
| `content` | `text` | 快照正文 | 当时版本的完整 Markdown 内容 |
| `lifecycle` | `varchar(64)` | 快照生命周期状态 | 保存快照生成时的状态 |
| `compiled_at` | `timestamptz` | 当时版本的编译时间 | 直接复制自文章记录 |
| `source_paths` | `text[]` | 当时版本的来源路径 | 直接复制自文章记录 |
| `metadata_json` | `jsonb` | 当时版本的元数据 | 直接复制自文章记录 |
| `summary` | `text` | 快照摘要 | 允许为空 |
| `referential_keywords` | `text[]` | 当时版本的明确性关键词 | 直接复制自文章记录 |
| `depends_on` | `text[]` | 当时版本的上游依赖 | 直接复制自文章记录 |
| `related` | `text[]` | 当时版本的相关概念 | 直接复制自文章记录 |
| `confidence` | `varchar(32)` | 当时版本的置信度 | 允许为空 |
| `review_status` | `varchar(64)` | 当时版本的审查状态 | 允许为空 |
| `snapshot_reason` | `varchar(32)` | 快照原因 | 常见值有 `article_upsert`、`correction`，回滚时会带 `rollback|from:快照ID` |
| `captured_at` | `timestamptz` | 快照采集时间 | 真正的快照生成时刻 |

### 运行备注

- `articles` 上有触发器 `trg_capture_article_snapshot`，插入和更新都会自动留痕。
- 治理服务还会主动插入额外快照，所以这张表不只是触发器副本。
- 它不对 `articles` 做外键，是为了保留独立历史，不受主表删除/重建影响。

## 4.6 `article_vector_index`

### 作用

文章向量索引表，为语义检索保存 embedding。它是一篇文章一条向量记录的结构。

### 主要关系

- 主键：`concept_id`
- 外键：`concept_id -> articles.concept_id`

### 字段说明

| 字段 | 类型 | 作用 | 备注 |
| --- | --- | --- | --- |
| `concept_id` | `varchar(128)` | 关联文章概念标识 | 一篇文章仅保留一条当前向量 |
| `model_name` | `varchar(128)` | Embedding 模型名 | 用于区分使用了哪种向量模型 |
| `content_hash` | `varchar(64)` | 向量输入内容哈希 | 由标题、摘要、正文、来源路径做 SHA-256；未变化时可跳过重建向量 |
| `embedding` | `vector(1536)` | 文章向量 | 当前基线是 1536 维 |
| `updated_at` | `timestamptz` | 向量更新时间 | 每次向量 upsert 刷新 |

### 运行备注

- `ArticleVectorIndexService` 会先比较 `content_hash`，内容没变时不重算 embedding。
- 向量检索按距离排序，再回连 `articles` 取标题、正文和来源。
- 如果库里没有 `pgvector` 类型，这张表和对应功能会整体跳过。

## 4.7 `pending_queries`

### 作用

待确认问答表，承接查询接口返回后的人工确认闭环。它是“答案还没最终沉淀进知识库前”的暂存区。

### 主要关系

- 主键：`query_id`
- 无外键
- 确认后会沉淀到 `contributions`，然后从本表删除

### 字段说明

| 字段 | 类型 | 作用 | 备注 |
| --- | --- | --- | --- |
| `query_id` | `varchar(64)` | 待确认问答唯一标识 | 一次查询一条记录 |
| `question` | `text` | 用户原始问题 | 后续确认贡献时会沿用 |
| `answer` | `text` | 当前待确认答案 | 纠错后会被覆盖成修订版 |
| `selected_concept_ids` | `text[]` | 命中的文章概念 ID | 用于回放文章证据 |
| `source_file_paths` | `text[]` | 命中的源文件路径 | 用于回放源文件 chunk 证据 |
| `corrections` | `jsonb` | 纠错历史 | 常见结构含 `version`、`correction`、`previousAnswer`、`revisedAnswer`、`correctedAt` |
| `review_status` | `varchar(32)` | 查询审查状态 | 使用查询链路状态，不是文章状态；常见大写枚举值见文首 |
| `created_at` | `timestamptz` | 创建时间 | 首次生成 pending 时写入 |
| `expires_at` | `timestamptz` | 过期时间 | 当前服务默认 TTL 是 7 天 |

### 运行备注

- `PendingQueryService.correct()` 会重写 `answer` 并追加 `corrections`。
- 当前实现里纠错不会改 `review_status`，只刷新 `expires_at`。
- 管理台、MCP 和查询确认接口都围绕这张表运转。

## 4.8 `contributions`

### 作用

用户确认后的贡献沉淀表，保存已经确认的问答结果。它代表“最终采纳”的反馈，而不是临时草稿。

### 主要关系

- 主键：`id`
- 无外键

### 字段说明

| 字段 | 类型 | 作用 | 备注 |
| --- | --- | --- | --- |
| `id` | `uuid` | 贡献主键 | 确认时生成新的 UUID |
| `question` | `text` | 已确认问答对应问题 | 一般来自 `pending_queries.question` |
| `answer` | `text` | 最终确认答案 | 可能是原 pending 答案，也可能是人工导入最终答案 |
| `corrections` | `jsonb` | 纠错与导入历史 | 除了普通修订历史，还可能追加 `type=imported-answer` 的导入记录 |
| `confirmed_by` | `varchar(100)` | 确认人 | 常见来源有 `system`、`inspection-import` 或管理侧传入值 |
| `confirmed_at` | `timestamptz` | 确认时间 | 最终沉淀时间 |

### 运行备注

- `pending_queries` 确认后会写入本表，然后删除 pending 记录。
- 它更像“知识反馈台账”，不是文章正文表。

## 4.9 `llm_usage`

### 作用

LLM 调用用量台账，用于统计模型调用次数、token 消耗和美元成本。

### 主要关系

- 主键：`id`
- 无外键

### 字段说明

| 字段 | 类型 | 作用 | 备注 |
| --- | --- | --- | --- |
| `id` | `bigserial` | 用量记录主键 | 单条调用台账的主键 |
| `call_id` | `varchar(64)` | 调用链路标识 | 同一业务调用链可复用一个 ID |
| `model` | `varchar(64)` | 模型名称 | 例如具体的 LLM/Embedding 模型标识 |
| `purpose` | `varchar(32)` | 调用用途 | 由业务传入，常见风格如 `compile-article`、`synthesis-index`、`cross-validate` |
| `input_tokens` | `int` | 输入 token 数 | 成本统计依据之一 |
| `output_tokens` | `int` | 输出 token 数 | 成本统计依据之一 |
| `cost_usd` | `numeric(10,6)` | 美元成本 | 精确到 6 位小数 |
| `called_at` | `timestamptz` | 调用时间 | 默认当前时间 |

### 运行备注

- `AdminUsageQueryService` 会基于它做按模型、按用途的成本汇总。
- `purpose` 目前是自由字符串，没有枚举约束。

## 4.10 `synthesis_artifacts`

### 作用

合成产物表，保存知识库级的全局产物，如索引、时间线、取舍分析和缺口分析。

### 主要关系

- 主键：`id`
- 唯一键：`artifact_type`

### 字段说明

| 字段 | 类型 | 作用 | 备注 |
| --- | --- | --- | --- |
| `id` | `bigserial` | 产物主键 | 数据库内部主键 |
| `artifact_type` | `varchar(32)` | 产物类型 | 当前标准产物是 `index`、`timeline`、`tradeoffs`、`gaps` |
| `title` | `varchar(255)` | 产物标题 | 用于导出文件标题和展示 |
| `content` | `text` | 产物正文 | 典型是 Markdown 内容 |
| `compiled_at` | `timestamptz` | 产物生成时间 | 每次重新合成会刷新 |

### 运行备注

- 这张表按 `artifact_type` 做 upsert，所以同一类型始终只有一条“当前版本”。
- `VaultExportService` 会把它映射成 `index.md`、`timeline.md`、`tradeoffs.md`、`gaps.md`。

## 4.11 `compile_jobs`

### 作用

后台编译任务表，用来跟踪编译请求从排队到完成的整个生命周期。

### 主要关系

- 主键：`job_id`
- 无外键

### 字段说明

| 字段 | 类型 | 作用 | 备注 |
| --- | --- | --- | --- |
| `job_id` | `varchar(64)` | 编译任务唯一标识 | 队列和管理接口都用它 |
| `source_dir` | `varchar(1024)` | 编译输入目录 | 表示这次任务针对哪个源目录 |
| `incremental` | `boolean` | 是否增量编译 | `false` 表示全量，`true` 表示增量 |
| `orchestration_mode` | `varchar(32)` | 编排模式 | 当前默认值是 `service` |
| `status` | `varchar(32)` | 任务状态 | 常见值：`QUEUED`、`RUNNING`、`SUCCEEDED`、`FAILED` |
| `persisted_count` | `integer` | 已落库文章数 | 成功后用于展示本次任务产出量 |
| `error_message` | `text` | 最后一次错误信息 | 失败时记录 |
| `attempt_count` | `integer` | 已尝试次数 | `markRunning()` 时递增，`retry()` 不会清零 |
| `requested_at` | `timestamptz` | 请求入队时间 | 用于排序和排队 |
| `started_at` | `timestamptz` | 实际开始时间 | 抢占成功后写入 |
| `finished_at` | `timestamptz` | 结束时间 | 成功或失败时写入 |

### 运行备注

- 任务调度读取“最早一个 `QUEUED` 任务”作为下一个执行对象。
- `retry()` 会把失败任务重新置回 `QUEUED`，但保留累计重试次数。

## 4.12 `quality_metrics_history`

### 作用

质量指标历史表，保存每次质量度量的快照，用于管理台趋势图和治理分析。

### 主要关系

- 主键：`id`
- 无外键

### 字段说明

| 字段 | 类型 | 作用 | 备注 |
| --- | --- | --- | --- |
| `id` | `bigserial` | 历史记录主键 | 单次指标采样一条 |
| `measured_at` | `timestamptz` | 采样时间 | 记录生成时刻 |
| `total_articles` | `int` | 文章总数 | `articles` 当前总量 |
| `passed_articles` | `int` | 审查通过文章数 | `review_status=passed` 的文章数量 |
| `pending_articles` | `int` | 待审文章数 | 除通过、需人工复核外的其余文章数量 |
| `needs_review` | `int` | 需人工复核文章数 | 实际来自 `review_status=needs_human_review` 的文章数量 |
| `contributions` | `int` | 贡献总数 | `contributions` 表行数 |
| `source_count` | `int` | 源文件总数 | `source_files` 表行数 |
| `review_pass_rate` | `numeric(5,2)` | 审查通过率 | 百分比，范围通常是 `0-100` |
| `grounding_rate` | `numeric(5,2)` | 来源可追溯率 | 含 `source_paths` 的文章占比 |

### 运行备注

- 这张表是“时间序列快照”，不是当前值表。
- 趋势服务会拿一个时间窗口内的最早/最晚记录做差值。

## 4.13 `repo_snapshots`

### 作用

整库快照主表，记录某次“知识库整体状态冻结”的头信息。

### 主要关系

- 主键：`id`
- 被 `repo_snapshot_items.snapshot_id` 引用

### 字段说明

| 字段 | 类型 | 作用 | 备注 |
| --- | --- | --- | --- |
| `id` | `bigserial` | 整库快照主键 | 一次整库冻结对应一个 ID |
| `created_at` | `timestamptz` | 快照创建时间 | 快照生成时写入 |
| `trigger_event` | `varchar(64)` | 触发事件 | 例如 `compile.full`、`compile.retry`、`governance.correct` |
| `git_commit` | `varchar(64)` | 对应 Git 提交哈希 | 可空，用于把数据状态和代码版本对齐 |
| `description` | `text` | 快照描述 | 补充上下文说明 |
| `article_count` | `integer` | 文章数量 | 当前设计只存文章数，不是全部明细总数 |

### 运行备注

- `RepoSnapshotService.snapshot()` 会先写主表，再写明细表。
- 这张表是整库快照的“头”，真正冻结的内容在 `repo_snapshot_items`。

## 4.14 `repo_snapshot_items`

### 作用

整库快照明细表，保存一次整库快照中每个实体的冻结载荷。

### 主要关系

- 主键：`id`
- 外键：`snapshot_id -> repo_snapshots.id`

### 字段说明

| 字段 | 类型 | 作用 | 备注 |
| --- | --- | --- | --- |
| `id` | `bigserial` | 明细主键 | 单条快照实体的主键 |
| `snapshot_id` | `bigint` | 所属整库快照 ID | 指向 `repo_snapshots.id` |
| `entity_type` | `varchar(32)` | 实体类型 | 当前代码写入 `article`、`artifact`、`contribution` |
| `entity_id` | `varchar(255)` | 实体业务标识 | 文章用 `concept_id`，产物用 `artifact_type`，贡献用 UUID |
| `content_hash` | `varchar(128)` | 实体内容哈希 | 对 `payload` 做 SHA-256 后得到 |
| `payload` | `jsonb` | 实体完整快照 JSON | 保存实体序列化后的冻结内容 |

### 运行备注

- 它是 repo snapshot 真正有价值的部分，因为可用于审计、比对和导出。
- `(entity_type, entity_id)` 上有索引，便于按实体查历史。

## 4.15 `flyway_schema_history`

### 作用

Flyway 迁移历史表，记录数据库迁移脚本的执行情况。它不承载业务数据，但对排查库结构问题非常重要。

### 主要关系

- Flyway 自管表
- 无业务外键

### 字段说明

| 字段 | 类型 | 作用 | 备注 |
| --- | --- | --- | --- |
| `installed_rank` | `int` | 迁移执行顺序 | Flyway 内部排序主字段 |
| `version` | `varchar(50)` | 迁移版本号 | 可为空，重复/基线类迁移可能没有版本 |
| `description` | `varchar(200)` | 迁移描述 | 来自脚本命名 |
| `type` | `varchar(20)` | 迁移类型 | 如 SQL 等 |
| `script` | `varchar(1000)` | 脚本文件名 | 例如 `V1__baseline_schema.sql` |
| `checksum` | `int` | 校验和 | 用于检测脚本是否被改动 |
| `installed_by` | `varchar(100)` | 执行迁移的数据库用户 | 一般是连接数据库所用用户 |
| `installed_on` | `timestamp` | 安装时间 | Flyway 写入 |
| `execution_time` | `int` | 执行耗时（毫秒） | 用于排查慢迁移 |
| `success` | `boolean` | 是否成功 | `false` 时通常意味着迁移失败或中断 |

### 运行备注

- 排查“为什么本机表结构不对”时，先看它通常最有效。
- 如果版本、checksum 或 success 异常，优先怀疑迁移未正确执行。

## 5. 关系速记

可以把当前 schema 理解成下面这条主线：

1. 源文件进入系统，落到 `source_files`
2. 源文件被切块，落到 `source_file_chunks`
3. 编译生成知识文章，落到 `articles`
4. 文章切块和向量化，分别落到 `article_chunks`、`article_vector_index`
5. 文章每次变化自动留痕到 `article_snapshots`
6. 查询结果先进入 `pending_queries`
7. 人工确认后沉淀到 `contributions`
8. 全局产物、任务、质量和整库快照分别进入 `synthesis_artifacts`、`compile_jobs`、`quality_metrics_history`、`repo_snapshots` / `repo_snapshot_items`

## 7. 建议关注的几个后续治理点

这部分不是表定义本身，但和你后续维护数据库非常相关：

- `articles.updated_at` 当前没有在仓储 `upsert` 时自动刷新，字段语义和实现已经有偏差
- `articles.lifecycle` 当前存在大小写混用，建议后续统一
- `articles.review_status` 与 `pending_queries.review_status` 名字相同但语义不同，做报表时要分开处理
- `article_chunks` 当前缺少 `(article_id, chunk_index)` 唯一约束，正常流程没问题，但从防御性设计看仍可补强
