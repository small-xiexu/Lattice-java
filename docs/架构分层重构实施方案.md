## 二、目标架构

### 2.1 整体分层原则

```
Trigger 层（api / cli / mcp）
    ↓ 只依赖 Facade / ApplicationService
Application 层（Facade、Orchestrator）
    ↓ 只依赖 Domain Service 接口
Domain 层（domain/ + service/ 核心编排）
    ↓ 通过接口依赖
Infrastructure 层（infra/ + 各模块 infra 子包）
```

- **Trigger 层**：只做入参校验和 DTO 转换，不含业务逻辑
- **Application 层**：编排调用顺序，不含领域规则
- **Domain 层**：领域实体、值对象、核心服务；模块间通过接口依赖，禁止直接引用其他模块的 service 实现类
- **Infrastructure 层**：数据库、Redis、LLM 客户端、文件提取器等外部依赖实现

### 阶段四：图节点统一到 graph/node/（P-04）

**目标**：消除节点定义分裂，`compiler/graph/node/` 是所有编译图节点的唯一归属地

**当前状态**：
- `compiler/graph/node/` 有 20 个节点（定义层）
- `compiler/service/` 有 `IngestNode`、`AnalyzeNode`、`BatchSplitNode`、`GroupNode`、`CrossGroupMergeNode`、`CompileArticleNode` 等节点实现

**实施步骤**：
1. 确认 `compiler/service/` 中的节点是独立实现还是 `compiler/graph/node/` 中节点的重复（读取代码确认）
2. 若是独立实现 → 迁移到 `compiler/graph/node/`
3. 若是重复 → 合并保留 `compiler/graph/node/` 中的版本，删除 service 中的

**验收标准**：
- [ ] `compiler/service/` 下无任何 `*Node` 类
- [ ] `compiler/graph/node/` 包含所有节点实现

## 五、重构执行规范

### 5.1 每个阶段的标准动作

```
1. 新建目标包目录
2. 移动源文件（修正 package 声明）
3. 全局搜索旧 import，批量更新
4. 执行编译：mvn compile
5. 执行测试：mvn test
6. 提交：一个阶段一个 commit，commit message 格式：
   refactor(arch): [阶段N] 说明
```

### 5.2 禁止事项

- 禁止在重构过程中同时修改业务逻辑
- 禁止跨阶段合并提交
- 禁止在未通过编译的状态下推进下一阶段
- 禁止修改测试类（测试类应自动适应包路径变更，如需修改说明存在循环依赖）

### 5.3 回滚策略

每个阶段完成后创建 git tag：
```
git tag arch-refactor-phase-N
```
如阶段失败，执行：
```
git revert 到上一个 tag
```

## 附录：vault vs governance 快照边界说明

| 维度 | `governance/SnapshotService` | `vault/VaultSnapshotService` |
|------|------------------------------|------------------------------|
| 存储位置 | PostgreSQL（article_snapshots 表）| 文件系统 + Git 仓库 |
| 触发场景 | 每次编译完成后自动触发 | 用户主动导出或定时同步 |
| 回滚粒度 | 单篇文章级别 | 整库文件级别 |
| 回滚方式 | 数据库 UPDATE | Git checkout / 文件覆盖 |
| 对外接口 | `AdminSnapshotController` | `AdminVaultController` + `VaultSyncCommand` |
| 设计意图 | 知识内容版本管理 | 知识资产对外交付与备份 |

两者互补，不存在替代关系：governance 快照是"内部数据库版本历史"，vault 快照是"对外发布与 Git 存档"。
