# 复杂文档结构化证据驱动问答增强方案

**文档版本**：v1.1  
**编写日期**：2026-05-04  
**适用范围**：`/Users/sxie/xbk/Lattice-java`  
**目标**：解释当前复杂长文档问答准确率长期停留在 `40%~46.7%` 区间的真实原因，并给出一套不依赖业务特判、可追溯、可直接落地的结构化证据改造方案

---

## 1. 结论先行

当前准确率长期卡在 `40%~46.7%`，主因**不是环境、不是真实模型没接上、也不是文档本身缺信息**。

当前《卡券三期-迁移方案.md》只是**问题暴露最充分的一份代表性长文档**，不是后续优化要绑定的特殊对象。

主因是：

1. compile 主产物仍然以“可读文章”为中心，而不是以“可问答证据卡”为中心。
2. review 能识别很多 compile 产物质量问题，但 fix 不能把它们稳定修成可直接问答的高质量卡片。
3. query 仍然主要按“主题相关性”选证据，而不是按“答案形态”选证据。
4. 结果就是：简单题能答，结构化复杂题反复失分。

一句话概括：

- 当前系统已经具备“找到相关内容”的能力。
- 但还不具备“稳定拿到正确结构化证据并完整作答”的能力。

如果继续沿着“细修排序 + 细修 compile prose + 细修 UI”推进，准确率大概率仍会在 `40%~50%` 区间反复震荡。

要想明显提升，必须把主链从：

- `长文档 -> 长文章 -> 从文章里找答案`

改成：

- `长文档 -> 结构化证据卡 -> 按答案形态检索证据卡 -> 证据覆盖校验 -> 生成回答`

## 3. 为什么会一直卡在这个区间

### 3.1 不是文档没有答案

这份文档确实复杂，但并不缺关键答案。

以下关键信息文档都明确存在：

- `Q11`：`externalSrkitTypeCodeList = [22,26,43,37]` 的权威配置
- `Q12`：9 项与 8 项配置分裂
- `Q14`：第一批到第六批的灰度顺序
- `Q15`：待确认事项、已确认事项

所以问题不是“文档本身答不了”，而是系统没有把这些信息稳定提取和命中。

### 3.2 compile 当前更像“写文章”，不是“制证据”

当前 article 形态仍然偏向：

- 概述段
- 说明段
- 汇总 prose
- 编译性摘要

而不是：

- 权威配置卡
- 差异对照卡
- 批次顺序卡
- 待确认事项卡
- 精确枚举卡

这导致：

- `Q07` 这种“哪几个接口 + 命中结论 + 优先级”题，答成场景大概述
- `Q11` 这种“权威配置 vs 硬编码差异”题，答成只摘到硬编码一侧
- `Q14` 这种顺序题，答成同主题散点事实
- `Q15` 这种状态清单题，答成长段说明而不是结构化清单

### 3.3 review 能识别问题，但没有形成高质量产出

当前 compile 结果里所有 article 都是 `needs_human_review`，这至少说明两件事：

1. reviewer 已经识别到问题。
2. fixer 没有把它们修成能稳定通过的高质量问答卡。

所以当前链路实际是：

- compile draft
- reviewer 发现问题
- fix 一轮
- 仍然质量不足
- 文章继续保留在知识池

这让 query 只能在一批“已知不够稳定”的文章里做相对最优选择。

### 3.4 query 选的是“同主题”，不是“同答案形态”

当前失败题几乎都是结构化问答题：

- 枚举题
- 对照题
- 顺序题
- 待确认/已确认分类题
- 权威口径题

这类题真正依赖的是：

- 列表块
- 对照块
- 顺序块
- 状态块
- 结论块

而不是一整篇“场景说明文”。

当前 query 的主问题不是完全没召回，而是：

- 召回到了同主题文章
- 但没有优先选到最适合回答该问题的结构化证据块

所以最终答案常见模式是：

- 方向对
- 主题也对
- 但关键口径不全

这也是为什么它看起来“像对了”，但判分经常不过。

## 5. 根因分层判断

为了避免继续误投资源，这里给出一个明确分层。

### 5.1 文档复杂度问题

占比：**20%**

当前代表性样本文档确实有这些天然难点：

- 版本修订多
- 代码证据、时序图、表格、结论混排
- 同一主题跨章节重复
- 存在“权威口径”和“旧实现差异”并存

这会提高问答难度，但不是主因。

### 5.2 compile 产物形态问题

占比：**35%**

compile 主产物仍是文章，不是证据卡。

### 5.3 review / fix 产物收口问题

占比：**20%**

review 能发现问题，但不能稳定产出 `passed` 级别证据。

### 5.4 query 证据选择问题

占比：**25%**

query 没有按答案形态优先选择结构化证据。

## 6. 目标架构

### 6.1 核心目标

把当前问答主链从：

- `Document -> Article -> Answer`

改成：

- `Document -> Evidence Cards -> Retrieval by Answer Shape -> Coverage Check -> Answer`

### 6.2 不再把“文章”作为唯一主产物

后续 compile 产物建议分成三层：

1. **Article Layer**
   - 继续保留
   - 用于背景解释、浏览阅读、知识沉淀

2. **Source Evidence Layer**
   - 继续保留 `source_file_chunks`
   - 作为权威事实兜底层
   - 用于校验证据卡是否忠于原文

3. **Evidence Card Layer**
   - 新增
   - 作为 query 主检索对象
   - 直接面向问答

### 6.3 Evidence Card 数据来源优先级

证据卡必须以原文事实为主源，避免把 article 摘要中的信息损耗继续放大。

| 优先级 | 输入 | 用法 |
| --- | --- | --- |
| P0 | `source_file_chunks` / 解析结构块 | 事实主源，负责表格、列表、编号、状态分组、代码片段等原始证据 |
| P1 | `AnalyzedConcept.sections` | 辅助识别章节边界、标题语义和上下文归属 |
| P2 | `article_chunks` | 只辅助补主题、摘要和背景，不作为精确事实的唯一来源 |

约束：

- Evidence Card 的 `evidenceText` 必须能回指到一个或多个 `source_file_chunks`。
- 只有 article 有而 source chunk 无法定位的事实，不得生成高置信证据卡。
- 卡片内容与 source chunk 冲突时，以 source chunk 为准，卡片进入 `conflict` 或 `needs_human_review` 状态。

### 6.4 Evidence Card 最小 Schema

第一版建议使用最小可追溯 schema，不要先追求复杂知识图谱化。

| 字段 | 含义 |
| --- | --- |
| `cardId` | 稳定主键，建议由 source、类型、结构块 hash 派生 |
| `sourceId` / `sourceFileId` | 所属知识源和原文件 |
| `cardType` | `FACT_ENUM` / `FACT_COMPARE` / `FACT_SEQUENCE` / `FACT_STATUS` / `FACT_POLICY` 等 |
| `answerShape` | 面向 query 的答案形态，可与 `cardType` 一致，也可细分 |
| `title` | 通用标题，不绑定业务专名规则 |
| `claim` | 该卡能直接支撑的核心结论 |
| `itemsJson` | 枚举项、对照双方、顺序步骤、状态分组等结构化内容 |
| `evidenceText` | 可直接引用的原文证据片段 |
| `sourceChunkIds` | 回指原文 chunk，支持追溯和引用 |
| `articleIds` | 可选，关联背景 article |
| `confidence` | 规则或模型生成的置信度 |
| `reviewStatus` | `valid` / `incomplete` / `conflict` / `low_confidence` / `needs_human_review` |
| `contentHash` | 去重、增量更新、失效判断 |
| `createdAt` / `updatedAt` | 过程记录字段 |

`itemsJson` 不建议只存纯文本，应按卡类型保留结构：

| 卡类型 | `itemsJson` 结构 |
| --- | --- |
| `FACT_ENUM` | `items[]`，每项包含 `label`、`value`、`evidenceSpan` |
| `FACT_COMPARE` | `sides[]` 或 `pairs[]`，保留对照双方、差异点、权威结论 |
| `FACT_SEQUENCE` | `steps[]`，保留顺序号、名称、说明、证据位置 |
| `FACT_STATUS` | `groups[]`，保留状态名和事项列表 |
| `FACT_POLICY` | `rules[]`，保留约束、适用边界、禁止项 |

### 6.5 Evidence Card 的通用类型

建议最少定义 5 类通用证据卡：

| 类型 | 适用问题 |
| --- | --- |
| `FACT_ENUM` | 接口列表、队列列表、场景列表、事项列表 |
| `FACT_COMPARE` | 权威配置 vs 实现差异、A/B 对照、代码/配置分裂 |
| `FACT_SEQUENCE` | 灰度批次、步骤顺序、阶段顺序 |
| `FACT_STATUS` | 已确认 / 待确认 / 删除 / 无流量 / 有流量 |
| `FACT_POLICY` | 强约束、原则、统一口径、迁移规则 |

注意：

- 这些类型必须由**通用结构特征**判定，不得使用业务词白名单。
- 例如通过列表密度、表格结构、标题语义、对照格式、编号模式、状态短语密度来识别。

后续可以扩展，但第一版不建议全部铺开：

| 类型 | 适用问题 | 第一版建议 |
| --- | --- | --- |
| `FACT_DEFINITION` | 术语、字段、参数定义 | 暂缓，可先由 `FACT_ENUM` 和 source chunk 覆盖 |
| `FACT_DECISION` | 结论、取舍、决策记录 | 暂缓，先观察 `FACT_POLICY` 是否被过载 |
| `FACT_CAUSAL` | 原因、影响、风险 | 暂缓，避免过早引入主观归纳 |
| `FACT_REFERENCE` | 路径、接口、类名、配置项引用 | 可作为 `itemsJson` 的字段级增强，不必单独建类型 |

### 6.6 Answer Shape 与 QueryIntent 分离

项目当前已经有偏业务语义的 Query Intent，例如代码、配置、故障、架构等。Evidence Card 改造不应把“答案形态”硬塞进现有 Query Intent。

建议新增独立维度：

| 维度 | 作用 | 示例 |
| --- | --- | --- |
| `QueryIntent` | 判断问题面向什么知识域或任务 | `CODE_STRUCTURE`、`CONFIGURATION`、`ARCHITECTURE` |
| `AnswerShape` | 判断答案需要什么结构 | `ENUM`、`COMPARE`、`SEQUENCE`、`STATUS`、`POLICY` |

检索策略由 `QueryIntent + AnswerShape` 共同决定：

- `QueryIntent=CONFIGURATION + AnswerShape=COMPARE`：优先 `FACT_COMPARE` 和 source chunk。
- `QueryIntent=GENERAL + AnswerShape=SEQUENCE`：优先 `FACT_SEQUENCE`。
- `QueryIntent=CODE_STRUCTURE + AnswerShape=ENUM`：优先 source chunk、graph，再补 `FACT_ENUM`。

## 7. 推荐改造路径

### 7.1 Phase A：新增结构化证据卡层

#### 目标

让 compile 在保留 article 的同时，额外产出面向问答的 evidence cards。

#### 输入

- 主输入：`source_file_chunks` / 原文解析结构块
- 辅助输入：`AnalyzedConcept.sections`
- 背景输入：`article_chunks`

#### 输出

新增一类可检索对象，例如：

- `fact_cards`
- `fact_card_chunks`
- `fact_card_vector_index`

第一版可以只落：

- `fact_cards`
- `fact_card_vector_index`

`fact_card_chunks` 可后置。卡片本身已经是结构化短证据，只有当单卡过长或表格跨段严重时，再拆二级 chunk。

#### 生成规则

必须坚持通用结构规则：

- 多项 bullet / numbered list -> `FACT_ENUM`
- 双列以上对照表 -> `FACT_COMPARE`
- `第一批/第二批/...`、`1. 2. 3.` 顺序结构 -> `FACT_SEQUENCE`
- `已确认/待确认/删除/移除/改为/无流量/有流量` 分组块 -> `FACT_STATUS`
- `原则/强约束/统一要求/不得/必须` -> `FACT_POLICY`

第一期不要求 LLM 独立“创作”卡片，建议采用：

1. 规则抽取候选结构块。
2. LLM 只做结构归一、标题压缩和字段整理。
3. 持久化前校验 `evidenceText` 必须能在 source chunk 中定位。
4. 无法定位原文 span 的卡片直接降为 `low_confidence` 或丢弃。

#### 预期收益

以当前代表性样本为例，最先改善的会是：

- `Q07`
- `Q09`
- `Q11`
- `Q12`
- `Q14`
- `Q15`

这是全方案里收益最大的改造。

### 7.2 Phase B：query 按答案形态先选证据卡

#### 目标

让 query 不再只按主题相关性检索，而是先判断问题需要什么答案形态。

#### 形态识别

仍然必须坚持通用规则，不得用业务词特判：

| `AnswerShape` | 问题形态 | 优先证据 |
| --- | --- | --- |
| `ENUM` | “哪几个 / 列出 / 包含哪些” | `FACT_ENUM` |
| `COMPARE` | “差异 / 对比 / 是否一致 / 如何看待” | `FACT_COMPARE` |
| `SEQUENCE` | “顺序 / 先后 / 第几批 / 步骤” | `FACT_SEQUENCE` |
| `STATUS` | “哪些已确认 / 哪些待确认 / 状态是什么” | `FACT_STATUS` |
| `POLICY` | “原则 / 规则 / 是否允许 / 必须 / 不得” | `FACT_POLICY` |

#### 召回顺序建议

对精确结构化问题，建议顺序为：

1. `fact_cards` lexical
2. `source_file_chunks` lexical
3. `fact_cards` vector
4. `source_file_chunks` vector 或已有 source 相关通道
5. `article_chunks`
6. `articles`

article 应降到背景层，不应再作为主证据层。

对宽泛解释型问题，article 仍可保留较高权重，但不能覆盖 `FACT_COMPARE / FACT_SEQUENCE / FACT_STATUS` 这类高结构证据。

### 7.3 Phase C：review/fix 转向“证据卡质量审查”

#### 当前问题

当前 review/fix 是按“文章”在审。

#### 改造目标

改成按卡类型审：

| 卡类型 | 审查重点 |
| --- | --- |
| `FACT_ENUM` | 是否列全、是否漏项 |
| `FACT_COMPARE` | 是否两边都在、是否有权威结论 |
| `FACT_SEQUENCE` | 是否顺序完整 |
| `FACT_STATUS` | 是否已确认/待确认边界清楚 |
| `FACT_POLICY` | 是否含明确约束、边界和禁止项 |

卡级 review 不建议只用 `passed / needs_human_review` 二值状态，建议使用：

| 状态 | 含义 | query 使用方式 |
| --- | --- | --- |
| `valid` | 可直接支撑答案 | 正常参与主检索 |
| `incomplete` | 证据可信但缺项 | 可参与，但覆盖校验需提示缺口 |
| `conflict` | 与其他卡或 source chunk 冲突 | 默认不进入最终答案，除非问题询问冲突 |
| `low_confidence` | 原文定位弱或结构不稳定 | 降权使用 |
| `needs_human_review` | 需要人工处理 | 默认仅作背景，不作关键结论 |

#### 效果

这能把“review 发现问题，但 fix 修不成可答证据”的问题收掉一大半。

### 7.4 Phase D：回答前增加证据覆盖校验

#### 目标

在生成最终答案前，检查是否覆盖了问题要求的关键维度。

#### 规则示例

| 问题形态 | 覆盖要求 |
| --- | --- |
| 枚举题 | 答案覆盖命中 `FACT_ENUM.items[]` 的主要项；覆盖率不足时返回 `PARTIAL_ANSWER` |
| 对照题 | 至少同时覆盖 `FACT_COMPARE` 的两侧和结论字段 |
| 顺序题 | 覆盖 `FACT_SEQUENCE.steps[]` 的起点、终点和中间顺序，不得打乱顺序 |
| 状态题 | 覆盖 `FACT_STATUS.groups[]` 中被问题点名的状态分组 |
| 规则题 | 覆盖 `FACT_POLICY.rules[]` 中的约束和适用边界 |

第一版不要让模型猜“应该有 N 项”。更稳的做法是：

- 从已命中的证据卡 `itemsJson` 得到候选全集。
- 答案生成后检查回答是否覆盖候选全集的关键 label/value。
- 覆盖不足时，要么补充遗漏项，要么把 `answerOutcome` 降为 `PARTIAL_ANSWER` 并说明缺口。
- 如果证据卡本身是 `incomplete`，不得把答案标为完整成功。

#### 意义

很多当前答案不是完全错，而是只答了一半。  
这层覆盖检查能直接减少“主题对但判分不过”的情况。

### 7.5 Phase E：保留 source chunk 作为权威兜底层

#### 原则

- `fact_card` 优先
- `source_file_chunk` 兜底
- `article` 只做解释

#### 为什么

source chunk 更接近原文事实，对：

- `Q11`
- `Q12`
- `Q14`

这种权威口径题尤其重要。

### 7.6 Phase F：直接切换与过程记录

当前项目是全新项目，不需要做线上灰度或兼容旧主链。这里需要的是分阶段验收，确保每一层改造都能解释效果变化。

| 阶段 | 行为 | 退出条件 |
| --- | --- | --- |
| 产卡验收 | compile 生成并落库 fact card | 卡片数量、类型分布、source 回指通过率稳定 |
| 检索验收 | query 直接检索 fact card，并记录 topK 命中 | 命中结构化题的目标证据覆盖率明显提升 |
| 融合验收 | fact card 与 source chunk 共同作为精确结构化题主证据 | 当前 15 题和非卡券回归无明显回退 |
| 回答验收 | 引入覆盖校验，生成最终答案 | 当前 15 题和结构化题集达标 |

这里的“过程记录”不是合规审计，而是每次 query 的可观测调试记录，用来回答：

- 系统把问题识别成了什么 `AnswerShape`
- 召回了哪些 `fact_cards` 和 source chunk
- 为什么这些证据排在前面
- 覆盖校验认为缺了什么
- 最终答案引用了哪些证据

过程记录里至少要能看到：

- 问题识别出的 `AnswerShape`
- 命中的 `fact_cards`
- 每张卡的 `cardType`、`reviewStatus`、`confidence`
- 覆盖校验结果
- 最终答案引用了哪些 source chunk 或 fact card

## 8. 不建议继续投入的方向

下面这些方向不是完全没用，但已经不是主战场：

1. 再继续微调 topic extractor
2. 再继续修一点点 article 标题
3. 再继续加少量 query rerank 权重
4. 再继续清库/重导/重编译
5. 再继续单独强化 `review_status` 降权

这些改动只能带来边际提升，不足以把准确率拉到明显更高水平。

## 9. 防特判治理要求

后续实施必须把“防止方案退化成文档特判”作为硬约束。

### 9.1 禁止项

主链实现中禁止出现下面这些内容：

- 针对某个文件名、资料源名、标题名的专门分支
- 针对某个业务词、术语、接口名、配置名的白名单路由
- 针对某类验收题编号的专门答案模板
- 针对某一份文档人工补写固定结论
- 通过 hardcode 将某几个 chunk 永久前置

### 9.2 允许项

允许保留的只有通用结构规则，例如：

- 列表 / 表格 / 对照 / 编号 / 状态短语识别
- 引用格式、路径格式、数字格式、键值对格式识别
- 通用答案形态分类
- 通用覆盖校验

### 9.3 回归要求

每做完一阶段，都必须增加一轮**非卡券文档**回归，至少验证：

1. 对别的复杂文档没有明显回退。
2. 新证据卡类型在别的文档中也能正常生成。
3. query 的形态匹配规则不是只对当前样本有效。

### 9.4 评审要求

后续所有 query / compile 改动，在合入前都要按下面标准自检：

- 这条规则是不是只因为当前样本失败才加的
- 如果把当前文档名、术语名、问题样式换掉，规则是否仍成立
- 这条规则能否用更抽象的结构条件表达

只要答案是否定的，就不能进入主链。

## 10. 失败模式与防护

Evidence Card 是必要改造，但会引入新的质量风险。实施前需要把这些失败模式纳入设计。

| 失败模式 | 表现 | 防护方式 |
| --- | --- | --- |
| 卡片碎片化 | 一个完整表格或清单被拆成多张互相缺项的卡 | 使用 source chunk 邻接窗口合并；同标题、同表格、连续编号优先合卡 |
| 重复卡冲突 | 多张卡描述同一事实，但口径不同 | 用 `contentHash`、source span 和时间/章节优先级去重；冲突卡进入 `conflict` |
| 旧结论覆盖新结论 | 版本修订、删除项、改为项被当作并列事实 | 识别修订、删除、废弃、改为等通用状态词；生成 `FACT_STATUS` 或降低旧卡权重 |
| 模型补全幻觉 | LLM 为了补齐结构自行添加原文没有的项 | 持久化前做 source span 校验；无法回指原文的字段不入高置信卡 |
| 表格跨 chunk 丢列 | 对照关系只抽到一侧 | 对表格块使用前后 chunk 拼接窗口；`FACT_COMPARE` 必须包含双方字段 |
| 卡片数量膨胀 | 噪声卡太多，query 召回变差 | 设置最小结构密度、最小 item 数、最大重复率；低置信卡不建向量 |
| 覆盖校验过严 | 答案明明够用却被误判为部分回答 | 覆盖校验先按命中卡内部全集判断，不猜文档外全集 |
| source 与 card 引用割裂 | 答案引用卡片，但用户无法追溯原文 | 每张卡必须保留 `sourceChunkIds`，最终引用优先落到 source chunk |

## 11. 预期收益

收益要按阶段看，不建议只用一次总准确率承诺替代过程指标。

### 11.1 如果只做小修

预期：

- 当前 15 题准确率大概率从 `40%` 提升到 `45%~50%`
- 结构化题仍会在“主题对、口径不全”的状态里震荡

### 11.2 完成 Phase A + 产卡验收

预期：

- 结构化证据卡的 source 回指通过率应达到 `90%+`
- 当前失败题中，`Q07 / Q11 / Q12 / Q14 / Q15` 的 topK 证据覆盖应明显提升
- 此阶段不强求最终答案准确率立刻提升，重点看证据层是否变好

### 11.3 完成 Phase B + 主证据融合

预期：

- 当前 15 题准确率有机会提升到 `60%+`
- 枚举、对照、顺序、状态题的命中证据应从 article 主导转为 fact card + source chunk 主导

### 11.4 完成 Phase C + Phase D

预期：

- 结构化题准确率有机会向 `70%~80%` 靠近
- 是否能继续冲到 `90%`，取决于证据卡生成稳定性、卡级 review/fix 质量和非卡券样本泛化能力

## 12. 推荐实施顺序

### 第 1 阶段：模型与落库

- 新增 `fact_cards` 数据模型、Repository、基础过程记录查询。
- 只实现 `FACT_ENUM / FACT_COMPARE / FACT_SEQUENCE` 三类规则抽取。
- compile 先产卡，并完成 source 回指校验。
- 验收重点：卡片数量、类型分布、source 回指通过率、无业务特判。

### 第 2 阶段：补齐卡类型与检索过程记录

- 补 `FACT_STATUS / FACT_POLICY`。
- 新增 `AnswerShape` 分类。
- query 直接检索 fact card，记录 fact card topK 命中。
- 验收重点：当前 15 题失败样本的目标证据是否能进入 topK。

### 第 3 阶段：主证据融合

- 新增 fact card lexical / vector 通道。
- 在结构化题中让 fact card 与 source chunk 共同成为主证据。
- article 降为背景层，避免摘要覆盖原文事实。
- 验收重点：当前 15 题准确率、非卡券复杂文档无明显回退。

### 第 4 阶段：卡级 review 与覆盖校验

- review/fix 从文章级扩展到卡级。
- 增加覆盖校验和 `PARTIAL_ANSWER` 降级。
- 对 `conflict / incomplete / low_confidence` 卡片做不同 query 策略。
- 验收重点：结构化题完整性、冲突处理、答案引用可追溯。

### 第 5 阶段：扩大回归集

- 用当前 15 题做固定回归。
- 补 20~30 题结构化题集。
- 至少加入 2 份非卡券复杂文档，覆盖表格、列表、状态、顺序、对照。
- 验证不是只对当前代表性样本有效。

## 13. 验收标准

后续验收不建议只看总体准确率，还要拆成证据层、检索层、答案层三类指标。

| 层级 | 指标 | 目标 |
| --- | --- | --- |
| 证据层 | fact card source 回指通过率 | `> 90%` |
| 证据层 | `valid / incomplete / conflict / low_confidence` 状态可解释 | 100% 可在过程记录中查看 |
| 证据层 | 非卡券文档能生成同类卡 | 至少 2 份复杂文档通过 |
| 检索层 | 结构化题目标证据进入 topK | `> 80%` |
| 检索层 | `AnswerShape` 分类可追溯 | 100% query 过程记录可见 |
| 答案层 | 枚举题准确率 | `> 70%` |
| 答案层 | 对照题准确率 | `> 70%` |
| 答案层 | 顺序题准确率 | `> 70%` |
| 答案层 | 状态题准确率 | `> 70%` |
| 答案层 | 当前 15 题总准确率 | Phase B 后 `> 60%` |

如果只看总体平均值，很容易继续掩盖结构化题的真实短板。

---

## 14. 最终建议

如果目标是“显著提升准确率”，建议立即停止继续做零散小修，转为：

1. 先实现可追溯的 `Evidence Card` 主产物层。
2. 再新增独立 `AnswerShape`，让 query 按答案形态优先检索 fact card 和 source chunk。
3. 最后把 review/fix 改造成卡级质量门禁，并在生成前做覆盖校验。

关键不是简单新增一张 `fact_cards` 表，而是把证据卡设计成可回指原文、可追溯、可覆盖校验、可直接替换 article 主链的事实对象。

如果继续沿着“文章中心”链路做细修，准确率大概率仍会在 `40%~50%` 区间反复震荡。
