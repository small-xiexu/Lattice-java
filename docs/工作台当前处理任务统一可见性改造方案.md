# 工作台当前处理任务统一可见性改造方案

**文档版本**：v1.0  
**编写日期**：2026-04-27  
**适用范围**：`/Users/sxie/xbk/Lattice-java`  
**设计目标**：解决“后台持续处理资料，但工作台看起来像空闲”的可见性缺口，把 `/admin` 工作台从“最近同步运行”升级为“当前处理任务”统一视图

---

## 1. 问题定义

### 1.1 真实用户问题

当前工作台存在一个非常明显的用户感知断层：

- 服务端实际上还在持续处理资料。
- 管理员从结构化日志、数据库或编译任务接口可以看到任务正在推进。
- 但 `/admin` 工作台里的“最近同步运行”却可能显示 `运行中 = 0`，甚至列表为空。

对用户来说，这会被自然理解成：

- 系统没有收到任务。
- 系统没有开始处理。
- 系统可能已经卡死。
- 需要再次点击上传、同步或重新提交。

这不是单纯的文案问题，而是一个运行态可见性缺口。

### 1.2 本次定位到的真实现象

在 `18096` 隔离实例上，已经观察到下面这组同时成立的事实：

- `compile_jobs` 中存在一条 `RUNNING` 任务。
- 该任务当前步骤会从 `review_articles` 推进到 `fix_review_issues`，进度持续变化。
- `source_sync_runs` 表为空。
- `/api/v1/admin/source-runs` 返回空数组。
- 工作台“最近同步运行”因此显示没有进行中任务。

这说明当前问题不是前端数值计算错误，而是页面使用的主数据对象过窄，漏掉了真实正在运行的后台任务。

---

## 2. 根因分析

### 2.1 当前页面只依赖 `source-runs`

工作台“最近同步运行”区块当前只读取：

- `GET /api/v1/admin/source-runs`

前端统计“运行中 / 待确认 / 疑似卡住 / 已完成 / 失败”时，也完全基于这批 `source-runs` 数据。

因此，只要某个后台处理没有进入 `source_sync_runs`，页面就完全看不到它。

### 2.2 系统内部实际存在两类后台任务

当前系统至少存在两类与“资料进入知识库”相关的后台对象：

1. `source_sync_runs`
   - 语义：资料包上传、资料源同步、自动识别、人工确认等“同步运行”
   - 强项：能表达 `WAIT_CONFIRM`、归并决策、资料源关系等同步语义

2. `compile_jobs`
   - 语义：真正执行读取资料、生成文章、审查、修复、入库、向量刷新等编译过程
   - 强项：能表达 `RUNNING / STALLED / FAILED`、当前步骤、进度、心跳、租约等执行态信息

### 2.3 为什么 direct compile 会消失

当前 direct compile 路径会直接创建 `compile_job`，但不会自动创建 `source_sync_run`。

这意味着：

- 从系统内部看，任务确实存在，而且在运行。
- 从工作台看，因为没有对应的 `source_run`，所以任务等于不存在。

### 2.4 用户真正关心的对象并不是内部表

用户视角里，发起的是：

- “上传一批资料”
- “同步一个资料源”
- “让 docs/ 进入知识库”

用户并不关心它最终落成了：

- `source_sync_run`
- `compile_job`
- 还是别的内部对象

用户只关心一个问题：

- 这批资料现在到底有没有在处理，如果在处理，处理到哪了。

所以当前 UI 的核心问题不是“列表少一条”，而是“页面主对象选错了”。

---

## 3. 设计目标

### 3.1 目标

本方案要实现下面 5 个目标：

1. 只要后台仍有活跃处理，工作台首页就必须可见。
2. 用户能看到统一的“当前处理任务”，而不是被迫理解多个内部对象。
3. direct compile 与上传同步触发的任务都能进入同一块主视图。
4. 页面继续保留当前已经成熟的运行态信息：步骤、进度、最近推进时间、原因摘要、失败收口。
5. 第一版只做读模型统一，不强制重构现有写路径。

### 3.2 非目标

本方案第一版不追求下面这些目标：

- 不把所有 direct compile 强行改造成 `source_sync_run`
- 不改造现有 compile 图执行逻辑
- 不重做整个工作台 UI 风格
- 不在第一版同时重构资料源详情页的全部历史视图
- 不把工作台首页变成完整的运维调试页

---

## 4. 总体方案

### 4.1 主视图从“最近同步运行”升级为“当前处理任务”

产品层建议把工作台主区块从：

- `最近同步运行`

升级为：

- `当前处理任务`

该区块展示的不是某一张内部表，而是“当前所有会让用户等待结果的后台任务”。

### 4.2 保持写模型不变，新增统一读模型

底层仍然保留：

- `source_sync_runs`
- `compile_jobs`

但在管理端新增一个聚合读模型，把它们统一整理成：

- `processing task`

也就是“处理任务”。

这样做的原因是：

- 风险最小，不破坏现有同步语义
- 可以直接复用当前 compile 的步骤、进度、心跳、派生态
- 可以在不动执行链路的前提下，先解决首页可见性问题

### 4.3 页面只消费统一任务接口

前端不再自己同时调多个接口做拼接。

而是由后端提供一个统一接口，例如：

- `GET /api/v1/admin/processing-tasks?limit=10`

工作台首页以后只依赖这一个聚合接口。

## 5. 统一任务模型设计

建议新增一组管理端只读响应对象：

- `AdminProcessingTaskListResponse`
- `AdminProcessingTaskSummaryResponse`
- `AdminProcessingTaskItemResponse`

### 5.1 列表响应

`AdminProcessingTaskListResponse` 建议包含：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `summary` | object | 顶部统计卡片用汇总 |
| `items` | list | 当前处理任务列表 |

### 5.2 汇总字段

`AdminProcessingTaskSummaryResponse` 建议包含：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `runningCount` | int | 运行中任务数 |
| `waitingCount` | int | 待确认任务数 |
| `stalledCount` | int | 疑似卡住任务数 |
| `succeededCount` | int | 最近成功完成任务数 |
| `failedCount` | int | 最近失败任务数 |

### 5.3 单条任务字段

`AdminProcessingTaskItemResponse` 建议包含：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `taskId` | string | 前端唯一主键 |
| `taskType` | string | `SOURCE_SYNC` / `STANDALONE_COMPILE` |
| `sourceRunId` | long nullable | 关联同步运行主键 |
| `compileJobId` | string nullable | 关联编译任务主键 |
| `sourceId` | long nullable | 资料源主键 |
| `sourceName` | string nullable | 资料源名称 |
| `sourceType` | string nullable | 资料源类型 |
| `title` | string | 用户可读标题 |
| `status` | string | 统一展示状态 |
| `derivedStatus` | string nullable | 统一派生状态 |
| `stage` | string | 用户可读阶段名称 |
| `stepCode` | string nullable | 内部步骤编码 |
| `stepLabel` | string | 用户可读步骤名称 |
| `progressCurrent` | int nullable | 当前进度分子 |
| `progressTotal` | int nullable | 当前进度分母 |
| `progressText` | string | 当前进度文案 |
| `reasonSummary` | string | 原因摘要 |
| `errorCode` | string nullable | 稳定错误码 |
| `errorMessage` | string nullable | 原始错误摘要 |
| `requestedAt` | string nullable | 请求时间 |
| `startedAt` | string nullable | 开始时间 |
| `lastProgressAt` | string nullable | 最近推进时间 |
| `finishedAt` | string nullable | 结束时间 |
| `sourceNames` | list | 来源文件预览 |
| `actions` | list | 可执行动作集合 |

## 6. 聚合规则

### 6.1 数据来源

聚合任务由两部分组成：

1. 所有最近的 `source_sync_runs`
2. 所有最近的 standalone `compile_jobs`

这里的 standalone `compile_jobs` 指：

- `source_sync_run_id is null`

也就是没有挂在某条同步任务下面的编译任务。

### 6.2 去重规则

必须保证同一条真实处理只在页面出现一次。

建议规则如下：

1. 如果某条 `compile_job` 已经关联了 `source_sync_run_id`
   - 由 `source_sync_run` 主导展示
   - 不再单独生成 standalone 编译任务卡片

2. 如果某条 `compile_job` 没有关联 `source_sync_run_id`
   - 生成一条 `STANDALONE_COMPILE` 任务

这样可以避免：

- 上传触发的任务在首页出现两条
- direct compile 明明在跑却完全消失

### 6.3 标题生成规则

任务标题建议按下面顺序选择：

1. `sourceName`
2. `sourceNames` 的首项或前两项预览
3. `sourceDir` 的目录名
4. `处理任务 #ID`

这样 direct compile 至少能展示成：

- `docs`
- `项目全流程真实验收手册`
- `README.md 等 7 个文件`

而不是一条没有语义的空白任务。

### 6.4 排序规则

建议按下面优先级排序：

1. 活跃任务优先
2. `lastProgressAt / updatedAt / requestedAt` 倒序
3. 历史任务再按时间倒序

这样首页总是优先展示当前最值得用户关注的任务。

## 7. 状态口径设计

### 7.1 顶层展示状态

统一任务的主状态建议收敛为：

- `QUEUED`
- `MATCHING`
- `WAIT_CONFIRM`
- `MATERIALIZING`
- `COMPILE_QUEUED`
- `RUNNING`
- `STALLED`
- `SUCCEEDED`
- `SKIPPED_NO_CHANGE`
- `FAILED`

### 7.2 统计卡片归类

`运行中`

- `QUEUED`
- `MATCHING`
- `MATERIALIZING`
- `COMPILE_QUEUED`
- `RUNNING`

`待确认`

- `WAIT_CONFIRM`

`疑似卡住`

- `STALLED`

`已完成`

- `SUCCEEDED`
- `SKIPPED_NO_CHANGE`

`失败`

- `FAILED`

### 7.3 阶段映射

建议统一使用用户语言，而不是直接暴露底层编码：

| 内部状态/步骤 | 用户阶段 |
| --- | --- |
| `QUEUED` | 等待开始 |
| `MATCHING` | 自动识别中 |
| `WAIT_CONFIRM` | 等待人工确认 |
| `MATERIALIZING` | 资料物化中 |
| `COMPILE_QUEUED` | 等待编译 |
| `INGEST_SOURCES` | 读取资料文件 |
| `COMPILE_NEW_ARTICLES` | 生成文章草稿 |
| `REVIEW_ARTICLES` | 审查文章草稿 |
| `FIX_REVIEW_ISSUES` | 修复审查问题 |
| `PERSIST_ARTICLES` | 写入知识库 |
| `REBUILD_ARTICLE_VECTORS` | 刷新文章向量 |
| `REBUILD_SOURCE_VECTORS` | 刷新资料向量 |
| `FINALIZE_JOB` | 完成收口 |

该映射建议直接复用当前工作台里已经存在的步骤映射规则，不重新发明第二套口径。

### 7.4 疑似卡住判定

`STALLED` 的判定继续沿用现有 `CompileJobDerivedStatusResolver`：

- 运行租约过期
- 最近心跳超过阈值
- 最近进度更新时间超过阈值

这样可以保证：

- 执行态判断仍然收敛在后端
- 前端只负责展示，不负责再做一轮卡住推导

## 8. 后端实现方案

### 8.1 新增聚合服务

建议新增：

- `AdminProcessingTaskService`

职责：

- 查询最近 `source_sync_runs`
- 查询最近 standalone `compile_jobs`
- 做去重、排序、状态归一、标题拼装、原因摘要拼装
- 输出统一任务响应

### 8.2 新增控制器接口

建议新增：

- `AdminProcessingTaskController`

接口：

- `GET /api/v1/admin/processing-tasks?limit=10`

### 8.3 现有接口保留

下面这些接口应继续保留，不做破坏性修改：

- `/api/v1/admin/source-runs`
- `/api/v1/admin/source-runs/{runId}`
- `/api/v1/admin/jobs`
- `/api/v1/admin/jobs/{jobId}`

原因：

- 资料源详情页、调试页和历史工具仍然可能依赖原始对象
- 聚合接口服务于首页统一展示，不替代所有底层接口

### 8.4 仓储层优化

当前 `CompileJobService.listJobs()` 走的是全量 `findAll()`。

为了避免首页轮询历史作业全表，建议新增：

- `CompileJobJdbcRepository.findRecent(int limit)`
- `CompileJobJdbcRepository.findRecentStandalone(int limit)`

必要时也可以新增：

- `findRecentActive(int limit)`

目标是让首页轮询只读取最近 N 条和活跃任务，不因历史数据不断积累而退化。

### 8.5 统一动作生成

后端建议直接给前端返回任务动作类型，而不是让前端再靠状态和字段自行推断。

第一版建议支持的动作：

- `CONFIRM_NEW_SOURCE`
- `CONFIRM_APPEND_SOURCE`
- `CONFIRM_UPDATE_SOURCE`
- `RESYNC_SOURCE`
- `VIEW_COMPILE_DETAIL`

这样可减少前端分支判断复杂度。

## 9. 前端实现方案

### 9.1 页面命名调整

建议把当前工作台里的区块标题从：

- `最近同步运行`

调整为：

- `当前处理任务`

说明文案同步改成：

- 先看系统当前正在处理什么，再看每条任务的当前阶段、当前进度、最近推进时间和原因摘要。

### 9.2 数据加载调整

当前：

- `loadRecentRuns()` 请求 `/api/v1/admin/source-runs`

改造后：

- `loadProcessingTasks()` 请求 `/api/v1/admin/processing-tasks`

工作台首页以后只消费聚合任务视图。

### 9.3 卡片渲染复用

当前运行卡片已经具备下面这些成熟能力：

- 状态 badge
- 当前阶段
- 当前步骤
- 当前进度
- 最近推进
- 原因摘要
- 失败收口

因此第一版不建议重做视觉样式，而是复用现有卡片骨架，只把数据对象切换成统一任务对象。

### 9.4 动作差异

对于 `SOURCE_SYNC` 类型任务：

- 保留人工确认
- 保留重新同步资料源

对于 `STANDALONE_COMPILE` 类型任务：

- 第一版建议只提供“查看编译详情”
- 不建议首页直接提供“重新编译目录”

原因是首页应优先承担“可见性”和“引导”职责，不应立刻演化成技术调试入口。

### 9.5 空状态文案

当前空态文案中的“同步”表述需要升级为“处理任务”口径。

建议：

- `当前没有正在推进的资料处理任务`
- `最近还没有处理记录`

避免页面继续给用户传达“系统只认同步任务”的旧心智。

## 10. 为什么不直接把 direct compile 写成 `source_sync_run`

本方案明确不建议第一版这样做，原因如下：

1. `source_sync_run` 有自己完整的同步语义
   - 归并判断
   - `WAIT_CONFIRM`
   - 资料源匹配
   - 同步动作

2. direct compile 并不天然具备这些字段

3. 如果为了展示强行伪造一条同步运行
   - 会把同步语义与编译语义混在一起
   - 后续排障和审计边界会更模糊

所以第一版最稳妥的做法是：

- 保持底层对象不变
- 通过聚合读模型统一展示

---

## 11. 为什么不让前端双接口拼装

本方案同样不建议前端自己同时请求：

- `/api/v1/admin/source-runs`
- `/api/v1/admin/jobs`

然后自行去重和归类。

原因：

- 去重规则属于领域逻辑
- 状态归一属于领域逻辑
- 标题生成属于领域逻辑
- 动作可用性也属于领域逻辑

如果把这些都放到前端：

- JS 分支会快速膨胀
- 多个页面可能各自实现一套不一致规则
- 后续维护和排障成本会持续上升

因此聚合应在后端完成。

## 12. 实施步骤

### Phase 1：统一任务读模型落地

- 新增 `AdminProcessingTaskService`
- 新增 `AdminProcessingTaskController`
- 新增 recent / standalone recent 仓储查询
- 输出统一 `processing-tasks` 聚合接口
- 工作台首页改为依赖聚合接口

这一阶段完成后，应立即解决：

- 后台很忙但前台显示空闲
- direct compile 任务完全不可见

### Phase 2：文案与标题收口

- 优化 direct compile 标题
- 统一原因摘要
- 调整空状态文案
- 调整区块标题为“当前处理任务”

### Phase 3：资料源详情联动

- 把资料源详情中的“同步历史”视图升级为“处理历史”
- 在资料源详情中纳入与其关联的 compile 处理

### Phase 4：文档与验收口径回写

- 更新验收手册
- 更新启动配置清单中工作台观察说明
- 把“最近同步运行”的产品口径切换为“当前处理任务”

## 13. 测试清单

### 13.1 后端测试

建议至少覆盖下面场景：

1. `source_sync_run + compile_job` 关联场景只生成 1 条聚合任务。
2. standalone compile 会生成 1 条独立任务。
3. `WAIT_CONFIRM`、`RUNNING`、`STALLED`、`FAILED` 归类正确。
4. 统一标题生成优先级正确。
5. 排序优先展示活跃任务。
6. `summary` 汇总数与 `items` 一致。

### 13.2 前端测试

建议至少覆盖下面场景：

1. 聚合接口为空时，展示正确空状态。
2. 有 standalone compile 时，“运行中”不再错误显示为 0。
3. `review_articles`、`fix_review_issues` 这种长耗时步骤能展示在任务卡片里。
4. `STALLED` 会进入“疑似卡住”卡片统计，而不是直接消失。
5. `WAIT_CONFIRM` 任务仍保留人工确认操作。
6. `FAILED` 任务仍保留资料源重试入口。

### 13.3 真实回归

建议至少做 4 条真实链路回归：

1. 上传文件触发一条正常同步任务。
2. Git 或目录资料源触发一条正常同步任务。
3. direct compile `docs/` 触发 standalone compile。
4. 通过长耗时或人为延迟制造 `STALLED`，确认首页状态正确变化。

## 14. 验收标准

本方案实施完成后，工作台首页至少要满足下面 5 条标准：

1. 只要后台仍有活跃处理，首页绝不能显示“当前没有正在推进的资料处理任务”。
2. direct compile 任务必须能在首页看到。
3. 用户必须能看到当前阶段、当前进度、最近推进时间和原因摘要。
4. `RUNNING -> STALLED` 必须是显式可见的状态变化，不能静默消失。
5. 上传同步触发的任务不能因为聚合而重复展示两条。

---

## 15. 一句话结论

这次要修的不是一个显示 bug，而是工作台主视图对象选错导致的可见性缺口。

最合理的方案不是继续强化“最近同步运行”这张单表，而是把首页升级成“当前处理任务”统一视图：

- 写模型继续保留 `source_sync_runs` 和 `compile_jobs`
- 读模型新增统一聚合
- 前端只消费统一任务接口

这样既能立即解决“后台很忙但前端看不到”的问题，也不会把同步语义和编译语义硬揉成一套难以维护的模型。
