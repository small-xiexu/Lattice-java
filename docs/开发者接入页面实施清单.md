## 1. 文档目标

本文档不是技术实现细节说明，也不是纯视觉稿，而是站在产品实施视角，回答下面这个问题：

> Lattice 后台里的“开发者接入”页面，应该怎么设计、怎么分期、怎么验收？

本文默认基于当前仓库现状：

- 后台产品面已拆分为 `/admin`、`/admin/ask`、`/admin/settings`、`/admin/developer-access`
- 项目定位已经明确包含 `Web / HTTP API / CLI / MCP`
- MCP HTTP 已有真实验收
- MCP stdio 已有命令入口与 bridge 实现
- CLI 已可通过源码仓库模式执行，但当前仍偏开发者使用方式，尚未形成真正产品化的“复制即用”体验

本文适用对象：

- 产品经理
- 前端开发
- 后端开发
- 负责后台信息架构与页面落地的设计同学

---

## 2. 一句话结论

“开发者接入”不应该做成一篇埋在文档里的说明，而应该做成：

- 首页可发现
- 设置页可承接
- 示例可复制
- 首次接入可验证
- 高级模式可折叠

换句话说，这个页面的目标不是“告诉用户支持 CLI / MCP”，而是“让用户进来就能复制一段可用模板”。

---

## 3. 当前基线判断

### 3.1 已经具备的能力

- 后台已完成普通用户页与管理员设置页拆分
- MCP HTTP 已真实跑通
- `mcp-stdio` 子命令已存在
- stdio bridge 已存在，能够把远端 HTTP MCP 包装成本地 stdio MCP
- CLI 已具备 `status`、`query`、`source-list`、`source-sync` 等基础命令
- CLI 已支持统一 `--server` 远端模式

### 3.2 当前真正缺的不是协议，而是产品化包装

现在的问题不是“项目有没有 CLI / MCP”。

现在的问题是：

- 用户进后台看不见这件事
- 用户即使知道支持 CLI / MCP，也拿不到一段能直接复制的模板
- CLI 现阶段更接近源码仓库用法，还没有官方包装脚本
- MCP 虽然能力已有，但后台没有形成面向用户的接入页

因此，这一轮工作的重点应放在：

- 信息架构补齐
- 页面模块补齐
- 模板可复制
- 验证路径清晰

而不是再去从零做协议层能力。

---

## 4. 设计原则

### 4.1 首页负责露出，不负责讲细

首页 `/admin` 是普通用户与管理员都会进入的入口页。

因此首页只做一件事：

- 告诉用户“除了网页后台，还支持开发者接入方式”

首页不应该承担：

- 详细接入说明
- 大段配置模板
- 长命令展示

### 4.2 开发者接入页负责完整承接

真正完整的接入说明，应统一落在 `/admin/developer-access` 这个独立一级入口里。

推荐入口名：

- `开发者接入`

不建议命名为：

- `CLI 配置`
- `MCP 说明`
- `高级设置`

原因是页面实际承载的是一整套“接入入口地图”，而不是单一能力说明。

### 4.3 先复制，再理解

这个页面首先应该让用户得到一段“复制后就能用”的内容，再补充说明和 FAQ。

优先级顺序应当是：

1. 复制模板
2. 首次验证步骤
3. 成功后会看到什么
4. 补充原理说明

### 4.4 MCP 放在 CLI 前面

当前产品定位里，MCP 更贴近“AI 客户端接入”的主叙事，也更适合被后台直接展示成产品能力。

因此页面顺序建议：

1. MCP
2. CLI
3. HTTP API

### 4.5 高级模式必须折叠

当前 CLI 仍偏源码仓库模式，不适合把复杂启动方式直接暴露在首屏。

如果暂时没有统一脚本，则应当：

- 主推“推荐模式”
- 把源码仓库命令放入“高级模式”折叠区

不能把所有复杂命令默认展开，否则页面会明显偏工程文档，而不是产品页面。

## 5. 页面放置方案

### 5.1 首页放置

页面：

- `/admin`

放置位置建议：

- 首页 Hero 区之后
- 资料管理主区域之前

新增卡片：

- 标题：`开发者接入`
- 说明：`除了网页后台，Lattice 还支持 CLI、HTTP API、MCP 等接入方式。适合脚本调用、系统集成和 AI 客户端使用。`
- 按钮：`查看接入方式`

首页卡片作用：

- 提高可发现性
- 建立“系统不只有网页入口”的认知
- 引导进入左侧独立的“开发者接入”入口

### 5.2 开发者接入页放置

页面：

- `/admin/developer-access`

页面内二级标签：

- `接入概览`
- `MCP`
- `CLI`
- `HTTP API`
- `FAQ`

左侧一级入口顺序建议：

1. 知识库管理
2. 知识问答
3. 管理员设置
4. 开发者接入

这样更符合用户心智：

- 先区分“后台配置”和“开发者接入”是两类不同任务
- 进入开发者接入后，再按二级标签处理 MCP、CLI、HTTP API 和 FAQ

---

## 6. 页面结构

推荐从上到下按下面顺序排布：

1. 页面标题区
2. 当前接入信息
3. 快速接入导航
4. MCP 快速接入
5. CLI 快速接入
6. HTTP API 说明
7. 常见问题

### 6.1 页面标题区

- 标题：`开发者接入`
- 副标题：`除了网页后台，Lattice 还支持 CLI、HTTP API、MCP 等接入方式。你可以直接复制下面的示例完成接入。`
- 顶部提示：`本页示例会根据你当前访问的服务地址自动生成，复制后通常不需要手动拼接地址。`

### 6.2 当前接入信息

推荐展示 3 个信息卡：

- 当前服务地址
- 当前 MCP 地址
- 当前服务状态

建议补一个按钮：

- `复制当前服务地址`

设计要求：

- 地址必须按当前访问页面的地址动态生成
- 不能写死 `8080`
- 也不能默认展示历史验收端口 `18082`

### 6.3 快速接入导航

建议使用 3 张导航卡：

- MCP 快速接入
- CLI 快速接入
- HTTP API 接入

每张卡只负责一件事：

- 告诉用户这条入口适合谁
- 把用户带到对应模块

不在导航卡里堆模板正文。

## 7. MCP 模块实施清单

### 7.1 模块定位

MCP 模块是当前页面的第一主模块。

原因：

- 项目本身定位强调 AI 客户端接入
- MCP 是最能体现“知识库 + MCP 后端”产品特征的能力
- 后台补这个模块，用户收益最直接

### 7.2 模块文案

- 标题：`MCP 快速接入`
- 说明：`适合 Claude Desktop、Codex 类客户端或其他支持 MCP 的工具。下面的配置会自动带入当前 MCP 地址，复制后通常可以直接使用。`

### 7.3 模块结构

建议使用 3 个子标签：

1. `HTTP 直连`
2. `stdio Bridge`
3. `首次验证`

### 7.4 模块必须具备的复制能力

- `复制 MCP HTTP 配置`
- `复制 stdio Bridge 配置`
- `复制 MCP 验证步骤`

### 7.5 模块必须具备的辅助提示

每个模板卡片下都应出现下面这类提示：

- `当前 MCP 地址已自动填入`
- `你只需要改客户端里的显示名称`
- `接通后先看工具列表，再调用业务工具`

### 7.6 MCP 关键风险提示

页面里必须有一条高亮提示：

`注意：调用 lattice_query 后，结果会进入待确认生命周期。如果客户端没有继续 confirm、correct 或 discard，pending 记录会保留。`

这条提示不能放在深层折叠区，必须默认可见。

### 7.7 MCP 交互要求

- 每个复制按钮都要有 toast 提示
- 每个模板都支持“展开查看内容”
- 页面要支持用户先复制，再决定是否阅读原理说明

---

## 8. CLI 模块实施清单

### 8.1 模块定位

CLI 模块是当前页面的第二主模块。

它更适合：

- 命令行调试
- 自测脚本
- 批处理任务
- 运维巡检

### 8.2 模块文案

- 标题：`CLI 快速接入`
- 说明：`适合命令行、自测脚本、批处理和运维任务。下面的命令会自动带入当前服务地址，复制后可直接执行。`

### 8.3 模块结构

建议使用 4 个子标签：

1. `单次执行`
2. `环境变量模式`
3. `高级模式`
4. `首次验证`

说明：

- 如果后续补了统一官方脚本，可以去掉“高级模式”
- 如果暂时没有统一脚本，则必须保留“高级模式”，把源码仓库命令放进去

### 8.4 CLI 模块必须具备的复制能力

- `复制状态检查命令`
- `复制首次问答命令`
- `复制资料源列表命令`
- `复制资料源同步命令`
- `复制环境变量示例`
- `复制 CLI 验证步骤`

### 8.5 CLI 模块必须具备的辅助提示

每条命令下都要告诉用户：

- `执行成功后应看到什么`
- `你只需要改哪里`

例如：

- 状态检查：应返回知识库状态摘要
- 首次问答：应返回答案与引用信息
- 资料源同步：只需要替换 `sourceId`

### 8.6 CLI 当前产品化限制

当前仓库里的 CLI 更接近开发者使用方式，而不是完整产品包装。

因此，建议明确区分两类展示：

#### 推荐模式

面向大多数用户，优先展示：

- 已有统一脚本的简化命令

例如概念上应展示为：

- `./bin/lattice-cli status --server <base-url>`

#### 高级模式

面向源码仓库用户，折叠展示：

- `java -cp ... com.xbk.lattice.cli.LatticeCliMain ...`

不要把源码仓库模式放在首屏默认展开。

### 8.7 我们建议新增的官方脚本

为了让前端真正做到“复制即用”，建议在工程侧补两个官方脚本：

- `bin/lattice-cli`
- `bin/lattice-mcp-bridge`

这样做的收益非常直接：

- 前端模板会明显更简洁
- 用户不需要理解类路径拼装
- 页面会更像产品，而不是工程说明

## 9. HTTP API 模块实施清单

HTTP API 一期不建议做得太重。

这一块更适合作为补充说明，而不是首轮主战场。

### 9.1 模块定位

适合：

- 业务系统集成
- 自建管理平台
- 服务编排

### 9.2 文案建议

- 标题：`HTTP API 接入`
- 说明：`如果你是在接业务系统、管理平台或服务编排，优先考虑 HTTP API。`

### 9.3 页面内容建议

只需要放：

- 简短说明
- 当前服务地址
- 路径判断提示

例如：

- 接 AI 客户端：优先 MCP
- 接脚本与命令行：优先 CLI
- 接业务系统：优先 HTTP API

一期不建议在这里放太多接口细节和大段 JSON。

---

## 10. FAQ 实施清单

建议默认折叠展示，避免页面过长。

至少应覆盖下面这些问题：

- 普通用户需要看这个页面吗
- CLI、HTTP API、MCP 有什么区别
- 为什么 MCP query 后还会有待处理记录
- 第一次接入应该先试哪一个
- 这些入口是不是三套不同系统

FAQ 的目标不是教学，而是减少初次接入时的误解与焦虑。

---

## 11. 前端交互清单

### 11.1 必做交互

- 所有复制按钮都有 toast 提示
- 所有模板支持展开查看
- 每个模板都显示“已自动带入当前地址”
- 复制失败时有兜底提示
- 手机端支持纵向排布和正常复制

### 11.2 不建议做的交互

- 不要首屏自动展开全部模板
- 不要在首屏展示大量工程命令
- 不要让用户自己手动拼接 `/mcp`
- 不要把接入信息混在模型配置表单中

## 12. 工程实现建议

### 12.1 前端文件落点建议

建议在现有后台结构上继续扩展：

- 首页入口卡修改：`src/main/resources/static/admin/index.html`
- 设置页标签与内容：`src/main/resources/static/admin/settings.html`
- 样式扩展：`src/main/resources/static/admin/admin.css`

### 12.2 脚本文件建议

如果继续在同一个脚本里堆交互逻辑，`settings.js` 会越来越重。

因此更推荐：

- 保留 `settings.js` 处理现有管理员配置交互
- 新增 `developer-access.js` 专门处理开发者接入页的复制、地址生成、折叠与 toast

### 12.3 后端接口建议

一期优先不新增后端接口。

可以优先依赖前端运行时能力：

- 当前服务地址：从页面 `location` 推导
- MCP 地址：前端拼接 `/mcp`
- 服务状态：复用现有健康检查接口

只有在后续需要展示更多动态信息时，才考虑补专门的接入信息接口。

## 13. 分期建议

### 13.1 一期

目标：

- 让用户发现有开发者接入能力
- 让用户拿到第一批可复制模板

范围：

- 首页入口卡
- 设置页 `开发者接入` 标签
- 当前接入信息
- MCP 快速接入
- CLI 快速接入
- FAQ
- 复制交互与 toast

### 13.2 二期

目标：

- 让模板从“可用”升级到“更易用”

范围：

- 增加官方脚本包装
- 丰富客户端模板
- 增加接入状态说明
- 增加更多验证模板

### 13.3 三期

目标：

- 提升诊断与自助排障能力

范围：

- 一键下载配置模板
- 接入诊断
- 更多连接状态提示
- 更完善的错误定位与 FAQ

---

## 14. 验收清单

页面上线后，至少要满足下面这些验收点：

- 用户进入后台 5 秒内能发现“开发者接入”
- 用户进入接入页 10 秒内能找到一段可复制模板
- MCP 用户不需要自己拼接 `/mcp`
- CLI 用户不需要自己猜 `--server` 参数位置
- 用户能知道第一次应该先做什么验证
- 用户能知道成功后应看到什么
- 用户能在页面里理解为什么 MCP query 会产生 pending
- 手机端可以正常查看与复制
- 页面不会把普通用户主流程打断

---

## 15. 最终建议

如果只给一句最终建议，结论是：

后台这轮不需要再去“开发 MCP stdio 协议能力”，而是应当把已经存在的能力做成一个真正可被用户使用的产品页面。

因此，这一轮实施重点应该是：

- 补信息架构
- 补页面模块
- 补可复制模板
- 补首次验证闭环
- 有条件的话，顺手补统一 CLI / Bridge 脚本

只有这样，“支持 CLI / MCP” 才会从工程事实，变成用户真的能感知、能使用、能成功接入的产品能力。
