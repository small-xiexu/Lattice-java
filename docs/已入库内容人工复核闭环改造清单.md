## 当前结论

`/admin` 工作台已经能展示 `review_status=needs_human_review`，但还没有“人工处理完这条复核”的闭环。

当前状态：

| 层级 | 已有能力 | 缺口 |
| --- | --- | --- |
| 前端 | `已入库内容` 列表与详情能展示“需人工复核”徽标 | 无复核筛选、无确认通过、无提交复核意见、无复核历史 |
| 后端 | `AdminArticleController` 支持列表、详情、纠错、生命周期切换 | 无人工复核动作接口，无人工复核审计 |
| 数据 | `articles.review_status`、`article_snapshots.review_status` 已存在 | 无复核人、复核时间、复核意见、状态流转记录 |
| Compile | `ReviewDecisionPolicy` 会把高严重度问题标成 `needs_human_review` | 标记后只能展示，无法由管理员闭环 |
| 质量统计 | `StatusService`、`QualityMetricsService` 会统计非 passed / needs_human_review | 人工确认后无法自然降低待复核数 |

---

## 目标状态

管理员在 `/admin?tab=knowledge-articles` 可以完成下面闭环：

1. 筛选所有“需人工复核”的知识条目。
2. 打开条目详情，看到正文、摘要、主要参考文件、完整来源列表、技术信息。
3. 对照来源文件后选择：
   - `确认通过`：把 `review_status` 从 `needs_human_review` 更新为 `passed`。
   - `提交修正`：输入复核意见，触发现有文章纠错链路，修正后保持待再审状态。
   - `归档 / 废弃`：确认内容不可用时走现有生命周期能力。
4. 页面刷新后，列表徽标、详情状态、状态摘要、质量指标同步变化。
5. 后端保留审计记录：谁在什么时候对哪篇文章做了什么动作、前后状态是什么、意见是什么。

---

## 状态语义

| 状态 | 中文展示 | 含义 | 允许人工动作 |
| --- | --- | --- | --- |
| `passed` | 已通过 | 已可作为稳定依据 | 可重新标记待复核，P1 再做 |
| `pending` | 待处理 | 系统流程中的待审查状态 | 不作为 P0 人工入口重点 |
| `needs_review` | 待复核 | 人工修正或自动修复后仍需再看 | 可确认通过、可继续修正 |
| `needs_human_review` | 需人工复核 | 系统审查后认为必须人工确认 | 可确认通过、可提交修正、可归档 / 废弃 |

P0 只要求打通 `needs_human_review / needs_review -> passed` 和 `needs_human_review -> corrected -> needs_review`。

## 风险与约束

- 人工确认通过不能只改数据库状态，必须同步正文 frontmatter，否则后续快照、导出或重新入库可能出现状态漂移。
- 同一 `conceptId` 可能存在不同 `sourceId` 的文章，人工复核接口必须保留 `sourceId` 消歧能力。
- `needs_human_review` 不是错误状态，文章仍可能可见；文案要表达“可见但建议复核后作为稳定依据”。
- `提交修正` 和 `确认通过` 不能合并成一个动作，否则会把未经验证的模型修正误标为稳定。
- 审计记录不要只存在前端或日志里，必须可通过接口查询。
- 未经过真实页面和接口验证前，不得在本文勾选验收项。
