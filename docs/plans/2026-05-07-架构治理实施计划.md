## 0. 台账规则与红线

### 0.1 台账规则

- 每完成一个可独立验收的事项，立即在本文回写为已完成，不允许等到阶段末统一勾选。
- 未完成事项必须写明 `进行中 / 阻塞 / 待验证` 之一，并补最小备注：做到哪、卡在哪、下一步是什么。
- 未经验证的事项禁止勾选完成。
- 状态约定与业务台账一致：`未开始 / 进行中 / 待验证 / 阻塞 / 已完成`。

### 0.2 红线（继承 AGENTS.md，并显式声明本计划新增红线）

- 不做旧逻辑兼容、不做灰度，按新项目重构。
- Query / 检索 / 回答主链零容忍业务专名、特定文档、特定题号、特定样例字符串硬编码；本计划落地后，红线扫描必须能持续跑且零命中。
- 拆分类时**禁止简单按行号腰斩**，必须按职责切；每个新出来的类只能承担一个明确职责，且必须有独立的单元测试入口。
- 拆分阶段的 PR/提交不得引入新的 LLM 调用、不得改变对外接口签名、不得改变 DB schema；只重排代码与最小内部接口。
- 任何"通用规则"必须由配置或参数承载，不允许在代码里写死中文关键词、文件名、字段名、概念名。

### 0.3 不在本计划范围

- 业务能力新增（fact card 模型升级、deep research 能力扩展、新场景接入）：归 `2026-05-05-当前剩余工作总清单.md`。
- DB schema 演进：本计划不动 `src/main/resources/db/schema.sql`；如必须动，单开 `docs/plans/<date>-schema-演进.md`。
- 鉴权 / 安全（Sa-Token、Spring Security 等）：用户已明确不引入；后台访问控制由部署侧（外层网关或网络隔离）解决，不在代码层做。

- 范围：
  - 包含：`AnswerGenerationService` 内全部特定问法判定 / 特定中文关键词列表 / 特定 HTTP 方法补齐逻辑。

  - 不包含：通用文本结构规则（引用格式、Markdown 表格、列表识别）。

- 动作清单：

- 背景：`compiler/graph/node/` 24 个节点全部无 `@Transactional`；当前依赖 `CompilePipelineService`（也无 `@Transactional`）兜底。`PersistArticlesNode` / `PersistSourceFilesNode` / `PersistSourceFileChunksNode` / `RebuildArticleChunksNode` / `RefreshVectorIndexNode` 至少 5 个节点在做跨表写入。
- 三选一显式选定：
  - 方案 A：每个 Persist*Node 内部自己开 `REQUIRES_NEW`，节点失败不影响其它节点状态（但 graph 整体最终一致性靠 `compile_job_steps` 状态机重放）。
  - 方案 B：整个 `CompilePipelineService.run()` 包一个大事务（仅适合短链路；长链路不可行，会持锁过久）。
  - 方案 C：Pipeline 不开事务；每个写节点 `REQUIRED`，但同一概念组的写操作合并到一个 service 方法里包事务。
- 推荐：**方案 C**。原因：编译链路长（含 LLM 调用），不能锁库；同概念组的多表写（`articles` + `article_chunks` + `article_source_refs` + 向量索引）必须原子。
- 动作清单：
  1. 在 `compiler/service/` 下落 `ArticleAtomicWriteService`，提供 `persistArticleAtomic(article, chunks, sourceRefs, vectors)` 一个方法，方法上 `@Transactional`。
  2. `PersistArticlesNode` 改为只做"准备数据 + 调 `ArticleAtomicWriteService.persistArticleAtomic`"，节点本身仍无 `@Transactional`。
  3. `RefreshVectorIndexNode` 单独走，**不与 article 写在同一事务**（向量库可能跨进程，混事务风险高）。失败由 `compile_job_steps.status` 标 `VECTOR_REFRESH_FAILED`，下一次 graph 重跑时由 `IncrementalDecisionPolicy` 决定补偿。
  4. 在 `compiler/graph/CompileGraphLifecycleListener` 上加事务策略注释，明确"节点级是否事务"的策略口径，留给后来者读。
  5. Query 侧不做强事务（多为读 + 审计写）；只对 `RetrievalAuditService.persistRun` / `DeepResearchAuditPersistenceService.persist*` 这类批量写做单方法 `@Transactional` 收口。
- 验收：
  - `compiler/service/ArticleAtomicWriteService` 存在并被 `PersistArticlesNode` 唯一调用。
  - 模拟"persist 中途异常"测试：`articles` 不应有半残记录（`article_chunks` 行数与 `articles.referential_keywords` 一致）。
  - 文档：在 `docs/数据库表结构详解.md` 末尾追加一节"编译节点事务边界"，给出方案 C 的口径与原因。
- 风险：
  - 事务方法被 graph 异步执行框架包了一层；如果 `AsyncNodeAction` 的执行线程没有 Spring 事务上下文，需要在 ServiceImpl 上显式声明，不能依赖 `@Async` 上的隐式继承。

### 5.C.2 RetrievalAudit / DeepResearchAudit 写路径事务化

- 状态：未开始
- 范围：仅 `RetrievalAuditService` / `DeepResearchAuditPersistenceService` 中的批量写方法。
