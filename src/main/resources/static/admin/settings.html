常见问题

默认折叠展示，只回答第一次接入最容易卡住的地方。

普通用户需要看这个页面吗

通常不需要。普通用户主要用知识库管理和知识问答；这个页面是给脚本、自建系统和 AI 客户端接入使用的。

CLI、HTTP API、MCP 有什么区别

CLI 适合脚本和命令行；HTTP API 适合系统集成；MCP 适合 Claude Desktop、Codex 类客户端和其他支持 MCP 的工具。

为什么 MCP query 后还会有待处理记录

因为 MCP 的 lattice_query 会像其他 query 入口一样进入待确认生命周期。若客户端没有继续 confirm、correct 或 discard，pending 记录会保留，这是当前产品语义的一部分。

第一次接入应该先试哪一个

接 AI 客户端先试 MCP 的工具列表与 lattice_status；接脚本先试 CLI 的 status；接系统集成先试 HTTP API 的健康检查和 query 示例。

这些入口是不是三套不同系统

不是。它们共用同一套后端能力，只是面向不同使用方式提供不同入口，因此语义应该尽量保持一致。

什么时候真的需要新增模型连接或改角色绑定

只有在切换模型供应商、补充新模型，或者明确要把“知识入库”和“知识问答”绑定到不同模型时，才需要改。普通使用者如果只是想问答质量更好，先确认资料质量和入库状态，不要直接来改角色绑定。

识别服务连接测试失败时应该先查什么

先查 Base URL、接口路径和凭证是否完整，再看是不是供应商类型选错了。扫描 PDF、图片识别相关问题大多出在这里；如果普通文本导入没问题，只是图片和扫描件失败，优先看 OCR / 文档识别，不要先怀疑整个知识库坏了。

切换 embedding profile 或看到维度不兼容时，下一步应该做什么

先看“向量检索与重建”卡片里的兼容性判断。如果 profile 维度和 schema 维度不一致，而历史向量又已经存在，就必须勾选“重建前先清空旧向量”再重建；否则你会遇到“配置已保存，但索引其实一直建不起来”的静默失败。

中文“为什么 / 原因”类问题不稳时，什么时候才需要调检索权重

先确认资料是否真的入库、向量索引是否已经建好、旧 query cache 是否已经失效。只有这些都正常，中文原因类问题仍然持续答偏或无引用时，才值得改检索权重；否则你很容易把“资料或索引问题”误判成“权重问题”。
