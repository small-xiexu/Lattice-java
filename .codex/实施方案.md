# Lattice Java 重构实施方案（分批版）

**方案版本**：v1.3  
**更新日期**：2026-04-14  
**依据文档**：`Lattice Java 重构技术方案设计文档.md` v1.3  
**技术路线**：Option B — Spring AI Alibaba + PostgreSQL + Redis  
**适用范围**：本方案用于 **v0.1 / v1.0 / v2.0** 的分批实施，不再按原 P0-P7 线性整包推进

---

## 1. 结论与实施原则

原设计方向成立，但对当前仓库现状而言，若直接按原 `P0-P7` 一次性推进，范围过大、外部依赖过多、失败成本过高。

当前仓库状态接近空白骨架：

- `pom.xml` 仅保留最小 Maven 配置
- `src/main/java/com/xbk/Main.java` 为示例程序
- 尚无 Spring Boot、数据库迁移、AI 接入、查询接口或 MCP 基础设施

因此本方案采用以下原则：

1. **先验证不确定技术点，再实现业务主链路**
2. **先交付“编得出、查得到、答得对”，再补治理和企业能力**
3. **v1.0 只做单应用，不引入微服务拆分**
4. **审批流、Lint、全量 MCP、生产级治理全部后置**
5. **原 P0-P7 保留为能力地图，不作为首轮执行顺序**

## 2. 分批总览

### 2.1 推荐版本边界

| 版本 | 包含批次 | 目标 |
|------|---------|------|
| `v0.1` | B0-B2 | 打通最小编译闭环 + 最小查询闭环 |
| `v1.0` | B0-B4 | 对外可演示、可集成、可纠错 |
| `v2.0` | B5 | 治理、审计、质量、生产化增强 |

### 2.2 批次概览

| 批次 | 名称 | 是否硬前置 | 核心目标 |
|------|------|-----------|----------|
| B0 | 技术基线 Spike | 是 | 锁定依赖、验证 Graph / MCP / PG / pgvector / 模型装配可行 |
| B1 | 最小编译闭环 | 是 | 从源目录生成 article 并落库 |
| B2 | 最小查询闭环 | 是 | 基于 article 支持查询、返回答案和来源 |
| B3 | 稳定化与反馈闭环 | 否 | 审查、缓存、WAL、pending/correct/confirm |
| B4 | 最小 MCP 接入 | 否 | 让 Claude Desktop / Cursor 通过 MCP 使用查询闭环 |
| B5 | 治理与生产化 | 否 | Lint、审批流、生命周期、监控告警、完整工具集 |

### 2.3 Stop/Go 原则

- B0 未通过，不进入 B1
- B1 未打通，不进入 B2
- B2 已打通即可形成 `v0.1`
- B3、B4 并行度有限，建议先 B3 后 B4
- B5 必须在系统已被真实使用一段时间后再做

## 3. 为什么不用原 P0-P7 直接推进

原方案的问题不在技术方向，而在首版范围：

- 把编译、查询、MCP、审批流、Lint、可观测性全部放进同一轮实施
- 把数据库复杂特性、AI 编排、外部审批依赖、MCP 接入一起引入
- 当前项目没有现成 Spring Boot 基础设施，所有能力都需要从零搭建

原方案更适合作为“能力全景图”，不适合作为“首轮执行顺序”。

因此，本文件采用：

- **原设计文档**：负责说明最终架构目标
- **本实施方案**：负责规定首轮如何分批落地
- **后续批次执行清单**：在每个批次启动时再细化为可执行 checklist

## 4. 分批实施方案

## B0：技术基线 Spike

> 目标：在开始业务实现前，验证所有高不确定性技术点，形成明确的 PASS/FAIL 结论和 fallback

- 进行中：B0 核心烟测已完成，Spring Boot、OpenAI、Anthropic、MCP、Graph、PostgreSQL JDBC/pgvector/FTS 全量验证结果均已回写到 `.codex/P0-0 技术基线验证记录.md`
- 下一步：补 Sa-Token 坐标确认与 B1 向量映射方案选型，收束 B0 剩余事项；项目默认回归使用全局 Maven 仓库 `/Users/sxie/maven/repository`

### 4.1 范围

- Spring Boot 3.x 可启动
- OpenAI ChatModel Bean 可装配
- Anthropic ChatModel Bean 可装配
- 至少 1 个 `@McpTool` 可被发现
- 至少 1 条最小 Graph/DAG 样例可装配并执行
- PostgreSQL 16 + `pgvector` 扩展可用
- 中文检索最小 SQL 烟测可执行
- Embedding 模型和向量维度锁定

### 4.2 明确不做

- 不做业务实体
- 不做编译流水线
- 不做查询接口
- 不做 WAL / 审批流 / Lint

### 4.3 交付物

- `.codex/P0-0 技术基线验证记录.md`
- 精确到 GAV 的依赖清单
- 最小可运行 Demo：
  - Boot Demo
  - Graph Demo
  - MCP Demo
  - PostgreSQL SQL Demo

### 4.4 验收标准

- 每个能力项都有 PASS/FAIL
- 每个 FAIL 项都有 fallback 或阻塞说明
- 锁定以下关键基线：
  - Spring Boot 版本
  - Spring AI BOM 版本
  - Spring AI Alibaba Graph 依赖坐标
  - MCP Server 依赖坐标
  - Embedding 模型与维度
  - 中文检索方案
  - 构建入口（`mvn` 或 `./mvnw`）

### 4.5 Go/No-Go

- 若 Graph 或 MCP 依赖无法稳定装配，允许退化为：
  - 编排先用本地 Service 编排，不强依赖 Graph
  - MCP 延后到 B4 再接入
- 若中文检索扩展受限，退化为：
  - `pg_trgm + referential_keywords @> + 可配置 FTS`

## B1：最小编译闭环

> 目标：先证明“能把源码编成文章并落库”，不追求一次做到最稳、最全

- 进行中：B1 第七个切片已启动，正在强化 `AnalyzeNode / CrossGroupMergeNode` 的语义与去重规则，减少后续接入真实 LLM 前的结构噪音
- 下一步：补分析与合并节点的行为测试，并让最小编译链路输出更稳定的合并结果

### 5.1 范围

- Spring Boot 单应用骨架
- 最小配置文件
- Docker Compose
- Flyway 最小迁移
- 编译主链路：
  - `IngestNode`
  - `GroupNode`
  - `BatchSplitNode`
  - `AnalyzeNode`
  - `CrossGroupMergeNode`
  - 最小提交逻辑
- article / source 基础表

### 5.2 首版必须收缩

- 不做 `ReviewerAgent`
- 不做 Redis WAL
- 不做 Vault 双写
- 不做 chunk embedding
- 不做 `SynthesisNode`
- 不做全量 JPA 特性覆盖

### 5.3 设计要求

- 优先保证“主链路通”
- `AnalyzeNode` 保留 JSON 容错，但只实现最必要的 2-3 层
- 提交阶段先以 PG 为唯一事实源
- 允许先不落 Vault，或把 Vault 作为可关闭开关

### 5.4 建议实现边界

- 表结构优先保留：
  - `articles`
  - `article_chunks`
  - `source_files`
- `article_embeddings` 允许延后到 B2/B3

### 5.5 验收标准

- 给定测试 `sourceDir`
- 能完成采集、分组、分批、分析、合并、文章生成
- PG 中存在 article 记录
- 失败场景有可读日志

### 5.6 本批产出价值

完成 B1 后，系统已经具备“知识编译器”最小雏形，可以开始验证 prompt 质量、分组策略和表结构是否合理。

## B2：最小查询闭环

> 目标：尽快形成业务可感知价值，让系统从“会编译”变成“能回答”

### 6.1 范围

- `QueryController`
- `FtsSearchService`
- `RefKeySearchService`
- `RrfFusionService`
- 最小答案生成服务
- 文章来源返回

### 6.2 首版收缩策略

- 向量检索可以延后到本批后半段，或延后到 B3
- PendingQuery 暂不做持久化闭环
- 先不做 `correct / confirm / discard`
- 不做 MCP 接入

### 6.3 关键实现建议

- 首版查询排序优先依赖：
  - FTS
  - `referential_keywords` 精确匹配
  - RRF 融合
- 若 B1 尚未生成 chunk，则答案生成直接基于 article 全文
- 查询缓存可先不做，或只做最简单 Redis Key-Value

### 6.4 验收标准

- 能对 B1 产出的知识库发起查询
- 返回：
  - `answer`
  - `sources`
  - 命中的 article 信息
- 至少覆盖一类明确性知识问答，例如业务码、超时配置、系统关系

### 6.5 本批产出价值

完成 B2 后，即形成 `v0.1`，系统已可用于小范围演示和 prompt/检索质量验证。

---

## B3：稳定化与反馈闭环

> 目标：解决长任务稳定性、结果可修订、状态可恢复这三个核心问题

### 7.1 范围

- `ReviewerAgent`（单轮）
- `ReviewResultParser`
- Redis LLM Cache
- Redis WAL
- `PendingQueryService`
- `confirm / correct / discard`
- `Contribution` 写入闭环
- 如有必要，在本批补上向量生成与向量检索

### 7.2 重点约束

- v1.0 只允许单轮审查 + 单次修复
- 不引入多轮 Agent 循环
- PG 仍为唯一事实源
- WAL 只解决“恢复与补偿”，不承担永久存储职责

### 7.3 关键落地顺序

1. 先补 Redis Cache
2. 再补 `ReviewerAgent`
3. 再补 WAL 和补偿恢复
4. 再补 PendingQuery 的纠错闭环

### 7.4 验收标准

- 审查失败或超时有确定行为
- 编译中断后可恢复或重试
- `correct` 返回修订答案且仍保持 pending
- `confirm` 后能写入 contributions

### 7.5 本批产出价值

完成 B3 后，系统从 Demo 级演示进入可试运行阶段，具备最基本的“生成-审查-纠错-确认”闭环。

## B4：最小 MCP 接入

> 目标：让外部 AI 客户端能够实际使用系统，而不是先做完整企业治理

### 8.1 范围

- Spring AI MCP Server over SSE
- 最小 Sa-Token 鉴权
- 最小 MCP 工具集：
  - `lattice_query`
  - `lattice_query_pending`
  - `lattice_query_correct`
  - `lattice_query_confirm`

### 8.2 明确不做

- 不做 `lattice_compile`
- 不做 `lattice_propagate`
- 不做审批流
- 不做 10 个工具全集
- 不做复杂限流与租户治理

### 8.3 原则

- MCP 以“把已存在能力暴露出去”为目标
- 不允许 MCP 反向决定内部架构
- 内部 HTTP Service 先稳定，MCP 只做适配层

### 8.4 验收标准

- Claude Desktop / Cursor 可通过 MCP 连接
- 查询可返回结果
- 纠错与确认能力可用
- 未授权请求被拒绝

### 8.5 本批产出价值

完成 B4 后，即形成对外可集成的 `v1.0`。

## B5：治理与生产化

> 目标：在系统已具备真实使用价值后，再补质量、审计、审批和运维能力

### 9.1 范围

- Lint 六维检查
- 依赖图传播
- 生命周期定时任务
- 审批流
- 完整 MCP 工具集
- Trace / Metrics / Alert
- Vault/NFS 等部署级增强

### 9.2 排序建议

1. 先做可观测性
2. 再做生命周期与质量治理
3. 再做审批流
4. 最后补全剩余 MCP 工具和生产部署增强

### 9.3 验收标准

- 能识别质量问题并输出治理结果
- 有 Trace ID、核心指标和基本告警
- 高风险操作进入审批流
- 具备支撑持续运行的运营能力

### 9.4 注意

B5 不是“系统能不能用”的前提，而是“系统能不能长期稳定运营”的前提。

## 6. 数据层落地建议

原方案中数据层技术性很强，首轮不建议一步到位。

### 6.1 首轮推荐

- 简单 CRUD：JPA
- `tsvector` / `jsonb` / 数组 / HNSW / `@>` 查询：`JdbcTemplate` 或 jOOQ
- B1 暂不引入向量 ORM 映射；向量字段写入与检索后置到 B3，优先使用 `JdbcTemplate` 或 jOOQ 直接执行 SQL
- 若后续确实需要 JPA 映射向量列，再评估 `hibernate-vector`，不要默认自建 `PGvector` Converter
- Flyway 管控表结构

### 6.2 不建议首轮强行做到位的事项

- 所有 PG 特性全部走 JPA 映射
- 所有向量查询都封装进 Repository 抽象
- 在数据量很小时过早优化 HNSW 参数

### 6.3 原因

- 这类能力强依赖 PostgreSQL 原生语义
- 用 JPA 过度抽象会显著增加首版摩擦
- 当前目标是先打通主链路，而不是先追求 ORM 纯度

## 7. 风险与对应策略

| 风险点 | 风险说明 | 对应策略 |
|--------|---------|---------|
| Spring AI / Spring AI Alibaba 依赖漂移 | Graph、MCP、Model Starter 坐标可能变化 | B0 先锁精确版本和最小样例 |
| 中文检索扩展不稳定 | 部署环境可能不支持预期扩展 | 提前定义 `pg_trgm + @>` fallback |
| JPA 承载 PG 特性过重 | 首版开发成本高、调试困难 | 重 SQL 场景使用 `JdbcTemplate` 或 jOOQ |
| 双模型接入成本高 | 本地调试需要两套 Key | 允许本地先使用 stub/mock 或单模型降级 |
| WAL + PG 双写边界不清 | 容易把 Redis 当事实源 | 明确 PG 是唯一事实源，Redis 仅作暂存 |
| 向量检索收益不明确 | 数据量不足时收益有限 | B2 可先不上向量检索，B3 再补 |
| 范围再次膨胀 | 首版容易重新塞回审批流、Lint 等需求 | 批次验收前禁止跨批次加需求 |
