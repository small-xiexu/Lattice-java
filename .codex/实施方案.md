## 1. 结论与实施原则

原设计方向成立，但对当前仓库现状而言，若直接按原 `P0-P7` 一次性推进，范围过大、外部依赖过多、失败成本过高。

当前仓库已不再是空白骨架，而是已经完成 B0-B4 的可运行基线：

- Spring Boot / Flyway / PostgreSQL / Redis / MCP 基础设施已落地
- 已具备最小编译闭环、最小查询闭环、稳定化反馈闭环与最小 MCP 接入
- 后续工作的重点已从“搭骨架”切换为“对齐原始项目能力闭环 + 再做 Java 增强”

因此本方案采用以下原则：

1. **先验证不确定技术点，再实现业务主链路**
2. **先交付“编得出、查得到、答得对”，再补治理和企业能力**
3. **v1.0 只做单应用，不引入微服务拆分**
4. **原始项目 parity 优先于 Java 增强项**
5. **审批流、Web 后台、向量增强、复杂治理全部后置**
6. **原 P0-P7 保留为能力地图，不作为首轮执行顺序**
7. **自 2026-04-15 起，B5 及后续执行以 `.codex/B5-B8 对齐原始项目完整改动方案.md` 为专项台账**

## 3. 为什么不用原 P0-P7 直接推进

原方案的问题不在技术方向，而在首版范围：

- 把编译、查询、MCP、审批流、Lint、可观测性全部放进同一轮实施
- 把数据库复杂特性、AI 编排、外部审批依赖、MCP 接入一起引入
- 当前项目没有现成 Spring Boot 基础设施，所有能力都需要从零搭建

原方案更适合作为“能力全景图”，不适合作为“首轮执行顺序”。

因此，本文件采用：

- **原设计文档**：负责说明最终架构目标
- **本实施方案**：负责记录 B0-B4 的分批落地与验收情况
- **B5-B8 对齐方案**：负责规定后续如何按原始项目源码继续推进
- **后续批次执行清单**：在每个批次启动时再细化为可执行 checklist

## B0：技术基线 Spike

- PostgreSQL 16 + `pgvector` 扩展可用
- 中文检索最小 SQL 烟测可执行

### 4.2 明确不做

- 不做业务实体
- 不做编译流水线
- 不做查询接口
- 不做 WAL / 审批流 / Lint

### 4.3 交付物

- `.codex/P0-0 技术基线验证记录.md`
- 精确到 GAV 的依赖清单
- 最小可运行 Demo：
  - Boot Demo
  - Graph Demo
  - MCP Demo
  - PostgreSQL SQL Demo

### 4.4 验收标准

- 每个能力项都有 PASS/FAIL
- 每个 FAIL 项都有 fallback 或阻塞说明
- 锁定以下关键基线：
  - Spring Boot 版本
  - Spring AI BOM 版本

  - MCP Server 依赖坐标
  - Embedding 模型与维度
  - 中文检索方案
  - 构建入口（`mvn` 或 `./mvnw`）

### 4.5 Go/No-Go

- 若 Graph 或 MCP 依赖无法稳定装配，允许退化为：
  - 编排先用本地 Service 编排，不强依赖 Graph
  - MCP 延后到 B4 再接入
- 若中文检索扩展受限，退化为：
  - `pg_trgm + referential_keywords @> + 可配置 FTS`

### 5.1 范围

- Spring Boot 单应用骨架
- 最小配置文件
- Docker Compose
- Flyway 最小迁移
- 编译主链路：
  - `IngestNode`
  - `GroupNode`

### 5.2 首版必须收缩

## B3：稳定化与反馈闭环

> 目标：解决长任务稳定性、结果可修订、状态可恢复这三个核心问题

- 已完成：B3 主闭环代码已落地，已补齐 `RedisQueryCacheStore` / `ReviewResultParser` / `ReviewerAgent` / `PendingQueryService` / `confirm-correct-discard` / `Contribution` 写入 / 编译提交阶段 WAL 重试恢复，以及 `/api/v1/compile/retry`；2026-04-15 使用 JDK 21 执行 `mvn -s .codex/maven-settings.xml test`，全量 35 个测试通过；同时已复用本机既有 `redis` 容器完成 `redis-cli ping` 与 `RedisQueryCacheStoreIntegrationTests` 真实 Redis 烟测
- 已完成：此前 `docker compose up -d redis` 的阻塞并非 Docker 守护进程异常，而是 Docker Hub 拉取 `redis:7.4-alpine` 时 TLS 握手超时；当前已改为复用本机已有 `redis:alpine3.21` 容器完成验证
- 已完成：编译侧 WAL 存储已从进程内实现演进到 Redis 实现；新增 `RedisCompilationWalStore`（Hash 存 pending、Set 存 committed、TTL 24h）与 `CompilationWalProperties`；`MergedConcept` / `ConceptSection` 加 `@JsonCreator/@JsonProperty` 支持 Redis 序列化往返；补充 `RedisCompilationWalStoreIntegrationTests` 5 个场景全部通过（2026-04-15）；主代码中的内存 WAL 实现已移除，仅在测试中保留轻量 fake
- 下一步：B3 已全面收口，可考虑进入 B4 最小 MCP 接入，或先收集真实使用场景再决定

### 7.1 范围

- `ReviewerAgent`（单轮）
- `ReviewResultParser`
- Redis LLM Cache
- Redis WAL
- `PendingQueryService`
- `confirm / correct / discard`
- `Contribution` 写入闭环
- 如有必要，在本批补上向量生成与向量检索

### 7.2 重点约束

- v1.0 只允许单轮审查 + 单次修复
- 不引入多轮 Agent 循环
- PG 仍为唯一事实源
- WAL 只解决“恢复与补偿”，不承担永久存储职责

### 7.3 关键落地顺序

1. 先补 Redis Cache
2. 再补 `ReviewerAgent`
3. 再补 WAL 和补偿恢复
4. 再补 PendingQuery 的纠错闭环

### 7.4 验收标准

- 审查失败或超时有确定行为
- 编译中断后可恢复或重试
- `correct` 返回修订答案且仍保持 pending
- `confirm` 后能写入 contributions

### 7.5 本批产出价值

完成 B3 后，系统从 Demo 级演示进入可试运行阶段，具备最基本的“生成-审查-纠错-确认”闭环。

## 6. 数据层落地建议

原方案中数据层技术性很强，首轮不建议一步到位。

### 6.1 首轮推荐

- 简单 CRUD：JPA
- `tsvector` / `jsonb` / 数组 / HNSW / `@>` 查询：`JdbcTemplate` 或 jOOQ
- B1 暂不引入向量 ORM 映射；向量字段写入与检索后置到 B3，优先使用 `JdbcTemplate` 或 jOOQ 直接执行 SQL
- 若后续确实需要 JPA 映射向量列，再评估 `hibernate-vector`，不要默认自建 `PGvector` Converter
- Flyway 管控表结构

### 6.2 不建议首轮强行做到位的事项

- 所有 PG 特性全部走 JPA 映射
- 所有向量查询都封装进 Repository 抽象
- 在数据量很小时过早优化 HNSW 参数

### 6.3 原因

- 这类能力强依赖 PostgreSQL 原生语义
- 用 JPA 过度抽象会显著增加首版摩擦
- 当前目标是先打通主链路，而不是先追求 ORM 纯度

## 7. 风险与对应策略

| 风险点 | 风险说明 | 对应策略 |
|--------|---------|---------|
| Spring AI / Spring AI Alibaba 依赖漂移 | Graph、MCP、Model Starter 坐标可能变化 | B0 先锁精确版本和最小样例 |
| 中文检索扩展不稳定 | 部署环境可能不支持预期扩展 | 提前定义 `pg_trgm + @>` fallback |
| JPA 承载 PG 特性过重 | 首版开发成本高、调试困难 | 重 SQL 场景使用 `JdbcTemplate` 或 jOOQ |
| 双模型接入成本高 | 本地调试需要两套 Key | 允许本地先使用 stub/mock 或单模型降级 |
| WAL + PG 双写边界不清 | 容易把 Redis 当事实源 | 明确 PG 是唯一事实源，Redis 仅作暂存 |
| 向量检索收益不明确 | 数据量不足时收益有限 | B2 可先不上向量检索，B3 再补 |
| 范围再次膨胀 | 首版容易重新塞回审批流、Lint 等需求 | 批次验收前禁止跨批次加需求 |
