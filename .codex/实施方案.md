## P2：编译流水线核心（IngestNode → AnalyzeNode → MergeNode）

> 目标：Pipeline 前三阶段可单独运行，输入 sourceDir 能输出 List<MergedConcept>

### P2-1 多格式文件采集 — IngestNode

- [ ] `RawSource` DO（filePath、content、fileSize、format 枚举）
- [ ] `IngestNode` 实现：
  - 使用 `VirtualThreadTaskExecutor` 并行读取文件
  - 支持 `.md/.java/.xml/.yml/.json/.vue/.js` 文本类（直接读取前 8K 字符）
  - 支持 `.pdf`（Apache PDFBox 提取文本）
  - 支持 `.xlsx`（Apache POI 逐 Sheet 提取）
  - 跳过规则：`node_modules, .git, target, dist, .class, .jar`
  - 大文件策略：超过 8KB 仅索引前 8K 字符（与原版一致）
- [ ] 单元测试：覆盖各格式采集 + 跳过规则验证

### P2-2 动态分组 — GroupNode

- [ ] `GroupNode`：按配置规则动态分组，禁止在代码和文档中硬编码具体系统名
- [ ] 配置外化：`application.yml` 中定义 `grouping.rules`
  - 支持路径前缀 / glob / regex 到 `groupKey` 的映射
  - 未命中显式规则时，可回退到顶层目录名
  - 无稳定目录结构时，落入 `defaultGroup`
- [ ] 分组结果：`Map<String, List<RawSource>>`（key 为动态 `groupKey`）

### P2-3 分批切割 — BatchSplitNode

- [ ] `BatchSplitNode`：每批不超过 40K 字符
- [ ] 批次 ID 生成：`SHA256(groupKey + batchIndex + sourceFileList)`
- [ ] 输出：`List<SourceBatch>`（含 batchId、groupKey、sources）

### P2-4 LLM 分析节点 — AnalyzeNode（含 4 层 JSON 容错）

这是最复杂的节点，需单独评审。

- [ ] `ConceptAnalysis` DO（概念列表、关系列表、争议列表）
- [ ] `AnalyzeNode` 核心逻辑：
  - 查询 `LlmCacheRedisRepository`（缓存 Key = `SHA256(model+systemPrompt+userPrompt)`）
  - 缓存命中则直接返回，TTL = 604800s（7天）
  - 缓存未命中则调用 OpenAI API（gpt-4o），结果写入缓存
- [ ] **4 层 JSON 容错解析**（`JsonRepairService`）：
  - Layer 1：`objectMapper.readValue` 标准解析
  - Layer 2：`TruncationRepairService` — 正则定位最后完整 JSON 对象（Pattern: `"concepts"\s*:\s*\[`），标记 `truncated=true`
  - Layer 3：`SalvageService` — 逐段正则抢救 id/title/description（Pattern: `(?:\{\s*"id"\s*:\s*"([^"]+)"\s*,\s*"title"\s*:\s*"([^"]+)")`），标记 `salvaged=true`
  - Layer 4：空结果保护 — 返回 EMPTY，不写缓存，不写 PG，记录 WARN
- [ ] `TruncationRepairService` 单元测试：覆盖截断场景

### P2-5 跨分组概念合并 — CrossGroupMergeNode

- [ ] `CrossGroupMergeNode`：合并所有动态分组的概念分析结果，按 concept_id 去重（142→16 概念目标）
- [ ] 合并策略：相同 concept_id 时合并 sources 列表，取最丰富描述
- [ ] 输出：`List<MergedConcept>`

### P2-6 Spring AI Alibaba Graph 接线（Phase 1-3）

- [ ] 使用 Spring AI Alibaba Graph API 声明 Phase1-3 的 DAG：
  ```
  START → IngestNode → GroupNode → BatchSplitNode
       → [并行] AnalyzeNode（每个 Batch）
       → CrossGroupMergeNode
  ```
- [ ] 配置节点超时（AnalyzeNode: 300s/batch）
- [ ] 配置分布式锁（analyze:lock:{batchId}）

**P2 验收标准**：对测试 sourceDir 运行编译，能输出 `List<MergedConcept>` 且 JSON 容错解析各层均有测试覆盖

## P3：编译流水线 — 审查写入阶段

> 目标：完整的 CompilerAgent → ReviewerAgent → WAL → IdempotentCommit 链路

### P3-1 Redis WAL 状态机

- [ ] `StagingStatus` 枚举：`QUEUED / COMPILING / PENDING_REVIEW / REVIEWING / READY / FIXING / REVIEW_SKIPPED / COMMITTED`
- [ ] `StagingRecord` DO（conceptId、status、content、retryCnt、reviewResult、createdAt）
- [ ] `WalService`：
  - `enqueue(jobId, conceptId)` → HSET staging:{jobId}:{conceptId} status QUEUED，TTL 7200s
  - `updateStatus(jobId, conceptId, status)` → HSET ... status {status}
  - `markReady(jobId, conceptId)` → HSET ... status READY + SADD staging:ready:{jobId} {conceptId}
  - `getReadyConceptIds(jobId)` → SMEMBERS staging:ready:{jobId}
  - `getRecord(jobId, conceptId)` → HGETALL staging:{jobId}:{conceptId}
  - `cleanup(jobId, conceptId)` → HDEL + SREM
- [ ] `WalService` 单元测试（Mock Redis）

### P3-2 CompilerAgent — 文章编译

- [ ] `CompilerAgent` 实现：
  - 使用 `spring-ai-starter-model-openai`（模型名配置化，v1.0 默认 `gpt-4o`），超时 180s
  - System Prompt：对应原版 `SYSTEM_COMPILE_ARTICLE`，外化为 `classpath:prompts/compile-article.txt`
  - 输入：`MergedConcept` + 相关源文件 Chunks（最多 `sourceMaxCharsForCompile` 字符）
  - 输出：完整 Markdown 文章字符串
- [ ] 编译完成后更新 WAL 状态：`COMPILING → PENDING_REVIEW`，content 写入 Redis Hash

### P3-3 ReviewerAgent — 对抗审查（v1.0 单轮）

- [ ] `ReviewResult` DO（pass: boolean、issues: List<Issue>、reviewStatus 枚举）
- [ ] `Issue` DO（severity: HIGH/MEDIUM/LOW、description、category）
- [ ] `ReviewerAgent` 实现：
  - 使用 `spring-ai-starter-model-anthropic`（模型名配置化），超时 120s（通过客户端配置 + TimeLimiter 实现）
  - System Prompt：嵌入 4 个检查维度（明确性知识完整性、溯源标注、跨系统一致性、知识分类正确性）
  - 外化为 `classpath:prompts/review-article.txt`
  - 超时则返回 `ReviewStatus.TIMEOUT`
- [ ] **ReviewerAgent 输出 4 层降级解析**（`ReviewResultParser`）：
  - 层 1：标准 JSON Schema 校验（pass + issues 字段）
  - 层 2：宽松 JSON 解析（容忍额外字段）
  - 层 3：正则提取 Issue（Pattern: `(?:问题|Issue|缺失|missing|遗漏|incorrect)[^\n]*?：?\s*([^\n]{10,200})`）
  - 层 4：兜底策略（超时 → TIMEOUT_FALLBACK，否则 → PARSE_FAILED）
- [ ] `ReviewResultParser` 单元测试：覆盖各层降级场景

### P3-4 AutoFixNode — 问题修复

- [ ] `AutoFixNode`：
  - 取 ReviewResult 中 Top5 Issues（按 severity 排序）
  - 构造修复上下文注入格式（`=== 审查意见 ===` 块 + 原始文章 + 补充源材料）
  - 调用 `CompilerAgent` 重写文章
- [ ] 修复后更新 WAL 状态：`FIXING → READY`

### P3-5 IdempotentCommitNode — 幂等提交与补偿恢复

- [ ] `IdempotentCommitNode` 实现：
  - `SMEMBERS staging:ready:{jobId}` 获取所有 READY 概念
  - 对每个概念：PG 事务内 UPSERT articles + article_chunks（文本、位置、统计信息，不在此阶段生成 embedding）
  - PG 提交成功后：写入 Obsidian Vault 文件（`concepts/{id}.md`，幂等）
  - 写文件成功后：HDEL staging:{jobId}:{conceptId} + SREM staging:ready:{jobId} {conceptId}
  - PG 失败则 Redis 保留（可重试），Redis 清理失败则依赖 PG UPSERT 幂等 + 定时对账恢复
- [ ] 集成测试：模拟 PG 失败，验证 Redis 暂存保留

### P3-6 BuildIndexNode — FTS + 向量索引更新

- [ ] `BuildIndexNode`：
  - 触发 PG 触发器更新 fts_vector（INSERT/UPDATE 自动触发，无需手动 UPDATE）
  - 调用 Embedding 模型（v1.0 默认 `text-embedding-3-small`）为文章 chunks 生成向量
  - 批量 UPDATE `article_chunks.embedding`
  - 按 article 聚合生成 `article_embeddings`
- [ ] Embedding 调用使用 `VirtualThreadTaskExecutor` 并行

### P3-7 SynthesisNode — 综合文档生成

- [ ] `SynthesisNode`：生成 index/timeline/tradeoffs 综合文档（调用 OpenAI 生成 Markdown）

### P3-8 Spring AI Alibaba Graph 接线（Phase 4-5）

- [ ] 将 P3-1~P3-7 节点接入 DAG：
  ```
  CrossGroupMergeNode
  → [并行，每个 Concept] StagingNode → CompilerNode → ReviewNode
    → [条件分支] AutoFixNode → MarkReadyNode
    → [超时分支] ForceCommitNode
  → IdempotentCommitNode → BuildIndexNode → SynthesisNode → END
  ```
- [ ] `maxReviewRounds` 从配置读取，默认值固定为 `1`；仅作为 v2.0 预留能力，不纳入 v1.0 验收范围
- [ ] `reviewTimeoutSec=120`（外化配置）

### P3-9 文章生命周期定时任务

- [ ] `ArticleLifecycleScheduler`（`@Scheduled` 每日凌晨）：
  - 扫描 `source_files.file_mtime`，比较 `file_mtime vs articles.compiled_at`
  - 过时则 `UPDATE articles SET lifecycle='STALE'`
  - 发布 `ArticleStaleEvent`，由 Spring `ApplicationEventPublisher` 异步分发
  - `ArticleStaleEventListener` 接收事件后直接调用编译服务触发增量重编（单机内直接调用，无需消息队列）

**P3 验收标准**：端到端编译测试，输入测试 sourceDir，PG 中能查到文章，Vault 目录有对应 .md 文件，WAL 状态最终清空

## P4：查询引擎

> 目标：完整的混合检索 + RRF 融合 + PendingQuery 状态机

### P4-1 三路并行检索

- [ ] `FtsSearchService`：
  - `plainto_tsquery(:ftsConfig, :userQuery)` + `fts_vector @@ query`（默认 `lattice_fts_cfg`）
  - `ts_rank_cd` 排名，LIMIT 20
  - 结果：`List<SearchResult>`（conceptId、rankScore、rank）
- [ ] `VectorSearchService`：
  - 查询文本 → Embedding 向量（v1.0 默认 `text-embedding-3-small`）
  - `article_chunks.embedding <=> :queryVector` HNSW 查询
  - 结果聚合到 article 维度（取 chunk 最高相似度），LIMIT 20
- [ ] `RefKeySearchService`：
  - 从查询文本正则提取业务码（businessSubTypeCode 等枚举格式）
  - `referential_keywords @> :extractedCodes` 精确查询
  - 同时支持 `kw ~* :regexPattern` 正则模糊匹配

### P4-2 RRF 融合排名算法

- [ ] `RrfFusionService`：
  - 权重配置：FTS=2.0，Vector=1.5，RefKey=3.0（外化配置）
  - 公式：`score += weight / (60 + rank + 1)`
  - positionBonus：Rank1 +0.05，Rank2-3 +0.02
  - verbatim 文件 `score *= 1.5`
  - 取 Top7 文章 ID

### P4-3 LLM 答案生成

- [ ] `AnswerGenerationService`：
  - 读取 Top7 文章全文 + source_chunks
  - 查询 `LlmCacheRedisRepository`（key = SHA256(query + articleIds)）
  - 命中则返回缓存（TTL 3600s）
  - 未命中则调用 LLM 生成答案，结果写入 Redis

### P4-4 PendingQuery 状态机

- [ ] `PendingQueryService`：
  - 生成 queryId（UUID）
  - Redis SETEX `pending:{queryId}` 604800s {queryState}
  - PG 同时写入 pending_queries 表（审计副本）
- [ ] 提供 confirm / correct / discard 三个操作：
  - `confirm(queryId)` → INSERT contributions + DELETE pending_queries
  - `correct(queryId, correction)` → 读取 pending 记录 → 重建 `selected_article_ids/source_file_paths` 上下文 → 生成 revised answer → UPDATE `pending_queries.answer/corrections/expires_at`
  - `discard(queryId)` → DELETE pending_queries + DEL Redis key

### P4-5 QueryController

- [ ] `QueryController`（`/api/v1/query`）：
  - `POST /query` — 触发三路检索 + RRF + LLM，返回 `{answer, queryId, sources}`
  - `POST /query/{queryId}/confirm`
  - `POST /query/{queryId}/correct` — 返回 `{revisedAnswer, queryId, correctionRounds, sources}`
  - `POST /query/{queryId}/discard`

**P4 验收标准**：基于 P3 产出的测试知识库发起查询请求，能返回正确答案 + queryId；`correct` 后能返回修订答案且记录仍保持 pending；`confirm` 后能在 contributions 表查到带 corrections 的记录

## v1.0 → v2.0 边界说明

以下能力有**外部依赖或部署环境约束**，v1.0 不实现，待 v2.0 完成：

| 能力 | 阻塞原因 | v2.0 计划 |
|------|---------|---------|
| 微服务拆分 | 无流量数据支撑拆分决策，提前拆只增复杂度无收益 | 观察 v1.0 瓶颈后按需拆分为独立进程 |
| NFS Vault 挂载 | 依赖 K8s 部署环境，单机不适用 | K8s CSI / NFS，评估文件锁定风险后实施 |
| 审批流对接 | 依赖公司内部审批系统 API，属外部依赖 | 对接后替换 v1.0 的 Mock 实现 |
| 双轮审查循环 | 需要额外验证 Agent Framework/重试收益，v1.0 先不放大范围 | v2.0 再评估 `maxReviewRounds > 1` |

> **v1.0 明确纳入范围的增强项**：级联重编触发（`ApplicationEvent` 单机内直接驱动，无需消息队列）、Redis 单节点（单机运行最终形态，Cluster 仅预留）。

## 风险与注意事项

| 风险点 | 说明 | 应对措施 |
|-------|------|---------|
| Spring AI / Spring AI Alibaba 依赖映射 | Graph、Agent Framework、MCP Server 分属不同模块，starter 名称若选错会导致 P0 阻塞 | 把 `P0-0` 作为硬前置，先锁定精确 artifact、版本和最小可运行样例 |
| 中文分词方案与部署环境兼容性 | 业务码（1210/1310 等）可能被错误切分，且扩展可用性受部署形态影响 | 使用双写策略：`referential_keywords` 同时存储完整字符串和纯数字；扩展不可用时退化到 `pg_trgm + @>` |
| JPA + HikariCP + 虚拟线程 | HikariCP 与虚拟线程存在已知 pinning 问题 | 使用 HikariCP 5.1+，`spring.datasource.hikari.allow-pool-suspension=false` |
| OpenAI + Claude 双 API 密钥管理 | 本地开发需要两套密钥 | 使用 `.env.local`（git ignore），生产使用 K8s Secret |
| Embedding 模型与向量维度耦合 | 模型切换会影响 `vector(n)` 列定义与索引重建 | v1.0 固定 `text-embedding-3-small + vector(1536)`，升级模型单独走迁移 |
| 向量索引冷启动 | 文章数量 < 100 时 HNSW 查询质量无法验证 | 准备充足的测试数据集（至少 50 篇文章） |
