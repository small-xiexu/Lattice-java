## 一、结论先行

### 1.1 原始项目的真实形态

原始 Lattice 不是一个“Web 后台 + 中心化数据库产品”，而是一个以 **CLI / MCP 驱动** 的知识编译器，核心载体是：

- `concepts/*.md`：知识文章正文
- `_contributions/*.md`：确认后的用户贡献
- `_meta/pending-queries.jsonl`：待确认查询
- `_meta/search.db` / `_meta/llm-cache.db`：SQLite 检索与缓存加速层
- `_registry/`：小文件 / pdf / xlsx verbatim 保存
- Git：快照、历史、回滚

也就是说，原版的核心价值是“能力闭环”，不是“必须沿用文件系统 + SQLite 的实现细节”。

### 1.2 本文档的对齐原则

本次方案采用两层定义：

1. **能力对齐**：Java 版必须补齐原版已经具备的编译、查询、反馈、治理闭环。
2. **实现增强**：Java 版可以继续使用 PostgreSQL / Redis / Spring AI / Flyway / MCP / REST，但这些属于 Java 工程化选择，不等同于原版实现。

因此，本文中凡是涉及以下内容，统一视为 **Java 增强项**，而不是“原版 parity 必需项”：

- PostgreSQL / Redis 替代 SQLite / 文件落盘
- Spring AI Alibaba StateGraph
- Web 管理后台
- `pg_jieba`
- `pgvector`
- 异步任务 / jobId / 后台调度

### 1.3 v1.2 文档的主要问题

相比 v1.2，当前版本修正以下偏差：

- 不再把 Web 后台、`pg_jieba`、向量检索写成“对齐原版”的前置项
- 不再把查询主链误写成“文章不够时才去搜源文件”
- 不再把 `README` 写成高于 `overview / domain / CLAUDE / 架构` 的最高优先级
- 不再把 `review_status` 写成 `needs_review`，统一对齐原版语义为 `needs_human_review`
- 补齐 `lattice_query_discard`
- 补齐 `inspect + import-answers` 这条人工确认闭环
- 文件路径统一按当前 Java 仓库真实结构描述，避免继续沿用不存在的 `compiler/node/...`、`PendingQueryManagerImpl.java` 等旧路径

## 二、原始项目源码能力地图

### 2.1 采集层

原版 `readRawSources()` 实际支持：

| 类型 | 原版支持 | 说明 |
|------|---------|------|
| 文本类 | `.md .txt .markdown .java .xml .properties .yml .yaml .json .vue .js .css .html .sh .py` | 直接读取文本；Markdown 解析 frontmatter |
| PDF | `.pdf` | 使用 `unpdf` 提取文本，无可提取文本则跳过 |
| Excel | `.xlsx .xls` | 转为 CSV 风格文本 |
| 图片 | `.png .jpg .jpeg .gif .svg .drawio` | 只记录路径，不提取正文 |

注意：

- 原版 **不支持** `.docx`
- 原版 `.html` 是按文本读，不做专门 DOM 清洗

### 2.2 编译主链

原版编译主链可以概括为：

1. 多格式采集
2. 按系统分组
3. `selectKeyFiles()` 选高价值文件
4. 大输入按 `40K chars` 分批分析
5. LLM 输出概念 JSON，解析失败时做截断修复
6. 跨系统 merge 为更高层概念
7. `selectRelevantSections()` 按概念词挑源文件相关章节
8. 编译文章
9. 对抗审查
10. 自动修正
11. 先写 `_staging`，再原子提交
12. 生成 `index / timeline / tradeoffs / gaps`
13. 记录 manifest

### 2.3 查询与反馈闭环

### 2.4 治理与维护
