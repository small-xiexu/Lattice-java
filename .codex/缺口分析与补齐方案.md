### G1 — `lattice_correct` 只预览不纠错

#### 现状

`LatticeMcpTools.correctKnowledge()` 调用 `PropagationService.propagate(conceptId, correctionSummary)`，只做 BFS 图遍历返回下游列表，**文章 content 从未被修改**。

原版行为：读文章 → 找源文件证据 → LLM 交叉验证 → LLM 重写文章 → 写回 DB → 标记下游 → 返回结果。

#### 补齐方案

**新增文件**：`governance/ArticleCorrectionService.java`

```
依赖注入（构造器）：
  ArticleJdbcRepository articleJdbcRepository
  SourceFileJdbcRepository sourceFileJdbcRepository
  ArticleSnapshotJdbcRepository articleSnapshotJdbcRepository
  DependencyGraphService dependencyGraphService
  LlmGateway llmGateway

方法签名：
  public ArticleCorrectionResult correct(String conceptId, String correctionSummary)

执行流程：
1. articleJdbcRepository.findByConceptId(conceptId)
   → 找不到则抛 IllegalArgumentException("概念不存在: " + conceptId)
2. 从 articleRecord.getSourcePaths() 列表，逐一调用
   sourceFileJdbcRepository.findByPath(sourcePath)，收集 contentText（最多取前 3 个，各截取至 3000 chars）
3. 拼 crossValidatePrompt = 当前文章内容 + 用户纠正摘要 + 源文件内容片段
4. String validationJson = llmGateway.compile("cross-validate", SYSTEM_CROSS_VALIDATE, crossValidatePrompt)
   → 解析出 {"supported": boolean, "evidence": "..."}
5. 拼 correctionPrompt = 原文章 content + 用户纠正摘要 + validationJson.evidence（若 supported=true 则包含，否则空串）
6. String revisedContent = llmGateway.compile("apply-correction", SYSTEM_APPLY_CORRECTION, correctionPrompt)
7. 构造更新后的 ArticleRecord（复制原记录，替换 content，reviewStatus="needs_review"）
   → articleJdbcRepository.upsert(updatedRecord)
8. 构造 ArticleSnapshotRecord（snapshotId=-1，snapshotReason="correction"，capturedAt=OffsetDateTime.now()，其余字段复制更新后的记录）
   → articleSnapshotJdbcRepository.save(snapshotRecord)
9. DependencyGraphSnapshot snapshot = dependencyGraphService.snapshot()
   → 遍历 snapshot.getEdges() 找所有下游概念 ID（BFS，深度≤3）
   → 收集为 List<String> downstreamIds
10. 返回 ArticleCorrectionResult{conceptId, revisedContent, downstreamIds, validationSupported}
```

**新增数据类**：`governance/ArticleCorrectionResult.java`

```java
// 字段（均为 final）：
String conceptId
String revisedContent   // LLM 生成的完整修正后文章
List<String> downstreamIds  // 下游受影响的概念 ID 列表
boolean validationSupported // 源文件是否支持此次纠正
```

**新增仓储方法**：`ArticleSnapshotJdbcRepository.save(ArticleSnapshotRecord record)`

```sql
INSERT INTO article_snapshots (
    concept_id, title, content, lifecycle, compiled_at,
    source_paths, metadata_json, summary, referential_keywords,
    depends_on, related, confidence, review_status,
    snapshot_reason, captured_at
) VALUES (?, ?, ?, ?, ?, ?, ?::jsonb, ?, ?, ?, ?, ?, ?, ?, ?)
```

注意：`snapshot_id` 是 `BIGSERIAL`，不需要传入，传 -1 占位即可（SQL 中不包含该列）。
`source_paths / referential_keywords / depends_on / related` 用 `connection.createArrayOf("text", ...)` 转换。

### G2 — `propagate` 只计算影响，下游文章未真正更新

#### 现状

`PropagationService.propagate()` 只返回下游列表。下游文章内容从未被 LLM 重写。

#### 补齐方案

**新增文件**：`governance/PropagateExecutionService.java`

```
依赖注入（构造器）：
  ArticleJdbcRepository articleJdbcRepository
  ArticleSnapshotJdbcRepository articleSnapshotJdbcRepository
  LlmGateway llmGateway

方法签名：
  public PropagationExecutionResult executePropagation(String rootConceptId)

执行流程：
1. 查询有 upstream_corrections 指向 rootConceptId 的下游文章：
   articleJdbcRepository.findWithUpstreamCorrections(rootConceptId)
   → 返回 List<ArticleRecord>

2. 读取上游文章内容（rootConceptId 对应的文章）：
   Optional<ArticleRecord> root = articleJdbcRepository.findByConceptId(rootConceptId)
   若不存在则返回 PropagationExecutionResult{0, 0, 0}

3. 从 root.getMetadataJson() 中解析上游纠正摘要列表（从 upstream_corrections 数组中过滤 from=rootConceptId 的条目）
   实际上此处是对下游文章的 metadata_json.upstream_corrections 做过滤，找到 from=rootConceptId 的条目获取 summary

4. 对每篇下游文章：
   a. 拼 checkPrompt = 纠正摘要 + 下游文章 content
   b. String checkJson = llmGateway.compile("check-propagation", SYSTEM_CHECK_PROPAGATION_NEEDED, checkPrompt)
      → 解析 {"affected": boolean, "reason": "..."}
   c. 若 affected=true：
      - 拼 applyPrompt = 纠正摘要 + 下游文章 content
      - String updatedContent = llmGateway.compile("apply-propagation", SYSTEM_APPLY_PROPAGATION, applyPrompt)
      - 在 updatedContent 末尾（文章正文结尾，YAML frontmatter 之前）追加：
        <!-- propagated-fix: rootConceptId, {今天日期} -->
      - 构造更新后 ArticleRecord（保留原有字段，替换 content，reviewStatus="needs_review"）
      - articleJdbcRepository.upsert(updatedRecord)
      - articleSnapshotJdbcRepository.save(snapshotRecord)  // snapshotReason="propagation"
      - updated++
   d. 无论 affected 是否 true：
      - articleJdbcRepository.clearUpstreamCorrection(下游 conceptId, rootConceptId)
      - processed++
   e. 若 affected=false：skipped++

5. 返回 PropagationExecutionResult{processed, updated, skipped}
```

**新增数据类**：`governance/PropagationExecutionResult.java`

```java
int processed   // 总处理数
int updated     // 实际重写数（LLM 判断受影响）
int skipped     // 跳过数（LLM 判断不受影响）
```

**新增仓储方法**：`ArticleJdbcRepository`

```java
// 查找有 upstream_corrections 来自 rootConceptId 的下游文章
public List<ArticleRecord> findWithUpstreamCorrections(String fromConceptId)
```

```sql
SELECT <全部字段>
FROM articles
WHERE metadata_json::jsonb @> jsonb_build_array(
    jsonb_build_object('from', ?)
)::jsonb                       -- 不准确，改用 EXISTS
```

更准确的 SQL：

```sql
SELECT <全部字段>
FROM articles
WHERE EXISTS (
    SELECT 1
    FROM jsonb_array_elements(COALESCE(metadata_json::jsonb->'upstream_corrections', '[]'::jsonb)) AS elem
    WHERE elem->>'from' = ?
)
```

```java
// 清除下游文章 metadata_json 中来自特定上游的 upstream_corrections 条目
public void clearUpstreamCorrection(String downstreamConceptId, String fromConceptId)
```

```sql
UPDATE articles
SET metadata_json = jsonb_set(
    COALESCE(metadata_json::jsonb, '{}'),
    '{upstream_corrections}',
    (
        SELECT COALESCE(jsonb_agg(elem), '[]'::jsonb)
        FROM jsonb_array_elements(COALESCE(metadata_json::jsonb->'upstream_corrections', '[]'::jsonb)) AS elem
        WHERE elem->>'from' <> ?
    )
)::text
WHERE concept_id = ?
```

参数：fromConceptId, downstreamConceptId。

**新增 MCP 工具**（`LatticeMcpTools.java`）：

```

### G3 — Rollback / 快照还原缺失

#### 现状

`SnapshotService.snapshot(limit)` 只读，`ArticleSnapshotJdbcRepository` 无 `findBySnapshotId()` 和 `save()` 方法。

#### 补齐方案

**新增仓储方法**：`ArticleSnapshotJdbcRepository.findBySnapshotId(long snapshotId)`

```sql
SELECT snapshot_id, concept_id, title, content, lifecycle, compiled_at,
       source_paths, metadata_json, summary, referential_keywords,
       depends_on, related, confidence, review_status, snapshot_reason, captured_at
FROM article_snapshots
WHERE snapshot_id = ?
```

**修改 `SnapshotService.java`**，新增方法：

```
public RollbackResult rollback(String conceptId, long snapshotId)

执行流程：
1. articleSnapshotJdbcRepository.findBySnapshotId(snapshotId)
   → 找不到则抛 IllegalArgumentException("快照不存在: " + snapshotId)
2. 校验 snapshot.getConceptId().equals(conceptId)
   → 不匹配则抛 IllegalArgumentException("快照与概念不匹配")
3. 构造 ArticleRecord（从快照字段恢复，reviewStatus 保持快照值）
4. articleJdbcRepository.upsert(articleRecord)
5. 写入新快照留痕（snapshotReason="rollback|from:" + snapshotId）
   articleSnapshotJdbcRepository.save(rollbackSnapshotRecord)
6. 返回 RollbackResult{conceptId, restoredSnapshotId: snapshotId, restoredAt: OffsetDateTime.now()}
```

**新增数据类**：`governance/RollbackResult.java`

```java
String conceptId
long restoredSnapshotId
OffsetDateTime restoredAt
```

**新增 MCP 工具**：

```
@McpTool(name = "lattice_rollback",
         description = "Restore an article to a previous snapshot version")
public String rollback(
    @McpArg(name="conceptId") String conceptId,
    @McpArg(name="snapshotId") long snapshotId
)

实现：
  RollbackResult result = requireSnapshotService().rollback(conceptId, snapshotId)
  返回 JSON：{conceptId, restoredSnapshotId, restoredAt}
```

配合已有工具：`lattice_history(conceptId)` → 获取 snapshotId 列表 → 再调 `lattice_rollback`。

### G4 — selectKeyFiles 文件优先级打分未实现

#### 现状

`BatchSplitNode.split()` 按传入顺序均匀分批，高重要性文件可能落在靠后批次。

#### 补齐方案

**新增文件**：`compiler/service/FileRankingService.java`

```
依赖注入：CompilerProperties compilerProperties（无需 Spring，直接 new 也可）

方法签名：
  public List<RawSource> rank(List<RawSource> sources)

实现：
1. 对每个 RawSource，按以下规则计算 score（文件名不区分大小写，用 toLowerCase()）：
   文件名模式（starts-with / matches）     分值
   ─────────────────────────────────────   ────
   overview.*                              100
   readme.*                                 95
   claude.md                                95
   architecture.*                           85
   domain.*                                 80
   design.*                                 75
   api.*                                    70
   config.*                                 60
   其余文件：max(10, 50 - content.length() / 10000)
             （文件内容越长优先级越低，避免超大文件挤占第一批）

2. 若 compilerProperties.getFileRanking().getRules() 非空，
   优先用用户自定义规则（pattern → score），覆盖默认规则

3. 按 score 降序排列，score 相同时按 relativePath 字典序稳定排序
4. 返回排序后列表（不修改原列表）
```

**修改 `CompilerProperties.java`**，新增内部配置类和字段：

```java
// 新增字段：
private FileRanking fileRanking = new FileRanking();

// getter/setter：
public FileRanking getFileRanking() { return fileRanking; }
public void setFileRanking(FileRanking fileRanking) { this.fileRanking = fileRanking; }

// 新增内部类：
public static class FileRanking {
    private List<FileRankingRule> rules = new ArrayList<>();
    public List<FileRankingRule> getRules() { return rules; }
    public void setRules(List<FileRankingRule> rules) { this.rules = rules; }
}

public static class FileRankingRule {
    private String pattern;   // 文件名 glob 模式（如 "payment.*"）
    private int score;        // 优先级分值
    // getter/setter ...
}
```

YAML 配置（`application.yml` 新增）：

```yaml
lattice:
  compiler:
    file-ranking:
      rules:
        - pattern: "overview.*"
          score: 100
        - pattern: "readme.*"
          score: 95
```

**修改 `BatchSplitNode.java`**：

```
新增构造器参数：FileRankingService fileRankingService（可为 null，null 时跳过排序）

split(String groupKey, List<RawSource> rawSources) 改为：
1. List<RawSource> ranked = fileRankingService != null
       ? fileRankingService.rank(rawSources)
       : rawSources
2. 后续分批逻辑不变，只是把迭代对象从 rawSources 换成 ranked
```

`BatchSplitNode` 已经是 `new` 创建，不是 Spring Bean，修改构造器不影响 Spring 装配。在实例化 `BatchSplitNode` 的上层（`CompilePipelineService`）传入新建的 `FileRankingService` 实例即可。

### G6 — per-call token 上限未实现

#### 现状

`LlmProperties` 无 `maxInputChars` 字段。上传超大文件时，单次 API 调用可能因 context window 超限而报错。

#### 补齐方案

**修改 `LlmProperties.java`**，新增字段（紧接 `reviewEnabled` 之后）：

```java
/** 单次 LLM 调用的最大输入字符数（字符数 / 4 粗估 token 数）。默认 64000 chars ≈ 16K tokens */
private int maxInputChars = 64000;

public int getMaxInputChars() { return maxInputChars; }
public void setMaxInputChars(int maxInputChars) { this.maxInputChars = maxInputChars; }
```

**修改 `LlmGateway.java`**，在 `invoke()` 方法中，`ensureBudgetAvailable()` 之后、`llmClient.call()` 之前，插入截断守卫：

```
int totalInputLength = systemPrompt.length() + userPrompt.length();
if (totalInputLength > llmProperties.getMaxInputChars()) {
    int budget = llmProperties.getMaxInputChars() - systemPrompt.length() - 200;
    if (budget > 0) {
        userPrompt = userPrompt.substring(0, budget)
                   + "\n\n[... 内容已截断，超出单次调用字符限制 ...]";
    } else {
        // systemPrompt 本身就超限了，直接报警（不截断 system prompt，防止逻辑破坏）
        log.warn("System prompt length {} exceeds maxInputChars {}, skipping truncation",
                systemPrompt.length(), llmProperties.getMaxInputChars());
    }
    log.warn("LLM input truncated: original={} chars, limit={} chars, purpose={}",
             totalInputLength, llmProperties.getMaxInputChars(), purpose);
}
```

注意：`userPrompt` 在 `invoke()` 方法中是参数，需要先声明为局部变量副本，不直接修改形参：

```java
String truncatedUserPrompt = userPrompt;  // 局部副本
// ... 截断逻辑操作 truncatedUserPrompt ...
LlmCallResult llmCallResult = llmClient.call(systemPrompt, truncatedUserPrompt);
```

**`application.yml` 新增配置**：

```yaml
lattice:
  llm:
    max-input-chars: 64000
```

### G7 — `lint --fix` 自动修复缺失

#### 现状

`LintIssue` 只有 3 个字段（dimension / targetId / message），`LintService.lint()` 有 7 处 `new LintIssue(...)` 调用。直接修改构造器会破坏现有调用。

#### 补齐方案

**修改 `LintIssue.java`**，新增 2 个字段，**保留原 3 参数构造器**，新增 5 参数构造器：

```java
// 新增字段（均为 final）：
private final boolean fixable;
private final String fixSuggestion;   // null 表示无自动修复建议

### G10 — 语义分块算法粗糙

#### 现状

需确认 `ArticleChunkJdbcRepository` 的当前分块实现。若是固定步长，则检索时分块可能在句子或段落中间截断，影响向量检索质量。

#### 补齐方案

**新增文件**：`infra/chunking/SemanticChunker.java`

```
职责：按 Markdown 语义边界对文本进行分块

方法签名：
  public List<TextChunk> chunk(String content, int maxChars, float overlapRatio)

断点评分规则：
  断点类型                    评分
  ─────────────────────────   ────
  行以 "# " 开头（一级标题）   100
  行以 "## " 开头             90
  行以 "### " 开头             80
  行以 "#### " 开头            70
  行以 "##### " 开头           60
  行以 "###### " 开头          50
  行以 "```" 开头              80  （代码围栏开始/结束）
  行以 "---" 开头              60
  空行（连续空行算一次）        20
  列表项（行以 "- " 或 "* " 或 数字+"." 开头）  5

算法：
1. 按行分割 content，记录每行起始 charOffset
2. 为每个位置计算断点评分
3. 代码围栏追踪：遇到 ``` 则 inCodeBlock 取反，inCodeBlock=true 时所有位置评分清零（不在代码块内断裂）
4. 贪心策略：从 pos=0 开始，在 [pos, pos+maxChars] 范围内找评分最高的断点；
   若无高分断点（均为 0），则在 pos+maxChars 处强制切断
5. 重叠计算：下一块起始位置 = 当前断点 - (maxChars * overlapRatio)，向前找最近的空行
6. 返回 List<TextChunk>，每个 TextChunk{text, charOffset, chunkIndex}

默认参数：maxChars=3600, overlapRatio=0.15f
```

**新增数据类**：`infra/chunking/TextChunk.java`

```java
String text
int charOffset   // 在原始内容中的起始字符偏移
int chunkIndex   // 块序号（从 0 开始）
```

**修改 `ArticleChunkJdbcRepository.java`** 和 **`SourceFileChunkJdbcRepository.java`**：

```
在执行分块前注入 SemanticChunker（new SemanticChunker()，无需 Spring Bean）
将固定步长分割改为 semanticChunker.chunk(content, 3600, 0.15f)
```

**数据迁移策略**：

已有的 chunks 数据是用旧算法生成的。上线时建议：
1. 在 V15 迁移中 `TRUNCATE article_chunks; TRUNCATE source_file_chunks;`
2. 或提供一次性重建脚本：调用 `ArticleChunkJdbcRepository.rebuildAll()` 方法
   → 读取所有 articles 的 content，重新分块写入

不建议懒加载重建（会导致检索结果不一致）。
