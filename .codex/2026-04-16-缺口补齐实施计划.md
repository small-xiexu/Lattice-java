# 缺口补齐实施计划

> **For agentic workers:** REQUIRED: Use superpowers:subagent-driven-development (if subagents available) or superpowers:executing-plans to implement this plan. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 基于 [`.codex/缺口分析与补齐方案.md`](/Users/sxie/xbk/Lattice-java/.codex/%E7%BC%BA%E5%8F%A3%E5%88%86%E6%9E%90%E4%B8%8E%E8%A1%A5%E9%BD%90%E6%96%B9%E6%A1%88.md) 补齐 G1-G10 剩余缺口，完成知识纠错、传播、回滚、编译质量、质量趋势、源文件读取和可扩展性等核心治理闭环。

**Architecture:** 以现有 JDBC + MCP + Spring Boot 架构为基础，优先补齐最影响正确性与稳定性的闭环能力，再逐步增强编译质量、治理趋势、源文档读取与可扩展性。每个缺口都通过“仓储/服务/MCP/测试/迁移”闭环完成，优先采用小步 TDD 和独立可回归任务。

**Tech Stack:** Java 21、Spring Boot 3.5.1、Spring AI 1.1.2、PostgreSQL 16、Flyway、MCP Server、JUnit 5、AssertJ

**执行环境备注：** 2026-04-16 按用户要求已将工作树从 `codex/gap-fill-wave0` 切回 `main`，当前未提交改动已全部保留在 `main`，后续只在 `main` 分支继续开发。

## 排期总览

| 波次 | 范围 | 对应缺口 | 目标 | 预计工作量 |
|---|---|---|---|---|
| Wave 0 | 稳定性与核心纠错 | G6、G1、G2 | 防止超大 prompt 事故，补齐 correct / propagate 真执行闭环 | 3.5-4.5 人天 |
| Wave 1 | 编译质量与操作安全 | G4、G3 | 提升关键文件优先级与 rollback 安全能力 | 2-3 人天 |
| Wave 2 | 可观测性与知识读取 | G5、G8 | 质量趋势可追踪，开放 doc-toc / doc-read | 2-2.5 人天 |
| Wave 3 | 治理增强与可扩展 | G7、G10、G9 | lint 自动修复、语义分块、SCHEMA.md 自定义行为 | 3-4.5 人天 |
| 合计 | 全部缺口 | G1-G10 | 完成缺口文档定义范围 | 10.5-14.5 人天 |

## 依赖关系

- `G6` 必须先于 `G1/G2/G7/G9`，因为这些能力都会触发新的 LLM 调用。
- `G1` 必须先于 `G2`，因为传播依赖上游纠错标记和快照落盘。
- `G3` 依赖 `ArticleSnapshotJdbcRepository.save()` 落地，因此可以复用 `G1/G2` 已补出的快照写入能力。
- `G5` 依赖新增 Flyway 迁移 `V14`。
- `G10` 如果上线，需要安排 chunks 重建窗口。

## Wave 0 验收门槛

- `lattice_correct` 会真实改写文章，并留下 correction snapshot。
- `lattice_propagate` 会真实改写下游文章，并清理 upstream 标记。
- 超长输入会被截断，不会把 LLM 直接打爆。
