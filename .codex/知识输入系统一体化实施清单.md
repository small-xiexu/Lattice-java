## 实施总目标

- 完成 `documentparse / source / compile / query-governance` 四层职责收口，并保持 compile graph 只消费统一 `RawSource`
- 完成 OCR / 文档解析配置层与执行层，补齐 `.doc`、图片 OCR、扫描 PDF OCR 能力
- 完成 Source 领域模型、统一上传入口、自动识别归并、`WAIT_CONFIRM` 闭环与并发控制
- 完成多资料源 source-aware 数据模型切换，支持多个资料源共存且不发生 `concept_id / file_path` 冲突
- 完成 Git 资料源、管理员高级能力、前端状态展示与必要运维收口

## 执行顺序锁定

1. Phase A：架构地基收口
2. Phase B：文档解析配置层
3. Phase C：文档解析执行层
4. Phase D：资料源领域骨架
5. Phase E：统一上传与自动归并
6. Phase F：多资料源兼容改造
7. Phase G：Git 资料源与高级能力
8. Phase H：前端与运维收口

## 硬前置条件

- Phase C 仅做组件级验收，不引入“临时由 `IngestNode` 调用 `DocumentParseRouter`”的过渡实现
- `DocumentParseRouter` 的正式调用位置固定为 `SourceSyncWorker` 的预处理阶段
- `RawSource` 必须遵守主方案约定的最小契约：`sourceId / relativePath / extractedText / parseMode / parseProvider / contentHash / metadataJson`
- Phase E 首次真实写库前，必须完成旧全局唯一约束替换：
  - 移除 `articles.concept_id` 全局唯一
  - 移除 `source_files.file_path` 全局唯一
  - 建立 `unique(article_key)`、`unique(source_id, concept_id)`、`unique(source_id, relative_path)`
- `WAIT_CONFIRM` 的 confirm 成功后，必须同步回填：
  - `resolver_mode = MANUAL_OVERRIDE`
  - `resolver_decision = 用户确认值`
  - `sync_action = CREATE / UPDATE / APPEND` 的映射结果
- 本轮数据库按“全新项目重构”执行：
  - Flyway 目标结构直接收敛到 `V1__baseline_schema.sql`
  - `V2/V3/V4` 仅保留版本占位，不再承担旧结构迁移职责
  - 所有测试/开发 schema 统一按重建方式验证，不做历史数据兼容
