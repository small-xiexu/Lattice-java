# 向量检索链路三项优化技术方案

## 0. 文档目标

本文档定义 `Lattice-java` 向量检索链路的目标架构重构方案，聚焦 3 件事：

1. Embedding Provider 接入统一配置中心
2. 向量 schema 维度强约束与 ANN 索引固化
3. Chunk 级向量索引与 Query 图内并行召回

本文档按“全新项目、单基线重写”的前提编写，不考虑生产数据迁移，不保留旧配置 fallback，不为历史存量做兼容层。所有旧的向量相关实现都视为需要替换的存量代码。

---

## 1. 决策前提

### 1.1 项目口径

- 当前项目按“从 0 到 1 重构”处理
- 当前数据库迁移已经压平成单基线 `V1__baseline_schema.sql`
- 当前运行入口、Graph 主骨架、模型中心、后台配置页已存在
- 本轮不引入独立向量数据库，不拆独立检索服务
- 项目定位仍然是“知识库 + MCP 后端”

### 1.2 本轮硬约束

1. 不写旧配置兼容 fallback
2. 不写增量 migration 方案，直接重写基线 schema
3. 不保留“自由文本 embedding model + expectedDimensions”作为长期路径
4. 旧向量链路可以直接替换，不以最小改动为第一原则

### 1.3 设计原则

1. 复用现有模型中心，不另起一套 embedding 配置体系
2. 让 schema、profile、运行时调用保持强一致
3. 让 Query 主链继续 fail-open，但向量子系统内部配置错误必须显式报错
4. 让并行召回、向量写入、融合结果都具备可观测性
5. 让后续实现可以直接按文档拆成执行清单，不需要再补一轮“概念稿”

---

## 2. 当前问题

### 2.1 Embedding Provider 还没真正进入统一配置中心

当前后台虽然已经能保存 query 向量开关、embedding 模型名和维度，但运行时仍然依赖单个 `EmbeddingModel` Bean，导致：

- 后台保存的是“字符串模型名”，不是“具体 provider connection + model profile”
- 无法明确控制 embedding 走哪条 provider 连接
- chat 模型中心和 embedding 调用中心仍然是两套语义

### 2.2 向量 schema 与运行时配置仍可能漂移

当前文章级向量列物理维度固定，而后台配置允许修改期望维度，这会导致：

- 配置值和物理存储不一致
- 运行时只能依赖维度 mismatch 后的跳过或降级
- 管理员无法从后台清晰判断“当前到底是 schema 问题还是 provider 问题”

### 2.3 向量召回粒度偏粗，Query 召回仍然顺序执行

当前向量索引是文章级单向量索引，不是 chunk 级索引；同时 Query 检索阶段仍然是单节点顺序拉 5 路召回，导致：

- 长文档复杂语义被整篇文章向量稀释
- 复杂问题容易命中文章但命不中关键片段
- Query 延迟被顺序召回累加放大
- 现有 RRF 不能按通道权重做调优

---

## 3. 目标架构总览

### 3.1 总体思路

本轮重构后的目标链路如下：

1. 后台配置向量时，不再输入自由文本模型名，而是选择 embedding model profile
2. 编译图在文章切块后分别刷新文章级向量索引和 chunk 级向量索引
3. Query 图把 `FTS / RefKey / Source / Contribution / Article Vector / Chunk Vector` 拆为 6 个并行召回节点
4. `fuse_candidates` 从 `query_retrieval_settings` 读取权重，执行加权 RRF
5. 最终结果继续进入 `answer / review / rewrite`

### 3.2 目标分层

#### 配置层

- `llm_provider_connections`
- `llm_model_profiles`
- `query_vector_settings`
- `query_retrieval_settings`

#### 存储层

- `article_vector_index`
- `article_chunk_vector_index`

#### 编译层

- `refresh_article_vector_index`
- `refresh_chunk_vector_index`

#### 查询层

- `retrieve_fts`
- `retrieve_refkey`
- `retrieve_source`
- `retrieve_contribution`
- `retrieve_article_vector`
- `retrieve_chunk_vector`
- `fuse_candidates`

## 4. 数据库设计

## 4.1 总体策略

本轮直接重写 `V1__baseline_schema.sql`，不新增 `V2/V3/...` 增量迁移，不保留旧向量表结构的兼容写法。

同时，本轮要把现有 `V2__query_vector_settings.sql` 这种过渡脚本退场，收回到单基线。

### 4.1.1 迁移脚本要求

虽然本轮是单基线重写，但基线 DDL 仍要保持幂等：

- `CREATE TABLE IF NOT EXISTS`
- `CREATE INDEX IF NOT EXISTS`
- 条件创建 pgvector 相关表时，仍需先判断 `vector` 类型是否存在

这里的幂等是为了方便本地反复清库和重建，不是为了兼容历史版本。

## 4.2 `llm_model_profiles`

在现有模型档案表基础上直接加入以下字段：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `model_kind` | `VARCHAR(16) NOT NULL DEFAULT 'CHAT'` | 取值：`CHAT` / `EMBEDDING` |
| `expected_dimensions` | `INTEGER` | 仅 embedding 模型填写 |
| `supports_dimension_override` | `BOOLEAN DEFAULT FALSE` | 预留给后续 provider 能力识别 |

语义约束：

1. `CHAT` 模型可为空 `expected_dimensions`
2. `EMBEDDING` 模型必须填写 `expected_dimensions`
3. `query_vector_settings.embedding_model_profile_id` 只能引用 `model_kind='EMBEDDING'` 的 profile

## 4.3 `query_vector_settings`

去掉自由文本字段：

- `embedding_model`
- `expected_dimensions`

改为直接引用：

- `embedding_model_profile_id BIGINT NOT NULL REFERENCES llm_model_profiles(id)`

推荐表结构语义：

| 字段 | 说明 |
| --- | --- |
| `config_scope` | 当前固定 `default` |
| `vector_enabled` | 是否启用向量检索 |
| `embedding_model_profile_id` | 当前选中的 embedding profile |
| `created_by / updated_by / created_at / updated_at` | 审计信息 |

## 4.4 `query_retrieval_settings`

新增表，用于加权 RRF 与并行召回配置：

```sql
CREATE TABLE IF NOT EXISTS query_retrieval_settings (
    id                    BIGINT PRIMARY KEY DEFAULT 1,
    parallel_enabled      BOOLEAN      NOT NULL DEFAULT TRUE,
    fts_weight            NUMERIC      NOT NULL DEFAULT 1.0,
    source_weight         NUMERIC      NOT NULL DEFAULT 1.0,
    contribution_weight   NUMERIC      NOT NULL DEFAULT 1.0,
    article_vector_weight NUMERIC      NOT NULL DEFAULT 0.6,
    chunk_vector_weight   NUMERIC      NOT NULL DEFAULT 1.2,
    rrf_k                 INTEGER      NOT NULL DEFAULT 60,
    updated_at            TIMESTAMPTZ  NOT NULL DEFAULT CURRENT_TIMESTAMP
);

INSERT INTO query_retrieval_settings DEFAULT VALUES ON CONFLICT DO NOTHING;
```

这张表的作用是：

- 把权重配置从代码常量移到数据库
- 允许后台后续调权，而不需要重发版

## 4.5 `article_vector_index`

文章级向量索引表直接重写为：

```sql
CREATE TABLE article_vector_index (
    concept_id           VARCHAR(128) PRIMARY KEY,
    model_profile_id     BIGINT       NOT NULL REFERENCES llm_model_profiles(id),
    embedding_dimensions INTEGER      NOT NULL,
    index_version        VARCHAR(64)  NOT NULL,
    content_hash         VARCHAR(64)  NOT NULL,
    embedding            vector(1536),
    updated_at           TIMESTAMPTZ  DEFAULT NOW()
);

CREATE INDEX idx_article_vector_hnsw
    ON article_vector_index
    USING hnsw (embedding vector_cosine_ops)
    WITH (m = 16, ef_construction = 64);
```

字段语义：

- `model_profile_id`
  - 记录当前向量由哪一个 embedding profile 生成
- `embedding_dimensions`
  - 当前写入向量的实际维度
- `index_version`
  - 当前索引版本，建议由 `profileId + dimensions + strategy` 组合生成
- `content_hash`
  - 文章级内容哈希，使用 SHA-256 的十六进制字符串

文章级向量保留的目的不是最终细粒度召回，而是：

- 作为粗召回兜底
- 在 chunk 级索引未命中时继续提供语义候选

## 4.6 `article_chunk_vector_index`

新增 chunk 级向量索引表：

```sql
CREATE TABLE article_chunk_vector_index (
    article_chunk_id     BIGINT       PRIMARY KEY,
    article_id           BIGINT       NOT NULL,
    concept_id           VARCHAR(128) NOT NULL,
    chunk_index          INTEGER      NOT NULL,
    model_profile_id     BIGINT       NOT NULL REFERENCES llm_model_profiles(id),
    content_hash         VARCHAR(64)  NOT NULL,
    embedding            vector(1536),
    updated_at           TIMESTAMPTZ  DEFAULT NOW()
);

CREATE INDEX idx_chunk_vector_concept
    ON article_chunk_vector_index (concept_id, chunk_index);

CREATE INDEX idx_chunk_vector_hnsw
    ON article_chunk_vector_index
    USING hnsw (embedding vector_cosine_ops)
    WITH (m = 16, ef_construction = 64);
```

字段语义：

- `article_chunk_id`
  - 直接关联 `article_chunks.id`
- `content_hash`
  - chunk 级内容哈希，用于增量跳过
- `concept_id + chunk_index`
  - 用于召回后聚合与回放命中位置

## 5. Phase A：Embedding Provider 统一路由

## 5.1 目标

让向量调用不再依赖单个全局 `EmbeddingModel` Bean，而是和 chat 一样走“配置中心 -> 路由解析 -> 客户端构建 -> 运行时调用”。

## 5.2 `EmbeddingRouteResolution`

新增值对象：

| 字段 | 说明 |
| --- | --- |
| `profileId` | embedding profile 主键 |
| `providerType` | provider 类型 |
| `baseUrl` | provider 地址 |
| `apiKey` | 解密后的 API Key |
| `modelName` | 实际模型名 |
| `expectedDimensions` | 期望维度 |

## 5.3 `EmbeddingRouteResolver`

职责：

1. 输入 `embeddingModelProfileId`
2. 查询 `llm_model_profiles JOIN llm_provider_connections`
3. 要求：
   - `llm_model_profiles.model_kind = 'EMBEDDING'`
   - `llm_model_profiles.enabled = true`
   - `llm_provider_connections.enabled = true`
4. 输出 `EmbeddingRouteResolution`

错误处理：

- 如果 profile 不存在，直接抛 `EmbeddingProfileNotFoundException`
- 如果 profile 存在但类型不是 `EMBEDDING`，直接抛异常
- 如果 provider connection 不可用，直接抛异常

这里不写 fallback，也不写 legacy 兼容分支。

## 5.4 `EmbeddingClientFactory`

职责：

- 输入 `EmbeddingRouteResolution`
- 按 `providerType` 构建具体 `EmbeddingModel`

支持范围：

- `openai`
- `anthropic`
- `ollama`

约束：

1. 不注册为全局 Spring Bean
2. 由工厂按需构建
3. 允许在服务层做本地 client cache，但 cache key 必须与 `profileId + providerType + baseUrl + modelName` 绑定

## 5.5 `ConfiguredVectorEmbeddingService`

重写目标：

- 移除对单个 `EmbeddingModel` Bean 的依赖
- 注入：
  - `EmbeddingRouteResolver`
  - `EmbeddingClientFactory`
  - `QueryVectorConfigService`

执行流程：

1. 读取当前 `query_vector_settings.embedding_model_profile_id`
2. 调用 `EmbeddingRouteResolver.resolve(...)`
3. 调用 `EmbeddingClientFactory.getOrCreate(...)`
4. 构造 embedding request
5. 调用 provider
6. 输出结构化日志

统一日志前缀：

- `[VECTOR][EMBED]`

每条日志至少包含：

- `profileId`
- `modelName`
- `providerType`
- `dimensions`
- `latencyMs`
- `success`

## 5.6 事件与缓存失效

新增事件：

- `EmbeddingProfileChangedEvent`

触发时机：

- `QueryVectorConfigService.save(...)` 在 profile 变更成功后发布

监听方：

- `ConfiguredVectorEmbeddingService`
- 或单独的 `EmbeddingClientCache`

效果：

- 清理本地 embedding client cache
- 保证 admin 切换 profile 后，下一次调用立即使用新配置

## 5.7 外层失败语义

虽然 `EmbeddingRouteResolver` 不做 fallback，但 Query 主链仍保持 fail-open。

边界约束：

- `EmbeddingRouteResolver` 配置错误时抛异常
- `VectorSearchService` 捕获该异常后返回空列表
- `ArticleVectorIndexService / ArticleChunkVectorIndexService` 捕获该异常后跳过索引写入
- Query 主链继续依赖 `FTS / Source / Contribution` 继续执行

也就是说：

- 向量子系统内部：强约束
- Query 主链外部：fail-open

## 6. Phase B：Schema 维度强约束

## 6.1 目标

让“embedding profile 维度”和“数据库向量列维度”保持强一致，彻底移除后台动态改维度的语义。

## 6.2 维度策略

本轮不提供运行时动态改维度能力。

语义规定：

- 维度变更 = 修改基线 DDL + 切换 embedding profile + 全量重建向量索引
- 后台不再提供“直接编辑 expectedDimensions”的入口
- 后台只允许展示当前 profile 维度，不允许自由输入

## 6.3 `VectorSchemaInspector`

新增类：

- `VectorSchemaInspector`

职责：

1. 读取当前激活的 embedding profile 的 `expected_dimensions`
2. 读取数据库中 `article_vector_index.embedding` 列的物理维度
3. 对比后返回：
   - `profileDimensions`
   - `schemaDimensions`
   - `dimensionsConsistent`

实现说明：

这里不建议直接依赖 `information_schema.columns` 提取 pgvector 维度，因为该视图无法稳定暴露 `vector(n)` 的精确信息。

推荐实现：

- 使用 `pg_attribute / pg_class / pg_namespace` + `format_type(a.atttypid, a.atttypmod)`
- 从 `format_type(...)` 结果中解析 `vector(1536)` 这一类字符串

这和项目当前已有的向量列状态探测实现更一致，也更可靠。

## 6.4 后台状态接口

`GET /api/v1/admin/vector/status` 新增字段：

- `profileDimensions`
- `schemaDimensions`
- `dimensionsConsistent`
- `annIndexReady`
- `annIndexType`

状态语义：

- `dimensionsConsistent = false`
  - 表示当前 profile 与物理 schema 不一致
- `annIndexReady = false`
  - 表示 HNSW 索引不存在或不可用

这里不允许“维度不一致但静默继续通过”。

## 6.5 基线 DDL 与索引策略

因为本轮是基线重写，不是在线迁移，所以 HNSW 索引直接写入基线 DDL 即可，不需要使用 `CONCURRENTLY`。

原因：

- 基线初始化时表为空，不存在在线切换问题
- `CREATE INDEX CONCURRENTLY` 不能放入事务块，不适合当前基线建表语义

后续如果未来再增加“在线重建向量索引”的后台作业，再单独设计 `CONCURRENTLY` 版本的运维 SQL。

## 7. Phase C：Chunk 向量 + 并行召回

## 7.1 切块策略

chunk 向量构建必须采用滑动窗口重叠策略。

约束如下：

1. 使用现有 `article_chunks` 作为主输入
2. 相邻 chunk 之间重叠最后 1 句
3. 如果最后 1 句超过 50 token，则只保留 50 token 重叠
4. 每个 chunk 独立计算 embedding，不复用相邻 chunk 的向量

目的：

- 避免关键语义刚好被切在 chunk 边界处
- 降低长文档复杂问题的漏召回概率

## 7.2 编译图节点拆分

把当前单一的 `refresh_vector_index` 拆成两个独立节点：

1. `refresh_article_vector_index`
2. `refresh_chunk_vector_index`

职责划分：

- `refresh_article_vector_index`
  - 文章级向量索引
  - 用于粗召回与兜底
- `refresh_chunk_vector_index`
  - chunk 级向量索引
  - 用于细粒度召回

增量编译时：

- 只对变更文章的 chunk 重建向量
- 跳过逻辑通过 `content_hash` 判断

## 7.3 新增 `ArticleChunkVectorIndexService`

职责：

1. 扫描指定文章的 `article_chunks`
2. 对每个 chunk 生成 `content_hash`
3. 如果 `content_hash` 未变化，则跳过
4. 调用 `ConfiguredVectorEmbeddingService.embed(...)`
5. 写入 `article_chunk_vector_index`

统一日志前缀：

- `[VECTOR][CHUNK_INDEX]`

每条日志至少包含：

- `conceptId`
- `articleChunkId`
- `chunkIndex`
- `profileId`
- `latencyMs`
- `success`

## 7.4 Query 图并行召回

重写 `QueryGraphDefinitionFactory`，把原来的单节点 `retrieve_candidates` 拆成 6 个并行节点：

- `retrieve_fts`
- `retrieve_refkey`
- `retrieve_source`
- `retrieve_contribution`
- `retrieve_article_vector`
- `retrieve_chunk_vector`

并行节点写入 state / working set 时，key 必须完全隔离：

- `ftsHits`
- `refkeyHits`
- `sourceHits`
- `contributionHits`
- `articleVectorHits`
- `chunkVectorHits`

这 6 个节点完成后汇合到：

- `fuse_candidates`

由 `fuse_candidates` 统一读取所有 channel，再执行融合。

## 7.5 `ChunkToArticleAggregator`

新增类：

- `ChunkToArticleAggregator`

输入：

- chunk 级命中列表

每条命中至少包含：

- `articleId`
- `conceptId`
- `chunkIndex`
- `score`
- `chunkText`

输出：

- `QueryArticleHit` 列表

聚合规则：

1. 按 `concept_id` 分组
2. 同一 article 下所有命中 chunk 的 article 分数取 `max score`
3. 不做累加，不做平均
4. 保留：
   - 最高分 chunk 文本片段
   - 命中 chunk 数量
   - 最高分 chunk 的 `chunkIndex`

设计原因：

- 避免长文档因为命中 chunk 多而系统性占优
- 让 article 级候选仍然可与其他 channel 对齐

## 7.6 `RrfFusionService`

重写 `RrfFusionService`：

```java
fuse(Map<String, List<QueryArticleHit>> channelHits, Map<String, Double> weights, int limit)
```

约束：

1. `rrfK` 从 `query_retrieval_settings` 读取，不硬编码
2. 各 channel 权重从 `query_retrieval_settings` 读取
3. 默认权重为：
   - `ftsWeight = 1.0`
   - `sourceWeight = 1.0`
   - `contributionWeight = 1.0`
   - `articleVectorWeight = 0.6`
   - `chunkVectorWeight = 1.2`
   - `rrfK = 60`

这里不要求 `refkey` 单独配权，首轮可并入文本型召回统一权重策略；如果后续需要，也可再扩字段。

## 7.7 Query 召回耗时观测

在 Query 图检索阶段补齐统一耗时日志：

- `[VECTOR][RETRIEVE][START]`
- `[VECTOR][RETRIEVE][END]`

至少包含：

- `queryId`
- `mode=serial|parallel`
- `elapsedMs`
- `success`

这样可以在真实链路验收中直接比较串行召回与并行召回的总耗时差异。

## 8.1 向量配置页

只读展示：

移除：

- `query_retrieval_settings`

至少支持展示与保存：

- `parallelEnabled`
- `ftsWeight`
- `sourceWeight`
- `contributionWeight`
- `articleVectorWeight`
- `chunkVectorWeight`
- `rrfK`

## 8.3 重建入口

后台保留：

这里不再承担“改维度”的职责。维度问题必须通过基线 DDL 和 profile 一致性来解决。

## 9. 日志规范

所有向量相关操作统一使用以下前缀：

- `[VECTOR][EMBED]`
- `[VECTOR][INDEX]`
- `[VECTOR][CHUNK_INDEX]`
- `[VECTOR][RETRIEVE][START]`
- `[VECTOR][RETRIEVE][END]`

每条日志至少包含：

- 操作标识
- 关键 ID
- `latencyMs`
- `success`

建议最少字段：

## 12. 结论

这次不是“在旧向量方案上打补丁”，而是一次目标架构直达的重构：

1. Embedding 进入统一模型中心
2. Schema 与 profile 维度强一致
3. 向量索引升级为文章级 + chunk 级双层结构
4. Query 检索升级为图内并行多路召回
5. 融合逻辑升级为带权重的 RRF

后续所有实现、测试、后台改造与验收，都应以这份目标架构方案为准，不再回退到“旧字段兼容、旧表结构兼容、旧逻辑兜底”的思路。
