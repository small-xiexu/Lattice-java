# B9 原版能力超越设计方案

**文档版本**：v3.0（按用户校正重构）
**编写日期**：2026-04-16
**适用范围**：Lattice-java 后续"原版能力补齐并超越"阶段
**目标口径**：原始项目已有功能，Java 版必须具备；同时在服务化、治理、可运维性上整体强于原版

---

## 1. 背景与目标

当前 Lattice-java 已完成 B5-B8 主链重构，具备编译主链、检索主链、反馈闭环、治理能力、服务化能力。

若以"**原版有的能力 Java 版都要有，而且要更强**"作为验收标准，仍存在 5 类未完全收口的差距：

1. 缺少统一 CLI 入口（原版有完整命令树，本地自包含运行）
2. 缺少 Vault 导出 / 受控同步能力
3. 缺少整库级 snapshot / history / rollback
4. 缺少 stdio MCP 接入形态（原版支持 `node dist/cli.js mcp`）
5. 后台治理面尚未覆盖全部治理能力

**本阶段的核心约束**：不是向原版的本地工具形态退化，而是在保留 Java 版服务化优势的前提下，做到"原版能做的都能做，还有原版做不到的"。

---

## 2. 设计原则

### 2.1 数据真相原则

**数据库是系统主真相（source of truth）**。Vault 文件树是知识状态的可读投影，不是第二真相。

### 2.2 服务层优先原则

**CLI 和 stdio MCP 优先直接复用应用服务层，而不是依赖预启动的 HTTP 服务。**

- CLI / stdio MCP 以"独立模式"为默认，通过启动轻量 Spring 上下文（无 Web Server）直调 Service Bean
- "远程模式"（HTTP client 代理）作为可选补充，适用于服务已运行的场景
- HTTP API / Admin UI 复用同一套 Service 层

目标形态：

```text
CLI（独立） / stdio MCP（独立）
           ↓
   Application Service Layer          ← 主入口，共用
           ↓
Compiler / Query / Governance / Vault
           ↓
   PostgreSQL / Redis / LLM Gateway

CLI（远程）/ stdio MCP（bridge）
           ↓
      HTTP API / HTTP MCP             ← 可选补充
           ↓
   Application Service Layer
```

### 2.3 入口统一原则

所有入口（CLI / MCP / HTTP / Admin）共用同一套 Service 层，不允许 CLI 或 stdio MCP 绕开 Service 直写 Repository（基础设施命令除外）。

### 2.4 DB + 文件双轨版本管理原则

整库版本管理采用两条轨道并存，互为补充：

- **DB 快照轨**：事务安全，批量 rollback，用于灾难恢复和环境重放
- **JGit / Vault 轨**：文件可读，支持 diff 和历史审阅，用于人工审查和知识状态追溯

两轨在触发点上同步：重要操作完成后，同时写 DB 快照 + 提交 Vault Git。

### 2.5 受控同步原则

Vault 不是纯只读视图，但也不是无限制双向实时同步。

- 第一阶段：DB → Vault 稳定导出
- 第二阶段：Vault → DB 受控回写，有明确的允许范围、冲突检测、人工确认

不允许自动覆盖冲突，不允许回写非受管文件。

### 2.6 兼容优先原则

补齐原版能力时，优先保证与现有 HTTP API / 后台 / 测试体系兼容，不为追求形式一致而破坏现有生产化基础。

## 3. 当前差距总览

| 能力域 | 原始项目 | 当前 Java 版 | 结论 |
|---|---|---|---|
| CLI | 完整命令树，本地自包含 | 无 | 缺失 |
| Vault 文件知识库 | `concepts/*.md` + 合成产物 + `_contributions` | 仅落库 | 缺失 |
| 整库级 snapshot/history/rollback | Git 级版本控制 | 文章级 snapshot 仅有 Service，无 API | 缺失 |
| stdio MCP | 原生支持 | 仅 HTTP MCP | 缺失 |
| 后台治理面 | 无后台 | 有后台，治理页不完整 | 需增强 |

**当前代码现状补充**：

治理类 Service 均已存在（`LintService`、`InspectService`、`CoverageTrackingService`、`OmissionTrackingService`、`QualityMetricsService`、`SnapshotService`、`HistoryService` 等），但这些治理 Service **尚未暴露独立治理 API**。当前项目已具备 `overview / usage / articles / pending / compile job` 等管理接口；B9 Phase 0 的目标是补齐这些治理能力各自对应的独立 HTTP API，为 CLI 远程模式、后台治理面和整库 snapshot 铺路。

### 4.4 推荐实施顺序

1. **治理 API 补齐**（前置条件，为 CLI 远程模式和后台页面铺路）
2. **CLI 基座**（独立模式优先，远程模式跟进）
3. **DB 整库 snapshot + JGit Vault 双轨**
4. **Vault 受控同步**
5. **stdio MCP**（独立模式优先，Bridge 跟进）
6. **后台治理面补全**

## 5. 设计一：统一 CLI（服务层优先，HTTP 作补充）

### 5.1 当前现状

当前项目仅有 Spring Boot Web 入口（`LatticeApplication`），无 CLI。原始项目有完整命令树（`cli.ts`），可本地自包含运行。

### 5.2 目标

提供一个统一的 CLI 程序，作为运维、脚本、CI/CD 入口，**默认不依赖预启动的 HTTP 服务**。

### 5.3 双模式架构

#### 独立模式（默认）

CLI 启动轻量 Spring ApplicationContext（WebApplicationType.NONE），加载 `cli` profile：

```java
new SpringApplicationBuilder(LatticeCliConfig.class)
    .web(WebApplicationType.NONE)
    .profiles("cli")
    .run(args);
```

`cli` profile 加载范围：

| Bean 类型 | 是否加载 |
|---|---|
| DataSource / JdbcTemplate | ✓ |
| Redis | ✓（部分命令需要） |
| 全部 Service Bean | ✓ |
| 全部 Repository Bean | ✓ |
| Spring MVC / Tomcat | ✗ |
| MCP HTTP 端点 | ✗ |
| Admin 页面 | ✗ |
| 定时任务（编译作业轮询等） | ✗ |

预期启动时间：**5~10s**（含 HikariCP 连接池初始化、Redis 连接、Service/Repository Bean 扫描）。首次 JVM 冷启动（无 OS 文件缓存）可达 **15s 以上**。

这对 CI/CD 脚本和运维操作场景可接受，但不适合"交互式频繁调用"场景。后期如有需要可考虑以下优化路径：
- 开启 `spring.main.lazy-initialization=true`，将启动时间压缩至 3~6s
- 极端场景（如要求 < 1s）考虑 GraalVM native image，但构建复杂度高，不在 B9 范围内

#### 远程模式（可选补充）

通过 `--server` 参数或环境变量 `LATTICE_SERVER_URL` 指定已运行服务地址，CLI 改为 HTTP client 模式，委托给远程服务：

```bash
lattice-java compile --source ./doc.md --server http://localhost:8080
# 或
export LATTICE_SERVER_URL=http://localhost:8080
lattice-java compile --source ./doc.md
```

远程模式下无 Spring 上下文，冷启动 < 100ms。

## 6. 设计二：Vault 受控同步（DB 为主真相）

### 6.1 当前现状

原始项目直接写文件（`concepts/*.md`、合成产物、`_contributions`）。当前 Java 版仅落库，无文件导出能力。

### 6.2 目标

提供"DB → Vault 稳定导出 + Vault → DB 受控同步"的完整双向能力。

- **第一阶段**：DB → Vault 导出（全量 + 增量跳过）
- **第二阶段**：Vault → DB 受控回写（有限范围 + 冲突检测 + 人工确认）

### 6.3 终态目录结构

```text
~/.lattice/vaults/{project-name}/     -- 路径由配置指定，建议在项目目录外
  concepts/
    {conceptId}.md
  index.md
  timeline.md
  tradeoffs.md
  gaps.md
  _contributions/
    {type}-{yyyy-MM-ddTHH-mm-ss}-{id}.md
  _meta/
    export-manifest.json              -- 内容 hash，用于幂等导出
    sync-state.json                   -- 受控回写状态记录
    README.md                         -- 说明此目录为受管 Vault，修改规则
```

**Vault 路径建议在项目 Git 仓库目录外**，避免产生嵌套 Git 仓库（整库快照的 JGit 会在此目录初始化 `.git`）。

### 6.4 核心服务设计

```text
com.xbk.lattice.vault/
  VaultExportService.java         -- DB -> Vault 全量 / 增量导出
  VaultSyncService.java           -- Vault -> DB 受控回写入口
  VaultArticleWriter.java
  VaultArtifactWriter.java
  VaultContributionWriter.java
  VaultManifestStore.java         -- 读写 export-manifest.json
  VaultSyncStateStore.java        -- 读写 sync-state.json
  VaultConflictReport.java        -- 冲突报告
  VaultExportResult.java
  VaultSyncResult.java
```

### 6.5 导出策略（DB → Vault）

#### 6.5.1 概念文章

来源：`articles` 表  
输出：`vault/concepts/{conceptId}.md`  
内容保留：Markdown frontmatter、`[[wiki-links]]`、review / confidence / depends_on / related / referential_keywords

#### 6.5.2 合成产物

来源：`synthesis_artifacts`  
映射：`index` → `index.md`，`timeline` → `timeline.md`，`tradeoffs` → `tradeoffs.md`，`gaps` → `gaps.md`

#### 6.5.3 用户贡献

来源：`contributions` 表  
输出：`_contributions/{type}-{yyyy-MM-ddTHH-mm-ss}-{id}.md`

#### 6.5.4 幂等导出

- 导出时记录每个文件的 `contentHash` 到 `export-manifest.json`
- 重复导出时，hash 未变化的文件跳过写入，避免无意义 diff
- DB 中已删除的受管文件，导出时同步删除 Vault 中对应文件

### 6.6 受控回写规则（Vault → DB）

#### 6.6.1 允许回写的文件

| 文件范围 | 允许回写 | 说明 |
|---|---|---|
| `concepts/*.md` | ✓ | 人工编辑概念文章后回写 |
| `index.md` / `timeline.md` / `tradeoffs.md` / `gaps.md` | ✓ | 合成产物人工修订后回写 |
| `_contributions/*.md` | ✗ | 系统自动生成，不允许回写 |
| `_meta/` 下所有文件 | ✗ | 系统元数据，不允许回写 |

#### 6.6.2 Frontmatter 字段分类

基于当前 `ArticleRecord` 的实际字段，明确哪些字段允许人工编辑、哪些为系统保留：

**允许人工编辑**（Vault 回写时接受，写入 DB）：

| 字段 | 说明 |
|---|---|
| `content` | 文章正文，人工修订的核心字段 |
| `summary` | 文章摘要，人工可修订 |
| `title` | 标题，人工可修订 |
| `confidence` | 置信度（`high` / `medium` / `low`），人工评估 |
| `depends_on` | 上游依赖概念列表，人工可调整 |
| `related` | 相关概念列表，人工可调整 |
| `referential_keywords` | 参考关键词，人工可补充 |

**系统保留字段**（Vault 回写时忽略此字段，不写入 DB，以 DB 值为准）：

| 字段 | 说明 |
|---|---|
| `concept_id` | 唯一标识，系统分配，不允许修改 |
| `lifecycle` | 由生命周期状态机管理（`ACTIVE` / `ARCHIVED` 等） |
| `review_status` | 由审查流程写入（`pending` / `passed` / `needs_human_review` 等），不允许人工覆盖 |
| `compiled_at` | 编译时间戳，系统写入 |
| `created_at` / `updated_at` | 系统时间戳 |
| `source_paths` | 由编译流程写入的来源文件路径 |
| `metadata_json` | 系统内部元数据（含 upstream corrections 等），不允许人工覆盖 |

**回写验证规则**：
- 回写解析器发现 frontmatter 中含有系统保留字段时，跳过该字段并在日志中记录警告
- 不因保留字段存在而拒绝整个回写请求

#### 6.6.3 冲突判断

1. 导出时在 `export-manifest.json` 记录 `contentHash`
2. 回写时重新从 DB 读取当前 hash
3. 若 DB 当前 hash ≠ 导出时 hash，则判为冲突（DB 在导出后发生了变更）

#### 6.6.3 冲突处理

- 冲突时拒绝回写，报告冲突文件列表
- 不自动覆盖，需人工指定：`--force` 强制覆盖，或先重新导出后再回写

#### 6.6.4 回写命令

```bash
# 受控回写（遇冲突即停止）
lattice-java vault-sync --dir ~/.lattice/vaults/my-project --direction inbound

# 强制覆盖冲突文件
lattice-java vault-sync --dir ~/.lattice/vaults/my-project --direction inbound --force
```

### 6.7 验收标准

- 全量导出后 Obsidian 可直接挂载浏览
- 重复导出不产生无意义 diff
- 受控回写可识别并报告冲突，不自动覆盖
- 明确区分"允许回写"与"禁止回写"的文件范围

## 7. 设计三：整库级 snapshot / history / rollback（DB + JGit 双轨）

### 7.1 当前现状

原始项目使用 Git 进行整库版本控制，可 diff 和 rollback。当前 Java 版仅有文章级 snapshot（`SnapshotService` 已存在，但无 HTTP API）。

### 7.2 目标

新增整库级版本管理，兼顾：

- **事务安全**：DB 快照轨，保障 rollback 的一致性
- **文件可审阅**：JGit / Vault 轨，提供 human-readable diff 和历史

### 7.3 双轨设计

#### 轨道一：DB 快照轨

职责：事务性状态存储，驱动 rollback

- 触发点：重要操作完成后
- 存储：`snapshots` + `snapshot_items` 表（见 7.4）
- rollback 语义：批量 upsert 回目标快照状态，单个 DB 事务

#### 轨道二：JGit / Vault 轨

职责：文件级版本历史，支持 human-readable diff 和历史审阅

- 触发点：与 DB 快照轨同步触发
- 存储：Vault 目录下的 `.git`（由 JGit 管理）
- history 语义：`git log` 风格的提交列表
- diff 语义：文件级内容变更摘要

**两轨同时触发**，共同构成完整的版本管理体系：

```text
重要操作完成
     ↓
DB 快照（事务写入）+ Vault Git 提交（JGit）
     ↓
history 命令 = 读 DB 快照列表 + 补充 Git commit message
diff 命令    = 比较两个 Git commit 的文件差异
rollback 命令 = DB 快照事务回放 + Vault 重新导出 + Git 提交
```

### 7.4 快照存储设计与迁移策略

#### 7.4.1 现有文章级快照的实现现状

当前 `article_snapshots` 表由 **PostgreSQL 触发器**（`trg_capture_article_snapshot`）自动驱动：每次 `articles` 表发生 INSERT/UPDATE，触发器自动将所有字段平铺写入 `article_snapshots`。这是一个**扁平结构**（无 JSONB payload，每个字段独立一列）。

```text
articles 表 INSERT/UPDATE
        ↓ 触发器自动触发
article_snapshots 表（平铺 15 列）
```

#### 7.4.2 迁移策略：兼容演进（推荐）

B9 阶段**不迁移**现有 `article_snapshots` 表，采用兼容演进：

- **保留**：`article_snapshots` 表、触发器、`ArticleSnapshotJdbcRepository`、`SnapshotService` 不做任何改动
- **新增**：`repo_snapshots` + `repo_snapshot_items` 两张新表，专用于整库级快照
- **不合并**：文章级快照与整库级快照使用独立的表，不强行统一

原因：
- 触发器自动写入机制成熟稳定，迁移风险高、收益低
- 整库快照需要 JSONB payload 存多种实体类型，与文章级平铺结构本质不同，强行统一反而增加复杂度
- "统一表 + scope 字段"可作为后续独立清理项，不应成为 B9 的前置条件

#### 7.4.3 新增整库快照表（DB 快照轨）

仅用于 `scope='repo'` 的整库级快照：

```sql
CREATE TABLE repo_snapshots (
  id            BIGSERIAL PRIMARY KEY,
  created_at    TIMESTAMPTZ NOT NULL DEFAULT now(),
  trigger_event TEXT NOT NULL,   -- 'compile' | 'correct' | 'governance' | 'manual' | 'rollback'
  git_commit    TEXT,            -- 对应的 Vault Git commit hash
  description   TEXT,
  article_count INT
);

CREATE TABLE repo_snapshot_items (
  id            BIGSERIAL PRIMARY KEY,
  snapshot_id   BIGINT NOT NULL REFERENCES repo_snapshots(id),
  entity_type   TEXT NOT NULL,   -- 'article' | 'artifact' | 'contribution'
  entity_id     TEXT NOT NULL,
  content_hash  TEXT NOT NULL,
  payload       JSONB NOT NULL
);

CREATE INDEX idx_repo_snapshot_items_snapshot ON repo_snapshot_items(snapshot_id);
CREATE INDEX idx_repo_snapshot_items_entity   ON repo_snapshot_items(entity_type, entity_id);
```

**双轨对照**：

| 快照类型 | 表 | 触发方式 | rollback 语义 |
|---|---|---|---|
| 文章级（现有） | `article_snapshots`（平铺） | PostgreSQL 触发器自动写入 | 单篇 upsert 回历史字段 |
| 整库级（新增） | `repo_snapshots` + `repo_snapshot_items`（JSONB） | 代码显式调用 | 批量 upsert 回全库状态 |

### 7.5 JGit 集成（Vault 轨）

#### 7.5.1 Vault Git 仓库管理

- 路径：与 Vault 目录相同（`~/.lattice/vaults/{project-name}/`）
- 初始化：首次执行 `vault-export` 时，若无 `.git` 则自动 `git init`
- 提交时机：与 DB 快照同步，重要操作完成后立即提交

#### 7.5.2 提交规范

```text
[lattice:compile] persisted=12 articles
[lattice:correct] concept=payment-timeout
[lattice:governance] lint-fix, lifecycle-transition
[lattice:rollback] restored-to-snapshot=42
[lattice:manual] triggered by admin
```

#### 7.5.3 核心服务设计

```text
com.xbk.lattice.vault.snapshot/
  VaultSnapshotService.java     -- 双轨触发协调器（DB快照 + JGit提交）
  VaultGitService.java          -- JGit 操作封装（init / commit / log / diff）
  VaultHistoryItem.java
  VaultDiffSummary.java
  VaultRollbackResult.java
```

### 7.6 rollback 语义

整库 rollback 流程：

```text
1. 查 DB：repo_snapshot_items WHERE snapshot_id = target
2. 开 DB 事务：
   a. 批量 upsert articles（按 payload）
   b. 批量 upsert synthesis_artifacts
   c. 批量 upsert contributions
   d. 软删除 DB 中有但快照中无的受管记录
3. 提交 DB 事务
4. 触发 vault-export（将 DB 当前状态重新导出为 Vault 文件）
5. JGit 提交 Vault 变更（commit message 含 rollback 信息）
6. INSERT INTO repo_snapshots（trigger_event='rollback', git_commit=step5 的 hash, description='restored-to-snapshot={target}'）
```

步骤 3 失败 → 事务回滚，DB 不变，Vault 不变，操作幂等。  
步骤 4/5 失败 → DB 已更新，Vault 未更新，两轨暂时漂移。系统记录此状态，下次 `vault-export` 可自动修复。

### 7.7 CLI / API 能力

```bash
# 查看最近整库快照历史
lattice-java history --vault ~/.lattice/vaults/my-project --limit 20

# 查看目标快照的 diff 摘要
lattice-java diff --vault ~/.lattice/vaults/my-project --target 42

# 整库回滚到目标快照
lattice-java rollback --vault ~/.lattice/vaults/my-project --target 42
```

对应 HTTP API：

```text
GET  /api/v1/admin/snapshot/repo?limit=20    -- 整库快照列表（需新增）
GET  /api/v1/admin/snapshot/{id}/diff        -- diff 摘要（需新增）
POST /api/v1/admin/rollback/repo             -- 整库回滚（需新增）
GET  /api/v1/admin/snapshot/article?entityId={id}  -- 文章级快照列表（需新增）
POST /api/v1/admin/rollback/article          -- 文章级回滚（需新增）
```

### 7.8 验收标准

- `history` 命令可展示最近 N 次整库提交，包含 Git commit 信息
- `diff` 命令可展示目标版本与当前版本的文件级差异摘要（human-readable）
- rollback 在单 DB 事务内完成，事务失败则整体回滚
- rollback 后 Vault 文件内容与 DB 状态一致
- 文章级 snapshot / rollback 行为与现有一致

## 8. 设计四：stdio MCP（独立可用，Bridge 作补充）

### 8.1 当前现状

当前 Java 版仅支持 HTTP MCP（`/mcp`）。原始项目支持 stdio MCP，客户端通过 `node dist/cli.js mcp` 直接拉起。

### 8.2 目标

补齐 stdio MCP，**默认支持客户端直接拉起，不要求预启动 HTTP 服务**。

### 8.3 双模式设计

#### 独立模式（默认）

```bash
# 客户端配置（不依赖预启动服务）
{
  "mcpServers": {
    "lattice": {
      "command": "java",
      "args": ["-jar", "/path/to/lattice-cli.jar", "mcp-stdio"]
    }
  }
}
```

`mcp-stdio` 命令启动轻量 Spring 上下文（`WebApplicationType.NONE`，`mcp-stdio` profile），注册 MCP 工具，监听 stdin / stdout。

`mcp-stdio` profile 加载范围：

| Bean 类型 | 是否加载 |
|---|---|
| DataSource / Redis / Service Bean | ✓ |
| MCP 工具 Bean（LatticeMcpTools 等） | ✓ |
| Spring MVC / Tomcat | ✗ |
| 定时任务 | ✗ |

进程生命周期：随客户端拉起而启动，随客户端关闭而退出。

#### Bridge 模式（可选补充）

```bash
# 已有 HTTP 服务运行时使用
lattice-java mcp-stdio --bridge http://localhost:8080/mcp
```

Bridge 模式不启动 Spring 上下文，仅做 stdin/stdout ↔ HTTP MCP 的协议转发，冷启动 < 200ms。

### 8.4 工具定义统一

抽取 `ToolDefinition` 抽象，避免 HTTP 注解与 stdio 注册重复维护：

```text
com.xbk.lattice.mcp/
  ToolDefinition.java         -- name / description / parameters / executor
  ToolRegistry.java           -- 统一注册，HTTP MCP 和 stdio 双用
  stdio/
    StdioMcpBootstrap.java    -- 独立模式主循环
    StdioBridgeBootstrap.java -- Bridge 模式代理
```

现有工具（`LatticeMcpTools`、`PingTool`）注册到 `ToolRegistry`，HTTP MCP 和 stdio 均从此取工具定义。

### 8.5 验收标准

- 独立模式：客户端通过 `command + args` 直接拉起，无需预启动 HTTP 服务
- 工具名与 HTTP MCP 完全一致
- Bridge 模式可选，配置 `--bridge` 后代理到 HTTP MCP
- 两种模式返回语义一致

## 11. 风险与应对

### 11.1 CLI 独立模式启动时间

风险：`cli` profile 仍需启动 Spring 上下文，预期 5~10s；首次 JVM 冷启动（无 OS 文件缓存）可能达到 15s 以上。

应对：
- 初期接受，对 CI/CD 和运维脚本场景可接受
- `cli` profile 关闭所有非必要 Bean，启用 lazy-init
- 后期如需优化，可考虑 GraalVM native image

### 11.2 JGit 与项目 Git 仓库嵌套

风险：若 Vault 目录在项目目录内，JGit 初始化的 `.git` 会与项目 `.git` 嵌套，干扰 CI。

应对：
- Vault 路径默认配置在项目目录外（`~/.lattice/vaults/{project-name}/`）
- 文档明确说明，若用户自定义路径，不得放在项目目录内

### 11.3 双轨漂移（DB 更新后 Vault 导出失败）

风险：rollback Step 3 成功（DB 更新），Step 4 失败（Vault 导出），两轨状态不一致。

应对：
- 系统在 `repo_snapshots` 表记录当前 Git commit hash
- 若 hash 与实际 Vault Git HEAD 不一致，判定为"待修复漂移"
- 运维命令 `lattice-java vault-export --sync` 可修复漂移，重新对齐

### 11.4 受控回写冲突率高

风险：服务持续写库期间，Vault 人工编辑后回写冲突概率高。

应对：
- 明确受控回写是"维护窗口操作"，不是日常工作流
- 在 `_meta/README.md` 中写明：此目录为只读浏览用途，回写前应先检查冲突

### 11.5 stdio MCP 独立模式冷启动

风险：独立模式启动 Spring 上下文需 5~10s，频繁调用时影响体验。

应对：
- stdio MCP 进程生命周期是"拉起后持续运行"，不是每次调用重启，冷启动只发生一次
- Bridge 模式可作为替代，在服务已运行时使用

## 12. 最终结论

### 核心设计决策汇总

| 决策 | 选择 | 理由 |
|---|---|---|
| CLI 架构 | 服务层直调（独立）+ HTTP client（补充） | 原版本地自包含能力，同时保留远程便利性 |
| stdio MCP | Spring 独立模式（默认）+ Bridge（可选） | 不依赖预启动服务，与原版体验对齐 |
| Vault 方向 | 受控同步（双向，有限范围） | 兼容原版文件工作流，比原版更安全 |
| 版本管理 | DB 快照（事务）+ JGit（可读历史）双轨 | 事务安全 + 文件可审阅，强于原版纯 Git |
| 快照存储 | 兼容演进双表：`article_snapshots`（保留）+ `repo_snapshots`（新增） | B9 不迁移现有触发器表，"统一表"作为未来可选重构方向 |

### 强于原版的能力清单

1. **整库版本管理**：DB 事务快照 + JGit 文件历史双轨，比原版纯 Git 更安全、更可观测
2. **Vault 受控同步**：有冲突检测和人工确认，比原版直接写文件更安全
3. **CLI 远程模式**：委托已运行服务，原版没有
4. **服务化能力**：HTTP API + Admin UI + 编译作业队列，原版无
5. **后台治理面**：lint / inspect / coverage / omissions 全部可视化，原版无

### 兼容原版的能力清单

1. CLI 命令树（命令名与原版对齐）
2. Vault 目录结构（可被 Obsidian 直接挂载）
3. stdio MCP（与原版一致的客户端接入方式）
4. 整库 history / diff / rollback 语义

> **下一步建议**：基于本文档拆一份执行清单，Phase 0（治理 API 补齐）作为第一个可独立验收的里程碑，按 `/plan-checkpointing` 管理进度。
